# 参与写作

感谢你参与《深入理解 QEMU 设计原理》。本书希望解释代码背后的约束与设计选择，而不是把源码逐行翻译成中文。

## 章节结构

每章保存在一个独立 Markdown 文件中，建议保持以下结构：

```markdown
# 章节标题

本章要解决的问题，以及它和前后章节的关系。

## 本章目标

- 读完后可以解释什么；
- 可以沿哪条调用链定位代码；
- 可以完成哪个最小实验。

## 核心主题

正文……

## 小结
```

一级标题只用于章标题，不写“第 N 章”；二级、三级标题分别表示节和小节。尽量不要超过四级标题。

## 行文节奏与内容收敛

每一节先确定一个具体问题，再选择足够解释它的原理、源码和历史证据。邮件线程和 Git 提交只是论据，不按时间顺序全部搬进正文；无法帮助读者理解当前设计的材料，应留在研究记录中。

历史材料只保留能够解释设计转折的少量现场。QEMU 和 KVM 的早期故事不可避免会涉及 x86、Arm 或其他架构，但正文在说明问题与决策后，应回到 RISC-V target 展开当前源码、示例代码和实验。全书的实现锚点统一选择 `target/riscv/`、`hw/riscv/` 及其直接依赖；跨架构材料用于交代来路，不形成第二条案例主线。

历史回答“为什么走到这里”，原理和当前实现回答“这套系统怎样工作”。历史叙述不能挤占机制讲解：TCG、KVM、其他 Accelerator、外设建模、板卡建模以及常见调试方法都要形成能够独立学习和复核的技术路径。

行文保持简洁、通俗和轻快。段落长短可以变化，相关的短句应自然合并，用逗号控制停顿和信息密度；不要让每一段都遵循相同的句数，也不要机械地重复“概念、源码、总结”三段式。小标题服务于问题的推进，正文仍应读起来像一位工程师在解释真实的设计过程。

避免口号、宣传语、模板化过渡和空泛结论。涉及设计取舍时，要写清当时遇到的约束、候选方案、review 中的分歧、最终实现和代价。

每章正文至少包含 2500 个汉字，并把二级标题控制在 16 个以内。字数检查不统计 Markdown 标题、fenced div 标记、原始 LaTeX、URL 和代码块；思考题与参考答案属于正文的一部分，实验目录中的英文手册和代码不计入全书正文。全书不设置总字数门槛，验收单位是一个问题能否沿历史现场、候选方案、审查、当前实现和代价形成闭环。

每章至少安排一个可以独立复核的实验或源码验证入口。适合运行时观察的章节可以保留多个实验；以历史和协作为主的章节不为满足数量而制造伪实验。每个入口都要说明环境、命令、观察点、预期现象和能够支持的结论，不能只给出一串命令。

实验代码和手册不放进 `book/`，统一保存到 `experiments/part-XX-*/chapter-YY-*/*/`。篇、章和项目目录使用英文 kebab-case，项目的 `README.md` 也使用英文，至少包含 Purpose、Prerequisites、Files、Steps、Expected results、Cleanup 和 Troubleshooting。正文中的 `.hands-on` 使用中文说明，并链接到对应英文手册。每个实验项目应能单独运行，依赖、输入来源和清理步骤不能隐含在另一个实验中。

## 图片、表格与代码

图片统一放在 `book/images/`，文件名使用 `chNN-topic.svg` 或 `chNN-topic.png`。优先提交可编辑的 SVG：

```markdown
![从 guest 指令到 host 指令的转换路径](images/ch05-translation-pipeline.svg)
```

表格使用标准 Markdown 语法。代码块必须标注语言；省略代码时用注释明确指出，避免让片段看起来可以直接编译：

````markdown
```c
static void device_realize(DeviceState *dev, Error **errp)
{
    /* 只保留与当前讨论相关的路径。 */
}
```
````

行内标识符使用反引号，例如 `MemoryRegion`。命令行示例使用 `console`，纯输出使用 `text`。数学公式采用 Pandoc 支持的 LaTeX 形式：行内 `$x = y$`，块公式使用 `$$` 包围。

## 动态思考题

思考题应当紧跟它所检验的知识点，不在章末集中罗列。使用嵌套 fenced div 同时写题目和参考答案：

```markdown
:::: {.quick-quiz}
为什么 `MemoryRegion` 的配置树不直接用于每次访存？

::: {.quick-answer}
配置树适合表达组合关系，热路径则需要扁平化、可由 RCU 保护的查找结构。
:::
::::
```

构建时会自动完成以下工作：

- 按出现顺序生成“章号.题号”，例如“思考题 4.2”；
- 在正文当前位置渲染灰色题目框；
- 把嵌套的 `quick-answer` 移到“思考题答案”附录；
- 生成题目、答案和原题页码之间的双向链接。

每道题必须包含非空的 `quick-answer`。不要手工编号，也不要手工修改 `book/quiz-answers.md` 中的占位符。

每章通常安排二到四道随文思考题。题目用于检查状态所有权、设计代价、失败条件和不同路径的边界，避免询问某个字段名、年份或人物等可以直接查表的内容。

## 专题提示框

构建链路支持三种 fenced div，用来区分正文之外的信息：

```markdown
::: {.design-note}
这里解释一种设计为什么成立，以及被放弃的替代方案。
:::

::: {.source-path}
这里给出建议跟踪的入口函数、关键结构体和调用顺序。
:::

::: {.hands-on}
这里给出可以独立复现的命令、预期输出和观察点。
:::
```

不要把大段普通正文放进提示框。每个实验应注明 QEMU 版本、目标架构、宿主架构、配置命令和预期现象。

## 引用与版本

- 解释当前实现时，以 QEMU `v11.1.0-rc0` 为本轮源码审查基线，优先链接到 [QEMU 官方 GitLab](https://gitlab.com/qemu-project/qemu) 中的该 tag 或对应固定 commit，不使用 GitHub 镜像或 `master` 分支的浮动行号；目标发布线为 `v11.1.0`，正式 tag 出现后必须重新核验，不能直接沿用 RC 结论；
- 解释代码演进时，同时给出引入或重构该设计的 commit、qemu-devel 邮件线程和最终源码位置；
- 解释稳定接口或概念时，优先引用 QEMU 官方文档；
- 明确区分“协议规定”“QEMU 当前实现”和“作者的分析”；
- 性能结论必须附测试环境和测量方式。
