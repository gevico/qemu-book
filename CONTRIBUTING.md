# 参与写作

感谢你参与《深入理解 QEMU 设计原理》。本书希望解释代码背后的约束与设计选择，而不是把源码逐行翻译成中文。

## 章节结构

每章保存在一个独立 Markdown 文件中，建议保持以下结构：

```markdown
# 章节标题

本章要解决的问题，以及它和前后章节的关系。

## 本章目标

- 读完后可以解释什么；
- 可以沿哪条调用链定位代码；
- 可以完成哪个最小实验。

## 核心主题

正文……

## 小结

## 思考题
```

一级标题只用于章标题，不写“第 N 章”；二级、三级标题分别表示节和小节。尽量不要超过四级标题。

## 图片、表格与代码

图片统一放在 `book/images/`，文件名使用 `chNN-topic.svg` 或 `chNN-topic.png`。优先提交可编辑的 SVG：

```markdown
![从 guest 指令到 host 指令的转换路径](images/ch05-translation-pipeline.svg)
```

表格使用标准 Markdown 语法。代码块必须标注语言；省略代码时用注释明确指出，避免让片段看起来可以直接编译：

````markdown
```c
static void device_realize(DeviceState *dev, Error **errp)
{
    /* 只保留与当前讨论相关的路径。 */
}
```
````

行内标识符使用反引号，例如 `MemoryRegion`。命令行示例使用 `console`，纯输出使用 `text`。数学公式采用 Pandoc 支持的 LaTeX 形式：行内 `$x = y$`，块公式使用 `$$` 包围。

## 专题提示框

构建链路支持三种 fenced div，用来区分正文之外的信息：

```markdown
::: {.design-note}
这里解释一种设计为什么成立，以及被放弃的替代方案。
:::

::: {.source-path}
这里给出建议跟踪的入口函数、关键结构体和调用顺序。
:::

::: {.hands-on}
这里给出可以独立复现的命令、预期输出和观察点。
:::
```

不要把大段普通正文放进提示框。每个实验应注明 QEMU 版本、目标架构、宿主架构、配置命令和预期现象。

## 引用与版本

- 解释当前实现时，优先链接到 QEMU 官方仓库中的固定 commit，而不是 `master` 分支的浮动行号；
- 解释稳定接口或概念时，优先引用 QEMU 官方文档；
- 明确区分“协议规定”“QEMU 当前实现”和“作者的分析”；
- 性能结论必须附测试环境和测量方式。
