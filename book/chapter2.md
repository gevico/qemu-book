# 从命令行到第一条客户机指令

虚拟机启动看起来像一段顺序代码，实际更像一场分阶段装配。命令行先形成配置，QOM 对象随后被创建，machine 再连接 CPU、内存和设备，直到固件入口和 vCPU 运行条件都准备好，客户机第一条指令才会执行。

## 本章目标

- 理解参数解析、对象创建、machine 初始化和 vCPU 启动的先后关系；
- 区分固件启动、直接加载内核与裸机程序三条 RISC-V 启动路径；
- 能从一条命令定位到 `virt` machine 和首条客户机指令。

## 配置先于对象

`-machine`、`-cpu`、`-smp`、`-m`、`-bios` 和 `-kernel` 并不会在解析瞬间完成硬件操作。它们先进入配置层，随后成为 machine、CPU 和设备属性。这个延迟让 QEMU 可以检查组合是否合法，也为错误回滚留下空间。

:::: {.quick-quiz}
哪些配置必须在设备 realize 之前确定？

::: {.quick-answer}
会改变设备拓扑、MemoryRegion、总线连接、IRQ 路由或 guest ABI 的结构性属性通常必须提前确定。realize 会校验依赖并建立运行期资源，之后再修改这些属性，已经创建的连接和状态可能立即失效。
:::
::::

## 初始化阶段为什么被拆开

QEMU 支持大量 machine、加速器和后端，初始化不可能由一个函数一次完成。全局类型注册、machine 选择、加速器初始化、板级装配、设备 realize、reset 和 runstate 切换各自承担一段职责，阶段之间通过对象属性和显式状态传递信息。

::: {.design-note}
拆分初始化阶段会增加追踪成本，但它让可选组件能够在明确的时点声明依赖。阅读启动代码时，先问“当前阶段允许创建什么、允许失败吗”，比只记函数顺序更可靠。
:::

## RISC-V 的启动入口

RISC-V `virt` 可以从 OpenSBI 等固件进入，也可以由 QEMU 帮助装载内核、initrd 和设备树。CPU reset vector、固件地址、内核入口与设备树地址共同决定第一条指令之前的内存布局。

:::: {.quick-quiz}
为什么“客户机第一条指令”不一定是操作系统内核的第一条指令？

::: {.quick-answer}
机器通常先进入复位向量或固件，固件完成特权级切换、平台发现和启动参数准备后才跳到内核。只有直接加载裸机程序或特定的无固件配置，首条客户机指令才可能属于目标程序本身。
:::
::::

::: {.source-path}
沿 `system/main.c`、`system/vl.c`、`hw/riscv/virt.c`、`hw/riscv/boot.c` 和 `target/riscv/cpu.c` 建立启动时间线。源码位置以官方 GitLab 的固定版本为准。
:::

## 启动路径的演进证据

启动代码经常因为参数兼容、初始化顺序或错误处理被重构。研究这类改动时，需要比较重构前后的阶段边界，再阅读 patch series 为什么要移动某个初始化动作，review 是否担心 machine compatibility、退出清理或其他架构回归。

:::: {.quick-quiz}
为什么仅凭函数调用顺序无法判断初始化阶段是否设计合理？

::: {.quick-answer}
顺序只描述一次实现。设计是否合理，还取决于失败能否回滚、不同 machine 能否复用、属性何时冻结，以及旧命令行是否继续工作。这些约束通常要从错误路径、兼容属性和历史 review 中才能看清。
:::
::::

::: {.hands-on}
本实验的操作文件与验收项见英文手册 [Inspect CLI to machine](../experiments/part-01-system-foundations/chapter-02-startup-path/inspect-cli-to-machine/README.md)。正文报告使用中文，从一条完整命令中挑选 `-machine`、`-cpu`、`-smp`、`-m`、`-bios`、`-kernel` 与一个设备选项，分别追到配置对象、QOM 属性和最终实例。每个选项至少记录“解析时点、消费时点、失败时点”三项，不要只画一条没有错误分支的调用链。
:::

## `main()` 很短，启动过程并不短

在 `v11.1.0-rc0` 中，[`system/main.c`](https://gitlab.com/qemu-project/qemu/-/blob/v11.1.0-rc0/system/main.c) 的 `main()` 主要把控制交给 `qemu_init()`，完成后释放启动期间持有的全局执行约束，进入主事件循环。入口变短是组织结果，不代表初始化变少。真正的编排仍在 [`system/vl.c`](https://gitlab.com/qemu-project/qemu/-/blob/v11.1.0-rc0/system/vl.c) 等文件展开，类型注册、后端准备、Machine 选择、加速器配置与板级装配还会调用更多子系统。

阅读启动代码时，可以先把时间线分成六段。进程级环境和早期选项必须先建立，之后 QEMU 初始化主循环与 trace 等基础设施；机器选项被解析并选择 `MachineClass`；加速器与 CPU 能力得到配置；后端、Machine 和板级对象按依赖建立；最后执行复位并切换 runstate，vCPU 才开始推进。这六段只是一张阅读导航图，不对应一套正式 API，遇到当前版本的细节调整时也不会丢失阶段含义。

顺序中最值得关注的是依赖，不是函数在文件中的物理位置。例如 trace 要足够早，才能观察后续初始化；Machine 必须先确定，才能解释某些 machine 属性；加速器会约束 CPU 能力，CPU 与板级中断拓扑又互相依赖；设备后端往往要在设备 realize 前存在。若把“代码写在上面”当成“总是先执行”，很容易被回调、注册表和延迟对象创建误导。

当前版本中，`qemu_init()` 会依次处理选项验证、早期配置、主循环和计时基础设施，随后创建 Machine、设置默认设备和后端、应用机器选项并配置加速器。这里的概括故意不写成永恒的精确序号，因为 `vl.c` 一直在重构；需要定位某个选项时，应在固定 tag 上从它的 QemuOpts/QAPI 定义追到最终消费者，再用 trace 或调试器确认本次路径。

## 参数解析不是立即执行

假设命令行中出现 `-m 2G`。解析器可以很早知道用户写了一个容量，却不能立刻随意分配系统 RAM，因为 Machine 可能选择内存后端、限制最大值、建立 NUMA 拓扑或要求特定对齐。`-device` 更明显：类型名、属性和总线可能先进入配置队列，等板级默认总线存在后才创建并 realize。把“识别语法”和“改变机器”分开，允许 QEMU 在掌握更多上下文后做一致性检查。

这种延迟也解释了为什么错误位置有时离命令行解析很远。一个属性字符串可以通过词法检查，却在类型转换、对象查找、总线匹配或 realize 时失败。报告错误时，QEMU 要保留足够的配置来源，让用户知道问题对应哪项输入。读源码不能把晚失败误认为“选项没有解析”，应问前一阶段保存了什么表示，后一阶段又增加了什么约束。

默认值进一步增加了追踪难度。`-machine virt` 会选择 MachineClass 的默认 CPU 和默认设备策略，省略 `-bios` 可能触发默认固件查找，`-nographic` 会同时影响显示和字符设备组合。默认行为可能由 Machine、目标架构、构建配置或加速器共同决定。实验若研究启动入口，必须显式记录固件选择；否则同一条简短命令在不同构建树上可能走向不同首地址。

:::: {.quick-quiz}
为什么“参数已经被解析”不能推出“对应设备已经存在”？

::: {.quick-answer}
解析阶段只需把文本变成受检查的配置表示，设备创建还依赖 Machine、总线、后端和属性冻结时点。QEMU 故意延迟一部分动作，以便组合配置、校验依赖并统一回滚。验证设备是否存在，应检查对象创建、realize 与运行时树，而不是停在选项解析函数。
:::
::::

## Machine 从类型变成这一次的对象

`qemu_create_machine()` 是当前启动主线上的重要锚点。它先依据已选类型取得 `MachineClass`，设置兼容属性与当前机器上下文，再用 `object_new_with_class()` 创建 `current_machine`，把对象接入 QOM 组合树。随后建立 `/machine/unattached`、`peripheral`、`peripheral-anon` 等容器，准备系统总线与机器内存，并应用机器默认选项。这个阶段只得到承载后续装配的根对象，尚未形成能够运行的 RISC-V 机器。

类型和实例必须分开看。`virt_machine_class_init()` 在类型层设置描述、板级 `init` 回调、默认 CPU、最大 CPU 数和 AIA、ACLINT、IOMMU 等属性；本次启动的 `MachineState` 才保存内存大小、CPU 拓扑、固件、内核文件与用户配置。类数据若被某个实例随意修改，另一台同进程实例或后续对象会受到污染；实例属性若在板级初始化后才改变，又可能与已建立拓扑冲突。

`qemu_create_machine_containers()` 所建立的对象路径承担实际职责。没有显式父对象的用户设备需要一个可追踪的所有权位置，QOM introspection 和生命周期清理依赖这些组合关系。不过，处于 `/machine/peripheral` 并不等于设备连在某条客户机总线上；QOM owner 与 qdev bus 是两种关系。第二章只用它定位创建阶段，第四章再把它们完全拆开。

::: {.source-path}
在固定源码中核对 [`qemu_create_machine()`](https://gitlab.com/qemu-project/qemu/-/blob/v11.1.0-rc0/system/vl.c)、`qemu_create_machine_containers()`、`virt_machine_class_init()` 与 `virt_machine_init()`。链接锚定 `v11.1.0-rc0`，函数行号只记录在实验日志中，因为正式 `v11.1.0` 若有调整，行号可能变化而符号语义仍可比较。
:::

## 为什么加速器必须较早参与

加速器不是启动末尾才替换的一根函数指针。KVM 需要检查 `/dev/kvm`、API 版本和 RISC-V capability，建立 VM 级内核对象；TCG 要准备翻译和执行环境。CPU 能暴露哪些扩展、vCPU 如何创建、内存怎样注册、固件入口怎样写入，都可能依赖加速器。因此，配置加速器和确定 CPU 模型之间有明确约束，不能先按一个假想 CPU 完成整台机器，再随意切换执行后端。

失败策略由用户选择决定。用户明确写 `-accel kvm` 时，缺少能力通常应该清楚失败，悄悄回退 TCG 会让性能、CPU 特性乃至实验结论失真；用户给出加速器列表时，才可以按配置尝试后继项。实验必须保存实际 `query-accels` 与启动日志，不能看到命令里有 `kvm` 就假定运行已经进入内核。

在 RISC-V 路径中，KVM CPU 初始化还要把宿主支持的 ISA 扩展映射到 QEMU 属性，处理寄存器同步与 vCPU reset。TCG 则能由软件实现目标扩展。两种路径最终都要给 `virt_machine_init()` 可用的 CPU 对象，但对象能接受的配置并不完全相同。启动阶段的早期 capability 校验，就是为了把这种差异变成可解释错误。

## `virt_machine_init()` 开始装配平台

进入 [`hw/riscv/virt.c`](https://gitlab.com/qemu-project/qemu/-/blob/v11.1.0-rc0/hw/riscv/virt.c) 的 `virt_machine_init()` 后，抽象 Machine 才逐渐变成 RISC-V 客户机可见平台。函数根据 socket 与 hart 拓扑创建 `TYPE_RISCV_HART_ARRAY`，设置 CPU 类型、hartid 起点与数量，然后 realize。接着选择 CLINT/ACLINT 以及 PLIC/AIA 路径，映射系统 RAM，创建 MROM、测试设备、fw_cfg、virtio-mmio、PCIe、串口、RTC、flash 等对象，并构造最终设备树。

这段函数很长，却不能简单等同于“设计失败”。板级代码的任务本来就是把许多独立组件连接成一份平台契约，连接关系需要在一个可审查的位置显现。真正需要判断的是：组件内部逻辑是否仍留在各自设备，实现顺序是否由依赖解释，重复模式是否有帮助函数，错误和属性是否清楚。把所有代码机械拆成小函数，未必减少平台耦合，反而可能让地址和中断关系散落。

装配顺序会影响设备树。板级代码要先知道 CPU、内存和设备实例，才能生成正确节点；fw_cfg 又可能承载生成后的 DTB 或启动信息。MROM 中的 reset vector 需要知道固件入口、DTB 地址和动态固件信息的位置。于是，源码中某些“先创建、后生成、再写 ROM”的次序来自数据依赖，不能只按设备类别重新排序。

板级初始化也承担拒绝非法拓扑的责任。例如 socket、hart 数与平台最大 CPU 数要一致，内存范围不能与固定设备窗口重叠，IOMMU 和 PCIe 连接要满足所选模式。验证越靠近拥有完整上下文的位置，错误越准确；验证太晚，会留下部分实现的对象；验证太早，又缺少实际布局信息。当前代码中的检查位置可以据此评估，而不是以“越早越好”概括。

## RISC-V hart 容器为何位于板级与 CPU 之间

`RISCVHartArrayState` 把一组 hart 的创建组织成可配置对象。`virt` 按 socket 设置 `hartid-base`、`num-harts` 与 `cpu-type`，容器在 realize 时创建具体 `RISCVCPU`。这让板级代码处理拓扑与编号，CPU 类型处理 ISA 和特权状态，二者通过属性相接。将 hart 编号直接硬编码进 CPU 类，会让 CPU 难以在其他 Machine 复用；让每个设备自己查 CPU，又会破坏明确拓扑。

客户机的 hartid 不一定等于数组下标，更不等于宿主线程号。启动 ROM 会读取 `mhartid` 放入 `a0`，固件与内核用它识别当前 hart；QEMU 需要让设备树 CPU 节点、中断上下文和实际 CPU 对象使用一致编号。多 socket 配置因此是验证拓扑的好工具，即使日常只启动一个 hart，也不应在分析里默认编号永远从数组位置直接推导。

CPU realize 完成后，还没有执行指令。realize 表示对象依赖和执行后端资源已经建立，runstate 与 vCPU 线程控制决定它何时运行。`-S` 正是利用这一区别：机器可以完全创建、监听 GDB、接受查询，却保持 CPU 停止。这个状态为第二个实验提供了稳定观察窗口。

## 三类装载物：固件、内核与 initrd

RISC-V 启动路径常被一条 `-kernel` 隐藏。实际上，固件、内核和 initrd 是三类具有不同职责与放置规则的镜像。固件通常在更高特权级运行，初始化平台并提供 SBI 服务；内核接收 hartid 与设备树，建立自己的页表和驱动；initrd 是内核启动早期使用的数据，不会成为 CPU reset 后直接执行的入口。

[`hw/riscv/boot.c`](https://gitlab.com/qemu-project/qemu/-/blob/v11.1.0-rc0/hw/riscv/boot.c) 把这些动作拆成 `riscv_find_and_load_firmware()`、`riscv_load_firmware()`、`riscv_load_kernel()`、`riscv_load_initrd()`、`riscv_load_fdt()` 等函数。内核装载会尝试 ELF、uImage 与原始镜像等格式，格式决定入口与段如何解释；initrd 位置还要考虑内存容量、内核范围和对齐。把它们统称为“把文件复制到 RAM”，会漏掉格式、入口和重叠检查。

`-bios default`、显式固件文件与 `-bios none` 表达不同意图。默认固件取决于构建和安装数据目录，显式路径更适合可复现实验，`none` 则要求启动方案能够在没有常规固件的情况下建立合法入口。`-kernel` 不自动保证镜像能在当前特权级直接运行；一份 Linux 内核通常依赖固件提供的 SBI 和规定的入口状态，裸机镜像则可能自己承担 M-mode 初始化。

:::: {.quick-quiz}
为什么同一个文件既能以 ELF 装载又可能以 raw 装载时，首条有效代码位置会不同？

::: {.quick-answer}
ELF 带有段地址和入口字段，装载器按程序头放置内容并返回声明的 entry；raw 镜像没有这些元数据，只能由平台规则或命令参数选择装载基址与入口。文件字节相同，不同格式解释仍可能让 PC 与内存布局不同，所以实验要记录装载器识别结果。
:::
::::

## reset vector 是一小段真实 RISC-V 代码

`virt` 并非简单把 CPU 的 PC 写成内核入口。当前 `riscv_setup_rom_reset_vec()` 会在 MROM 写入一小段 RISC-V 指令和数据。代码用 `auipc`、`addi` 等取得旁边的信息，读取 `mhartid` 放入 `a0`，把设备树地址准备到 `a1`，把 OpenSBI 动态固件信息地址准备到 `a2`，随后跳往计算出的启动入口。若相关扩展配置不同，个别指令还会相应调整。

这段 ROM 解决了几个交接问题。每个 hart 从共同 reset 区进入，却要携带自己的 hartid；设备树由 QEMU 在装配后生成，地址要在运行前告诉下一阶段；动态固件还需要知道下一镜像入口、模式及其他信息。把这些值编码在 ROM 旁边，能让复位后的代码用位置相关的短路径取得，从而避免每个 CPU 模型硬编码 Machine 细节。

RISC-V [`Machine-Level ISA`](https://docs.riscv.org/reference/isa/priv/machine.html) 规定复位进入 M-mode，并把 PC 放到实现定义的 reset vector，具体地址由平台决定。QEMU `virt` 选择自己的 MROM 布局，属于平台契约；OpenSBI 和 Linux 再分别规定阶段间寄存器约定。规范没有要求所有 RISC-V 机器都从 `0x1000` 启动，因此不能把 `virt` 的地址写成 ISA 事实。

::: {.design-note}
已核对事实：`v11.1.0-rc0` 的 `riscv_setup_rom_reset_vec()` 生成实际指令，并准备 `a0`、`a1`、`a2` 后跳转；RISC-V 特权规范把 reset vector 地址留给实现。上游代码直接说明的是机制。作者据此推断，把板级交接集中在短 ROM 中，可以避免固件格式与 CPU reset 逻辑彼此硬编码，也便于 Machine 在生成 DTB 后补入地址；这项工程解释应与后续提交历史共同验证。
:::

## OpenSBI 把 M-mode 平台细节挡在内核之前

OpenSBI 的[固件接口文档](https://github.com/riscv-software-src/opensbi/blob/master/docs/firmware/fw.md)区分 `FW_DYNAMIC`、`FW_JUMP` 与 `FW_PAYLOAD` 等形态。动态固件从前一阶段取得下一镜像信息，jump 固件跳到预先配置的位置，payload 则把下一阶段打包在固件映像内。无论采用哪种方式，固件都要建立适合 S-mode 软件的环境，并通过 SBI 提供跨实现的平台服务。

对 Linux 而言，[RISC-V 启动要求](https://docs.kernel.org/next/arch/riscv/boot.html)约定 `a0` 携带当前 hartid、`a1` 携带 DTB 地址，进入时页表关闭等。QEMU reset ROM 先把信息交给固件，OpenSBI 再按内核约定进入下一阶段。调试“内核第一行没有输出”时，应分别验证 ROM 到固件、固件到内核两次交接，而不是在 GDB 里看到 `a0/a1` 一次正确就宣布整条链完成。

固件还会改变可见日志与时间。OpenSBI banner 出现，说明 CPU 已执行到固件并且串口路径大体可用，却不证明内核入口正确；没有 banner 也不一定是 CPU 未运行，可能是构建关闭输出、串口配置不同或走了无固件路径。证据要贴合结论，串口字符只是启动时间线中的一个观测点。

## 直接内核启动也有明确的特权与状态约束

“直接内核启动”并不等于让 Linux 在任意 reset 状态下运行。若仍借助 OpenSBI，QEMU 只是把内核作为下一阶段装载，固件负责切到 S-mode；在 KVM 等路径中，也可能由加速器和架构代码直接设置 vCPU 的 PC、寄存器与特权状态。不同方案都必须满足内核启动协议，只是准备工作由谁完成不同。

RISC-V KVM 接入系列中的提交 [`ad40be270`](https://gitlab.com/qemu-project/qemu/-/commit/ad40be270) 专门处理由 KVM 直接启动内核，这说明启动不是完全独立于加速器的公共尾部。KVM 中最新 CPU 状态可能位于内核对象，TCG 中则由软件 CPU 循环掌握；设置 reset PC 和参数的接口自然不同。Machine 仍决定加载什么、DTB 在哪里，加速器路径负责把 vCPU 带到符合协议的入口状态。

这类差异也是实验使用 `-S` 的理由。暂停在“vCPU 尚未运行”时，可以检查 QEMU 已创建的机器和预设状态；一旦执行若干指令，固件可能立即改写寄存器，GDB 看到的就不再是 reset 交接。若要捕获真正的第一条指令，还需确认 GDB attach、KVM 调试接口和断点设置本身没有先让 CPU 跑过。

## DTB 是启动代码与平台模型之间的书面交接

RISC-V `virt` 生成设备树，描述 hart、内存、chosen 节点、串口、中断控制器、virtio-mmio、PCIe 以及可选 IOMMU 等。客户机固件和内核不应靠读取 QEMU 内部对象树发现设备，它们只看 DTB、规范化寄存器和总线枚举。DTB 因而是检验板级模型是否完整的一份客户机视图。

设备树中的 `reg` 地址应与 MemoryRegion 映射一致，interrupts 或 interrupt-parent 应与实际 IRQ 连接一致，CPU 节点的 hartid 和 ISA 描述应与 CPU 对象一致。三者任何一处错位，都可能出现“设备在 QEMU 中存在，驱动却找不到”或“驱动访问了错误地址”。启动实验应同时导出 DTB 与 `info mtree`，而不是只检查其中一个。

`riscv_load_fdt()` 把最终 FDT 放到客户机内存，reset ROM 或直接启动路径再把地址交给下一阶段。FDT 放置要避开内核、initrd 与固件，且位于客户机可访问范围。大内存、非常规镜像或多段 ELF 会让“找一个空闲高地址”不再简单，因此装载函数中的范围计算和错误检查属于平台正确性，不是无关的辅助代码。

## 从历史提交理解今天的 boot.c

RISC-V 启动代码经历过多次拆分和修正。提交 [`0ac24d56`](https://gitlab.com/qemu-project/qemu/-/commit/0ac24d56) 把启动辅助逻辑拆出，后续 [`b3042223`](https://gitlab.com/qemu-project/qemu/-/commit/b3042223) 完善固件装载，[`fdd1bda4`](https://gitlab.com/qemu-project/qemu/-/commit/fdd1bda4) 引入默认 OpenSBI，`dc144fe1` 增加动态固件支持。单看当前函数名，很容易把这些职责当作一次设计；历史显示它们随着固件协议和平台能力逐步收敛。

2023 年提交 [`487d73fc470d`](https://gitlab.com/qemu-project/qemu/-/commit/487d73fc470d) 继续整合内核初始化，对应邮件 Message-ID [`20230206140022.2748401-3-dbarboza@ventanamicro.com`](https://lore.kernel.org/qemu-devel/20230206140022.2748401-3-dbarboza%40ventanamicro.com/)。阅读该系列时，应比较拆分前后谁计算入口、谁装载 initrd 与 DTB，以及其他 RISC-V Machine 是否复用。重构本身不必改变客户机行为，它的价值可能在于消除各板级实现对装载规则的复制。

2024 年提交 [`55c136599`](https://gitlab.com/qemu-project/qemu/-/commit/55c136599) 修正了固件 ELF 入口处理，对应 Message-ID [`20240817002651.3209701-1-samuel.holland@sifive.com`](https://lore.kernel.org/qemu-devel/20240817002651.3209701-1-samuel.holland%40sifive.com/)。提交说明指出，旧路径可能把 ELF 段装到文件声明的位置，却仍跳向调用者提供的地址；修正后尊重 ELF entry。这个例子很能说明格式语义为何不能被“固件基址”覆盖，也说明启动 bug 可能在装载成功后才表现为无输出。

::: {.design-note}
上游提交能够直接证明启动辅助函数被拆分、OpenSBI 动态固件得到支持，以及 ELF entry 曾被修正。作者的解释是，这条演进把“选择镜像”“按格式装载”“决定下一入口”“准备阶段参数”逐步分开，减少板级代码各自猜测格式的机会。若要声称某次拆分专门为了某个未在提交或邮件中出现的用户，则证据不足，本书不作这种归因。
:::

## `vl.c` 重构告诉我们如何读初始化代码

QEMU 启动入口长期集中在 `vl.c`，文件庞大，许多工作又有历史命令行兼容要求。2020 年提交 [`f5c9fcb82d37`](https://gitlab.com/qemu-project/qemu/-/commit/f5c9fcb82d37) 抽出 `qemu_create_machine()`，`d8e4de41` 等后续提交继续移动 prelaunch 与通用初始化。提交说明坦率承认 `qemu_init()` 仍承载难以立刻整理的部分。这里的工程现实很重要：重构往往先建立可命名阶段，再逐步移动职责，不会等待一张完美架构图才开始。

判断这类重构，首先看客户机可见顺序和错误行为是否保持，再看新边界是否让调用者更明确。函数变短不是充分指标；若隐藏的全局状态更多、阶段前置条件没有表达，代码可能只是被搬家。反过来，即使 `qemu_init()` 仍长，只要 Machine 创建、加速器配置和后端阶段有可测试边界，重构就可能已经降低后续改动风险。

历史还提醒我们不要把当前目录当成概念定义。旧资料可能说入口在根目录 `vl.c`，当前版本已经使用 `system/main.c` 和 `system/vl.c`；概念“进程入口与启动编排”仍在，路径却变了。引用旧邮件时要把符号映射回当前 tag，而不是因为文件名不同就放弃有价值的设计讨论。

## reset、runstate 与第一条指令之间还有一道门

所有对象 realize 完成后，QEMU 会执行系统 reset，让 CPU 和设备进入平台定义的初始状态。reset 不只是把 PC 改回去，设备寄存器、中断线、队列状态和计时也要恢复。启动时 reset 与运行中用户请求的 system_reset 共享框架，却可能有不同原因和通知顺序。分析第一条指令前，要确认观察点是在 realize 后、reset 前，还是 reset 后。

runstate 决定虚拟机处于 prelaunch、running、paused、shutdown 等哪种状态。`-S` 让机器在启动完成后保持暂停，并不跳过 Machine 创建或 reset；`cont` 才允许 vCPU 前进。GDB stub、QMP 和主循环在暂停期间仍可工作，这就是“机器存在”和“CPU 正在执行”分离的运行时证据。

多 hart 又给“第一条指令”增加了限定。每个 hart 都有自己的 reset 入口，固件可能让非启动 hart 等待，操作系统稍后通过 SBI 或中断唤醒。实验若只观察 hart 0，应写“启动 hart 的第一条指令”；若研究所有 hart，则记录各自 PC、`mhartid` 和进入等待路径的时间，不能把一个核的轨迹复制给整台 SMP 机器。

## 启动失败的分层诊断法

第一层检查 QEMU 自己是否完成机器创建。命令行错误、固件文件找不到、地址重叠或 KVM capability 不足，通常会在客户机执行前由 stderr 报告。此时不应先调内核。保存退出码、完整错误与 `-version`，再定位是解析、对象属性还是 realize 失败。

第二层检查 reset ROM 与固件。用 `-S -s` 观察初始 PC，读取 MROM 指令和旁边数据，确认跳转入口已映射；继续到 OpenSBI 入口，检查 hartid、DTB 和动态信息。串口 trace 或 GDB 断点可以帮助分段，但要避免逐指令日志掩盖最早异常。

第三层检查固件到内核。确认 ELF entry、Linux 要求的 `a0/a1`、内核地址与 initrd/DTB 不重叠，再看内核早期页表和串口配置。如果 DTB 地址正确但内容来自另一版本，内核同样可能停住。每过一层，就留下一个能独立复现的断点或内存证据，诊断会从“黑屏”收敛成某次交接失败。

:::: {.quick-quiz}
`-S` 启动后能看到完整的 QOM 树和内存树，是否说明固件已经执行成功？

::: {.quick-answer}
不说明。`-S` 保持 vCPU 暂停，Machine 与设备已经创建并 reset，所以管理面可查询；固件可能一条指令也没执行。要证明固件运行，需要让 CPU 前进，并在固件入口、串口输出或状态变化处取得独立证据。
:::
::::

## 实验二：停在 reset vector，逐段验证交接

::: {.hands-on}
具体目录、调试脚本与验收项见英文手册 [Stop at reset vector](../experiments/part-01-system-foundations/chapter-02-startup-path/stop-at-reset-vector/README.md)。使用 `-S -s` 启动 RISC-V `virt`，先不要执行，读取初始 PC、`a0`、`a1`、`a2` 与 MROM 内容，把反汇编逐条对应到 `riscv_setup_rom_reset_vec()`。随后单步到固件入口，确认 `mhartid`、DTB 地址和动态固件信息；再到内核入口，按 Linux RISC-V 启动要求复核寄存器与页表状态。

实验至少做两组配置：一组使用明确版本的 OpenSBI 固件与内核，一组改变固件或采用手册允许的直接启动方案。记录入口、镜像范围、DTB 与 initrd 地址，不以“串口能打印”代替地址证据。若宿主支持 RISC-V KVM，可增加 KVM 对照；不支持时保留 TCG 结果，并从 `ad40be270` 的提交与当前 KVM 源码说明预期分流点，明确标为尚未实机验证。
:::

实验报告可以画一条很短的交接链：reset PC 指向哪段 ROM，ROM 跳到哪个 firmware entry，firmware 又以什么寄存器状态进入 kernel entry。每个箭头旁放一个来源，分别是当前源码、镜像格式、规范条文或运行证据。若某个箭头只来自推测，用不同标记留下，后续再补断点，而不是为了让图完整而填入未经验证的地址。

## 动态思考题

:::: {.quick-quiz}
如果 QEMU 成功把一个 ELF 固件的所有段复制到客户机内存，却跳到了用户给出的基址而不是 ELF entry，这属于装载错误还是执行错误？

::: {.quick-answer}
复制动作本身可能按段正确，错误出在“按格式解释入口”与“建立执行交接”之间。它说明启动路径不能只用装载成功与否二分，应分别验证范围、入口和寄存器协议。`55c136599` 的修正正提供了这类边界案例。
:::
::::

:::: {.quick-quiz}
把 DTB 放到客户机 RAM 并在 `a1` 写入其地址后，还需要验证哪些内容？

::: {.quick-answer}
要确认地址位于客户机可访问且不与固件、内核、initrd 重叠，DTB 头与大小有效，CPU、内存、设备地址和中断描述与本次运行的实际拓扑一致，还要确认下一启动阶段没有在使用前覆盖这段内存。寄存器正确只完成了指针交接。
:::
::::

:::: {.quick-quiz}
为什么在单 hart 实验中观察到 reset vector 正确，仍不能直接证明 SMP 启动正确？

::: {.quick-answer}
多 hart 还涉及 hartid 编排、非启动 hart 的等待和唤醒、中断上下文以及设备树 CPU 拓扑。单 hart 只验证启动 hart 的局部路径。SMP 结论需要分别观察各 hart 的初始 PC、`mhartid`、固件等待协议和后续唤醒。
:::
::::

## 早期选项与晚期选项为什么不能随意换位

启动编排里有一类选项必须在普通解析之前生效，例如影响帮助、版本输出、模块加载或进程环境的参数；另一类选项只有在 Machine 与对象类型可用后才有意义。若把它们统一塞进一次遍历，看起来代码更整齐，实际会产生循环依赖：解析设备属性需要类型注册，类型模块可能受早期配置影响，Machine 默认又会影响后续设备选择。

`qemu_init()` 因而包含多轮处理。第一轮不必创建机器，只提取足以建立解析环境的内容；中间阶段完成类型、主循环、trace 和基础对象；后续再消费需要完整上下文的配置。多轮扫描会增加读者负担，也有重复解释选项的风险，所以上游重构常把每轮允许的动作写进独立函数。评价时应检查轮次间保存的数据是否明确，不能只以“一遍循环更高效”为目标，启动阶段的这点遍历成本远小于语义错误。

帮助与错误输出也是阶段设计的一部分。`-machine help` 不应先构造一台完整机器，`-device help` 却要知道已注册设备类型；某个不可用 accelerator 的错误要在创建 vCPU 前报告，而某个设备缺少总线只能等 Machine 提供默认总线后判断。不同退出点让自动化脚本依赖的输出保持清楚，也让测试能覆盖“尚未改变外部状态”的失败。

## 默认设备是一段可见策略

`virt` 能用很短的命令启动，是因为 QEMU 会依据 MachineClass 与选项创建若干默认设备或默认后端。便利的另一面是隐式策略：用户再添加同类设备时，QEMU 要避免重复；`-nodefaults` 会关闭一批默认项，却不代表板级必需组件消失；`-nographic` 还会调整串口和显示后端。阅读启动结果时，必须区分板级不可缺的设备、Machine 默认创建的设备，以及用户显式 `-device` 添加的设备。

`qemu_create_default_devices()`、早期和晚期后端创建、命令行设备 realize 等阶段体现了这种区别。一个块后端可以先存在，控制器设备后创建并引用它；一个字符后端也可能在 UART realize 前准备。若先创建前端，属性解析时还找不到后端；若所有后端都无限提前，又可能缺少 Machine 决定的默认连接。阶段划分在这里是一种依赖排序。

默认值一旦被客户机观察，也可能进入兼容范围。新增默认设备会改变设备树、中断分配或枚举顺序，修改默认 CPU 扩展会改变内核探测。上游审查不会只问“功能是否工作”，还会问旧命令行是否出现新硬件、迁移目标是否理解状态。实验为了研究某个设备，最好使用 `-nodefaults` 后显式添加必要前端，但要先确认板级基础组件无法也不应被关闭。

## fw_cfg、ROM blob 与可修改 RAM 的角色不同

QEMU 向客户机交付启动数据并不只靠普通 RAM。MROM 保存 reset vector，flash 可以承载持久固件，fw_cfg 提供一套由固件读取的配置通道，设备树和其他 boot 信息还可能作为 ROM blob 注册。它们在客户机地址空间中的可见方式、复位语义和迁移处理不同。把所有数据都视为“某个 host 指针”，会漏掉写保护、刷新和生命周期。

ROM blob 往往在机器创建阶段由 QEMU 生成或加载，系统 reset 时重新把内容呈现给客户机；客户机写入是否生效取决于区域类型与设备语义。flash 则由设备模型实现擦写规则，固件更新可能持久化到后端。fw_cfg 的数据由索引选择并通过 MMIO 等接口读取，客户机看到的是设备协议，不是直接映射 QEMU 缓冲区。三者都能交付启动材料，却不能互换。

`riscv_rom_copy_firmware_info()` 和 reset vector 构造等函数把固件交接信息安排在 ROM 区域，`virt_machine_init()` 又会在生成 FDT 前后创建 fw_cfg。追踪时要写清数据何时定稿：若 DTB 仍会增加节点，过早复制就会得到旧内容；若 vCPU 已运行，再修改只读启动 blob 也不会自动通知固件。阶段不变量可以表述为“发布给客户机之后，内容与地址必须同时稳定”。

## 镜像布局是一道区间算术题

启动装载最常见的隐患来自区间，而不是单个地址。一个 ELF 有多个段，每段还有文件大小与内存大小；内核可能要求对齐，initrd 需要完整放入 DRAM，DTB 要留出增长空间，固件与 MROM 又占固定窗口。只比较起点，无法发现尾部越界或两个半开区间相交。代码审查时应统一写成 `[start, end)`，并检查加法溢出。

客户机内存大不代表任意镜像都能放下。DRAM 从 `0x80000000` 起，32 位字段、宿主 `size_t` 和客户机物理地址宽度在运算中可能混合；极大 `-m`、稀疏 NUMA 或不连续内存还会破坏“从基址到末尾连续可用”的假设。当前版本装载函数包含面向这些场景的计算，实验用普通 1 GiB 内存成功，不能替代边界条件审查。

出错信息最好包含冲突对象与范围，而不是只报“cannot load image”。开发者才能判断是文件格式、平台限制还是配置组合。若错误发生在部分数据已经写入 ROM 列表之后，回滚还要保证下次 reset 不会复制半成品。启动很少重复执行，不代表清理可以省略，管理工具与测试仍会走失败路径。

## 进入 vCPU 之前需要满足哪些不变量

可以把第一条指令前的条件写成一张检查表。Machine 和所有必需设备已经 realize，系统内存与 I/O 地址空间能查到，CPU reset 状态与加速器后端一致，中断线处于定义好的初始电平，reset vector 目标有可执行内容，DTB 与启动参数地址有效，runstate 允许执行。任何一项缺失，vCPU 线程即使成功创建也不该进入客户机。

这些不变量分属不同组件，QEMU 不会有一个函数逐项重新验证。Machine 装配负责拓扑，设备 realize 负责局部资源，reset 框架负责初态，加速器负责 vCPU 接口，runstate 协调最终放行。设计的难点正是让每层承诺可以组合。读者可以在 `-S` 状态用管理命令、GDB 和内存读取分别抽查，建立“启动完成但未运行”的证据集合。

内存屏障和锁也参与最后交接。创建线程的一方写入状态，vCPU 线程必须在规定同步之后看到；KVM 寄存器可能要显式写入内核；TCG CPU loop 可能缓存某些派生状态。不能因为初始化表面上单线程，就假定启动到运行的发布无需同步。第三章讨论 BQL、kick 和 queued work 时，会把这道门从并发角度重看一次。

## 可复现的启动追踪需要控制变量

一次有用的启动实验至少保存 QEMU 完整版本字符串、构建目标、加速器、Machine/CPU 属性、固件与镜像哈希、命令行、宿主体系结构和 DTB。若比较两个 tag，只改变 QEMU；若比较固件，保持 QEMU 与内核不变；若比较 TCG/KVM，则先确认客户机 CPU 能力和计时差异是否会影响观察。控制变量不需要昂贵设备，却需要纪律。

GDB 也可能改变路径。软件断点会修改客户机内存，硬件断点数量受后端限制，KVM 单步通过内核调试接口实现，TCG 单步则由软件执行循环控制。报告应注明断点类型与加速器，必要时用只读内存检查或 trace 交叉验证。若 GDB 显示的第一条指令与 trace 不同，先确认 attach 前 CPU 是否曾短暂运行。

时间戳不适合单独排列跨阶段事件，因为固件、客户机和宿主可能使用不同计时源。更稳妥的是记录因果标记：QEMU 写好 reset ROM，runstate 切到 running，hart 0 命中固件入口，串口产生首字符，内核命中 `_start`。这些点可以各自带时间，却先靠状态与地址连接，避免把无法同步的时钟强行对齐。

## 从一个失败实验反推阶段

假设 `-kernel` 指定的 ELF 能被识别，QEMU 没报错，OpenSBI 也打印了 banner，随后没有内核输出。第一步不要立刻怀疑串口，因为固件已经使用同一 UART。可以在内核 entry 下断点，若未命中，检查 OpenSBI 得到的 next address 与 ELF entry；若命中，检查 `a0/a1`、`satp` 和 DTB；若这些正确，再进入内核早期页表和 console。每一步都对应一段交接。

若换成 raw 镜像后能够启动，差异集中在格式入口与装载布局；若 TCG 成功而 KVM 失败，检查直接启动、CPU 扩展与内核状态同步；若单 hart 成功而多 hart 失败，检查 hart array、设备树和固件唤醒。这样的差分比漫无目的地打开更多日志有效，因为每次只改变一个阶段相关条件。

最后回到历史查证。若现象像 ELF entry 问题，`git log -S'riscv_load_firmware' -- hw/riscv/boot.c` 和 `55c136599` 会给出已知案例；若问题位于 KVM 入口，则从 RISC-V KVM 启动系列追邮件。历史不能替代复现，却能提示维护者曾经明确保护哪些不变量，并帮助避免重复提出已经被 review 否决的修法。

## 给启动路径画两张图，而不是一张巨图

第一张图只画对象和依赖：Machine 拥有 hart 容器与平台设备，设备连接总线、IRQ 与 MemoryRegion，固件后端和镜像作为输入。它回答“谁必须先存在”。第二张图画时间：解析、创建、属性设置、realize、装载、reset、runstate、执行。它回答“动作何时发生”。把两张图叠成一张，容易把所有权边误认为调用边，也容易把运行期关系误认为初始化顺序。

对每个关键节点可以标一个失败出口。文件打开失败发生在装载，CPU 属性不支持发生在加速器或 realize，地址重叠发生在布局，固件执行异常发生在客户机运行。错误落点若和预期阶段不符，就是值得继续追踪的信号。例如参数拼写错误直到 vCPU 运行才暴露，通常说明验证被推得过晚。

这两张图会在后续章节继续使用。第三章给时间图增加线程和事件域，第四章给对象图增加 class、child、link 与 bus，第五章把地址树和 FlatView 叠到平台节点上。第二章先固定首条指令前的同一台 `virt`，以后每增加一个维度，都能回到这个起点。

## 启动代码如何被自动化验证

手工 GDB 适合理解交接，回归保护还需要自动化。QEMU 的 qtest 能在不依赖完整客户机软件的情况下创建 Machine、访问设备寄存器并检查拓扑，功能测试可以启动固件或内核并等待可辨认输出，单元测试则覆盖纯粹的解析与数据结构。不同测试层对应不同失败范围，不应要求一条慢速系统启动测试承担所有边界条件。

reset vector 可以从两个方向测试。静态一侧检查生成指令与数据布局，运行一侧用最小裸机或固件确认 hartid、FDT 和跳转目标。镜像装载则需要覆盖 ELF entry、段边界、raw fallback、过大 initrd 与重叠错误。只保留“正常 Linux 能启动”的测试，会让格式边界在很久以后由某个特殊固件触发。

测试还要固定观察接口。直接匹配完整 OpenSBI banner 容易随版本变化，等待一个稳定标记或读取寄存器更可靠；检查内部函数调用次数会阻碍重构，检查客户机入口状态则更贴近契约。若补丁只移动 `qemu_create_machine()` 而不改行为，测试应继续通过；若修改 DTB 或默认设备，测试差异应该迫使作者解释客户机可见影响。

上游 review 常会问新路径是否覆盖其他 RISC-V Machine。即便本书只沿 `virt` 展开，阅读补丁也要注意公共 `hw/riscv/boot.c` 会被哪些板级调用。一个为 `virt` 修复的假设若破坏 spike 或真实开发板模型，上游不会接受。公共辅助函数的参数设计，正是在复用与平台差异之间表达边界。

## 启动协议是逐级收窄的承诺

从 QEMU 到 reset ROM，承诺是复位地址有合法指令和板级数据；从 ROM 到 OpenSBI，承诺是 hartid、FDT 与动态信息符合固件接口；从 OpenSBI 到 Linux，承诺又收窄为内核启动文档规定的寄存器、特权级和页表状态。每一级可以做更多工作，却不能让下一级依赖没有记录的偶然状态。

这类逐级承诺解释了为什么“在我的镜像上能跑”不足以证明入口设计正确。镜像可能碰巧忽略错误的 `a1`，单 hart 可能碰巧让数组下标等于 hartid，零初始化内存可能掩盖未设置寄存器。换一个遵守规范但不依赖这些偶然条件的固件，问题才显现。实验应主动改变一个维度，让隐含假设暴露。

协议边界还给调试带来一条实用规则：在交接点检查输入，不要先深入接收方数千行代码。若 Linux entry 的 `a1` 已错，继续追早期设备树解析只会看到连锁故障；若 OpenSBI entry 的动态信息正确，再进入固件内部才有意义。每次检查都能把责任范围缩小到交接前或交接后。

:::: {.quick-quiz}
一份自制裸机镜像启动时从不读取 `a1`，它成功运行能否证明 QEMU 生成并传递的 DTB 正确？

::: {.quick-answer}
不能。镜像没有消费该输入，成功只验证了它实际依赖的 reset PC、指令和少量平台条件。验证 DTB 需要读取地址、校验 blob，并把节点与运行拓扑对照；最好再使用真正依赖设备树的固件或内核完成交叉测试。
:::

## 小结前的检查清单

面对一条新的 `qemu-system-riscv64` 命令，先把隐式环境展开到足以复现：固定 QEMU tag、加速器、Machine、CPU、内存、固件、内核与后端。接着标出每项配置在哪里保存、何时被消费、错误在哪一阶段报告。进入板级代码后，确认 hart、内存、设备、DTB 和 ROM 的数据依赖，最后在 `-S` 状态验证 reset 后的不变量。

然后才让 CPU 前进。以 reset vector、固件入口和内核入口为三个断点，分别核对地址、镜像格式、寄存器和特权状态；多 hart、KVM 或非常规镜像属于新增变量，逐个引入。若当前源码解释机制，提交和邮件解释演进，实验解释本次行为，三种证据就能组成一条不会因函数改名而立刻失效的启动叙事。

## 小结

从命令行到首条指令，中间隔着配置、对象、拓扑和运行状态。后续章节会把这些阶段中的 QOM、内存和 CPU 边界分别展开。

这一长串阶段可以压回一个很实用的问题：下一阶段开始之前，上一阶段承诺了什么。解析承诺配置有类型且来源可追踪，Machine 创建承诺有一个可拥有后续对象的根，板级装配承诺客户机平台的地址与中断一致，装载器承诺镜像范围和入口符合格式，reset 承诺 CPU 与设备处在定义好的初态，runstate 最后决定是否放行。启动故障只要能落到某次承诺前后，就不再是一团“没有输出”。

本章也建立了后续共同使用的三个观测点：`-S` 下已经完成的机器、MROM 中可以反汇编的 reset vector、固件与内核之间可核对的寄存器协议。第三章会问这些动作由哪条线程推进，第四章会问各对象怎样创建和实现，第五章会问镜像、ROM、RAM 与设备地址如何进入统一分派。它们都从同一条首指令路径继续向下，不另起一台抽象机器。

实际排查时，宁可多保存一次版本、入口和寄存器，也不要用后来记忆补齐。当源码、运行结果与预期不一致，先把差异原样留下，再检查二进制是否对应同一 tag、固件是否来自同一构建、加速器是否真的启用。可复现的异常比修饰得很顺的成功故事更有研究价值。这份记录也应进入实验仓库，随 tag 一起留存，方便下一版逐项复核。

若要把本章方法用于新的启动故障，可以先做一次三分钟分类：QEMU 是否在客户机运行前退出，reset vector 是否能取到合法指令，固件入口与内核入口是否分别命中。三个问题把配置/realize、平台交接和客户机软件分开。之后再增加 DTB、initrd、SMP 或 KVM 变量，每次只移动一个条件，失败就能落回明确阶段，而不会从黑屏直接跳到整套内核调试。

发布新的书籍 tag 前，还应在干净环境重跑两组实验，保存 QEMU、OpenSBI 与内核镜像哈希，并检查源码链接仍指向约定 tag。若正式 `v11.1.0` 与 rc0 在启动文件中有差异，先逐项复核入口和布局，再更新正文，不用“通常没有变化”代替实际比较。

复核结果应把未验证环境单列出来，尤其是 RISC-V KVM，不以 TCG 推演冒充硬件运行。这样读者能够重复已完成部分，也知道下一次补证据应从哪里开始。入口、镜像范围和寄存器三项原始数据也要随报告保存。

建议把三个入口附近各保存一小段反汇编和内存摘要，并记录断点命中的是哪个 hart。这样固件升级后，即使串口文字改变，仍能比较控制流与交接参数；若正式 tag 调整了装载辅助函数，也能用运行证据判断客户机可见行为是否真的变化。
