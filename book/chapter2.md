# QOM 对象模型

QOM 为 QEMU 提供动态类型注册、单继承、无状态接口、属性和组合树。它不是独立于 C 的另一种语言，而是一组用 C 实现的运行时约定。

## 本章目标

- 区分 `Object`、`ObjectClass`、实例初始化和 class 初始化；
- 理解继承树、组合树和设备总线拓扑不是同一棵树；
- 能解释属性、link、child 与引用计数各自解决的问题。

## 从 TypeInfo 到实例

梳理类型注册、父类 class 构造、实例分配、instance init、property 设置、realize 和 finalize。说明 class 数据为何共享，而实例数据为何必须独立。

## 类型关系与对象关系

使用一个简化设备说明继承、接口、组合和所有权。分析 `object_dynamic_cast()`、类型宏和自动生成声明在可读性与安全性上的取舍。

## 属性系统

解释静态属性、动态属性、link 属性与 visitor 的关系。讨论属性为什么是控制面接口的一部分，以及属性可修改阶段为何需要严格限制。

::: {.design-note}
QOM 同时承担运行时多态和用户可配置对象模型两种职责。阅读代码时要先问当前调用是在解决“类型分派”还是“对象配置”，否则很容易把 class 回调和实例属性混在一起。
:::

## 对象树的观察方法

介绍 HMP `info qom-tree`、QMP introspection 和代码内 canonical path，说明如何把运行时树映射回初始化代码。

## 小结

QOM 的价值不在于模仿某种面向对象语法，而在于让类型、配置、生命周期和运行时观察遵循统一规则。

## 思考题

1. 为什么接口通常不保存实例状态？
2. composition child 和 link 的所有权语义有何不同？
3. class 初始化为何不能依赖某个具体实例？
