# 主循环、线程与并发上下文

很多 QEMU 问题看起来发生在某个函数里，真正的原因却是函数跑在错误的上下文。主循环、vCPU 线程、I/O 线程、协程和工作线程各自有调度规则，代码能不能阻塞、由谁持有状态、需要哪把锁，都从这里开始。

## 本章目标

- 区分 main loop、AioContext、vCPU thread、coroutine 与 thread pool；
- 理解 BQL、RCU、mutex 和原子操作保护的不同边界；
- 能画出一个跨线程请求从通知到安全点执行的路径。

## 主循环处理什么

主循环等待文件描述符、timer、bottom half 和事件通知，事件就绪后再分派回调。对象删除和 handler 注销必须考虑已经排队的回调，否则一次看似普通的热拔除就可能变成 use-after-free。

:::: {.quick-quiz}
bottom half 与 timer 的调度语义有什么差别？

::: {.quick-answer}
bottom half 表示当前回调之后尽快执行的延期工作，没有独立的时间截止点；timer 绑定某种时钟，达到期限后才具备运行条件。两者都由事件循环分派，但触发依据不同。
:::
::::

## AioContext 是执行域

AioContext 同时组织事件源和并发归属。设备后端迁移到 iothread 后，变化的不只是“换了一条线程”，调用者、锁顺序、回调重入和数据所有权也必须重新检查。

协程让异步状态机保持顺序写法，它仍运行在承载自己的线程上。直接调用阻塞系统调用，会把整个 AioContext 一起堵住。

:::: {.quick-quiz}
为什么协程不能自动把阻塞系统调用变成异步操作？

::: {.quick-answer}
协程只是保存和恢复用户态执行栈，没有新的内核调度实体。系统调用阻塞时，承载协程的线程也会睡眠；需要非阻塞 AIO 或 thread pool，才能让事件循环继续工作。
:::
::::

## vCPU 安全点与跨线程工作

架构状态通常由 vCPU 线程在明确边界上修改。其他线程通过 kick、queued work、`async_run_on_cpu()` 或 `run_on_cpu()` 发出请求，让变更在 TCG 或 KVM 都能接受的安全点发生。

:::: {.quick-quiz}
为什么其他线程不应直接修改 `CPURISCVState`？

::: {.quick-answer}
vCPU 可能正在使用 TCG 临时状态，或把最新寄存器保存在 KVM 内核对象中。直接写结构体会绕过退出、同步和内存序约定，产生数据竞争或让两份 CPU 状态互相覆盖。
:::
::::

## BQL 与 RCU 的历史位置

BQL 曾经为大量设备和全局状态提供粗粒度串行化，后来性能路径逐步移到更明确的锁和 AioContext。RCU 则适合读多写少的数据结构，让热路径读取稳定快照，更新方在 grace period 后再回收旧对象。

::: {.source-path}
从 `util/main-loop.c`、`util/async.c`、`util/aio-*`、`system/cpus.c` 和 `include/qemu/rcu.h` 观察执行上下文。Git 历史研究重点是 BQL 临界区缩小、iothread 接入和 AioContext 所有权变化。
:::

::: {.hands-on}
实验步骤和产物约定见英文手册 [Map thread topology](../experiments/part-01-system-foundations/chapter-03-main-loop-and-threads/map-thread-topology/README.md)。正文用中文记录：分别在单 hart 与多 hart 的 RISC-V `virt` 上采集宿主线程名称、线程 ID、QEMU runstate 和加速器，标出主循环、vCPU、IOThread 与工作线程。若某类线程没有出现，写明缺少的触发条件，不要按概念图补造运行结果。
:::

## 先把“并发”拆成三个问题

读到一个回调时，先问它在哪个宿主线程执行；再问它属于哪个事件域，由谁负责唤醒和轮询；最后问它正在触碰的状态由谁拥有。线程、事件域与数据所有权经常重合，却不是同一回事。主线程可以轮询主 `AioContext`，IOThread 轮询另一个 `AioContext`，vCPU 线程则可能因为一次设备访问暂时进入设备模型；一段协程还会在同一线程上挂起和恢复。

只写“异步执行”会遮住全部细节。异步可能表示回调稍后仍在原线程运行，也可能表示向另一线程发通知，还可能表示任务被线程池接走。三种路径的阻塞规则、锁和对象生命周期完全不同。后面的每个机制都会用同一组问题核对，不把异步当作自动正确的标签。

QEMU 的并发设计还横跨两类负载。vCPU 需要低开销推进客户机并及时响应 kick，I/O 需要等待宿主文件描述符和定时事件，管理面则要求可控的串行化与错误返回。把所有工作放进一个循环最容易保持一致，却会在多核和慢后端上形成瓶颈；把所有工作拆成独立线程，又会让每条共享状态都承担同步成本。当前结构是在两者之间不断移动边界的结果。

## 主循环是一台事件分派器

[`util/main-loop.c`](https://gitlab.com/qemu-project/qemu/-/blob/v11.1.0-rc0/util/main-loop.c) 中，`qemu_init_main_loop()` 建立主循环需要的通知与上下文，`main_loop_wait()` 驱动一次等待和分派。宿主平台不同，底层可能使用 poll、epoll 或相应机制，但上层关心的语义相近：收集文件描述符、定时器、bottom half 与通知源，计算等待时间，睡眠，醒来后分派已就绪工作。

事件循环不能永远阻塞在某个设备回调中。若主循环回调直接执行慢磁盘 I/O，QMP 响应、其他设备 timer 和关机事件都会被拖住；若它持有 BQL，vCPU 或其他需要同一锁的路径也可能停顿。于是，回调要么完成很短的状态推进，要么把长工作交给非阻塞接口、线程池或专属 IOThread，再在完成时通知回来。

`main_loop_wait()` 的 `nonblocking` 参数提醒我们，事件循环有时只允许探测一次，不能睡眠。嵌套等待和外部集成还会对轮询语义提出额外要求。写新代码时若假定“调用一次主循环就一定处理我的 BH”，很容易在没有实际唤醒或上下文不匹配时卡住；应使用对应 API 建立事件，并明确由哪个循环消费。

:::: {.quick-quiz}
主循环已经注册了一个文件描述符，另一个线程向该描述符关联的内核对象写入数据，是否保证回调立即在写线程执行？

::: {.quick-answer}
不保证。写入只会让事件源就绪并唤醒负责轮询的上下文，回调仍由该事件循环所属线程在后续分派中执行。若写线程还持有回调需要的锁，双方可能形成等待，因此必须把通知与回调执行分开分析。
:::
::::

## `AioContext` 同时表达事件集合与执行归属

`AioContext` 容纳文件描述符 handler、EventNotifier、timer、bottom half 和协程调度等事件源。它同时回答“监听什么”和“通常由哪个线程轮询这些事件”。主循环拥有默认的 `AioContext`，用户创建 IOThread 后，块后端或设备数据面可以迁移到另一上下文，由对应线程驱动。

固定 tag 下的 [`Multiple IOThreads`](https://gitlab.com/qemu-project/qemu/-/blob/v11.1.0-rc0/docs/devel/multiple-iothreads.rst) 文档特别提醒，主循环与 IOThread 都使用事件循环，却不能随意互换。主线程还承担 Machine 管理、QMP 和许多受 BQL 保护的设备逻辑；IOThread 面向显式支持 `AioContext` 的 I/O 路径。把一个旧设备简单绑定到 IOThread，如果内部仍访问只在 BQL 下安全的状态，性能问题会立刻变成数据竞争。

上下文切换真正要完成的是：停止旧上下文的事件、迁移 handler、更新所有权，再让新上下文开始轮询，给指针赋值只占很小一步。迁移期间排队的完成事件、协程和 BH 必须有确定去处。块层常用 drain 等机制把在途请求收束到可迁移状态；设备热拔除也要先阻止新请求，再等待旧回调不再引用对象。

[`util/async.c`](https://gitlab.com/qemu-project/qemu/-/blob/v11.1.0-rc0/util/async.c) 中的 `aio_context_new()` 创建上下文，`aio_notify()` 负责在需要时唤醒轮询者。通知只是提示“状态可能变了”，消费者应重新检查条件，不能把每次通知等同于恰好一个请求。合并通知能减少系统调用，也意味着算法必须容忍多次生产只收到一次唤醒。

## EventNotifier 是跨上下文的门铃

跨线程生产者常需要让睡眠中的事件循环醒来。EventNotifier 在宿主上通常借助 eventfd 或平台等价物，把“有工作”变成可轮询事件。生产者先发布队列或状态，再敲门；消费者醒来清除通知并读取队列。正确性来自状态本身和内存序，门铃只负责唤醒。

如果反过来先通知、后写队列，消费者可能醒来看到空队列，再次睡眠，而真正请求已经写入却没有新门铃。若每个请求都要求一次通知，又会在高并发下制造无谓系统调用。常见设计用原子状态从“空”切到“非空”时通知，消费者循环取尽工作，并在重新睡眠前处理并发生产的竞态。

这条模式也适用于 vCPU kick。其他线程设置退出或工作标志，再通过加速器相应方式让正在 TCG 或 KVM 中运行的 vCPU 返回安全点。kick 不是远程执行函数，它只保证 CPU 有机会观察请求。把门铃当成工作本体，是阅读并发代码最常见的误区之一。

## Bottom half：把工作推迟到当前事件域

BH 适合表达“现在不方便做，回到此 `AioContext` 的分派阶段尽快处理”。设备回调可以设置状态并 schedule BH，避免在深层调用栈中继续触发另一套设备逻辑；其他线程也可用 `aio_bh_schedule_oneshot()` 把一次性工作送到目标上下文。BH 没有独立截止时间，只在事件循环获得执行机会后运行。

schedule 同一个 BH 多次通常会合并为待处理状态，这对减少重复工作很有用，也要求回调从共享状态重新计算，而不是假定每次 schedule 对应一个事件。删除 BH 时还要考虑它是否已经排队或正在执行。对象 finalize 直接释放 opaque，而排队回调稍后仍解引用，就会形成 use-after-free。

BH 还常用于打断不安全的重入调用链。设备 MMIO 回调若立刻发起 DMA，DMA 又访问另一个设备甚至回到自己，原来假定线性的寄存器更新会被嵌套打断。推迟工作可以让当前操作先完成，却不自动解决生命周期和锁；回调执行前对象仍要存活，状态机仍要能辨认请求是否已取消。

:::: {.quick-quiz}
同一个 BH 被连续 schedule 十次，回调是否必然执行十次？

::: {.quick-answer}
通常不能这样假定。BH 调度可以合并待处理标志，语义更接近“请在目标上下文重新检查工作”。需要逐项处理的请求应进入队列，BH 回调循环消费；通知次数只用于唤醒，不能充当请求计数。
:::
::::

## Timer：时间到达后才具备运行条件

timer 与 BH 都由事件循环分派，差别在触发依据。timer 绑定 QEMU 的某类时钟和到期值，时钟到达后才 ready；BH 一旦被 schedule 就等待尽快运行。设备定时器必须选择合适时钟：虚拟时钟通常随客户机暂停而停，实时或宿主时钟有不同语义。选择错误会让迁移、暂停和确定性重放出现奇怪漂移。

到期时间也不等于精确执行时间。事件循环繁忙、线程被宿主调度延迟或锁被占用时，回调会晚于 deadline。设备模型应根据规范决定是累计多个 tick、只更新到当前时间，还是报告丢失事件，不能假定 timer 每隔固定宿主时间必然调用一次。

重置与删除 timer 同样涉及在途回调。对象 unrealize 时先取消未来 timer，并确保没有其他路径再次启动它；reset 时则按设备规范恢复 deadline 和寄存器。若 timer 回调从 IOThread 访问由主线程拥有的设备状态，还要显式跨上下文，而不是因为“只是计时”就跳过同步。

## Coroutine 让状态机写得顺，线程没有变

协程把“发起 I/O—等待—继续处理”的状态保存到独立栈上，使代码保持接近顺序流程。遇到等待条件时，协程 yield，承载线程回到事件循环；完成事件到达后，协程被重新 schedule，在合适上下文继续。它减少了手写回调状态机的分裂，不会创建新的内核线程。

因此，在协程中调用真正阻塞的宿主系统调用，仍会让承载 `AioContext` 的线程睡眠。要处理阻塞文件系统或库函数，需要使用线程池、异步内核接口或其他后端。协程也不能免除锁：它在 yield 处让出执行，恢复前其他协程和回调可能改变状态，跨 yield 持有普通 mutex 还可能造成长时间阻塞。

协程的上下文归属必须明确。一个协程在主 `AioContext` 创建，随后想在 IOThread 恢复，需要使用相应 reschedule 机制，而不是从另一线程直接 enter。当前 `util/async.c` 提供 `aio_co_schedule()`、`aio_co_reschedule_self()` 等路径，正是为了让恢复动作经过目标事件域。

## Thread pool 处理无法异步化的阻塞工作

某些宿主操作没有好用的非阻塞接口，或执行时间不可预测。线程池让事件循环提交任务，工作线程执行阻塞操作，完成后再向原 `AioContext` 报告。这样主循环不必因一次磁盘操作停住，也避免为每个请求创建线程。

提交任务时要冻结输入或转移所有权。设备若把指向可变请求结构的裸指针交给线程池，主线程随即 reset 或删除对象，工作线程会访问无效内存。取消不能只丢掉完成回调：底层系统调用可能已经执行，结果仍需回收，客户机可见状态则要按 reset/错误语义处理。

线程池能增加并行度，也会引入排队和无序完成。设备协议若要求请求顺序，需要在提交前串行、在完成时重排，或利用请求中的 barrier。把所有任务丢进池中并不会自动提高性能，锁争用、宿主 I/O 深度和缓存局部性都可能让吞吐下降。

## BQL 提供粗粒度串行化，但范围正在被明确

BQL，Big QEMU Lock，为大量传统设备、主循环控制和全局状态提供共同串行化边界。它使早期单线程假设能够在多 vCPU 和异步 I/O 环境中继续工作，开发者不必一开始给每个字段设计细锁。代价是只要长回调持锁，其他需要 BQL 的 vCPU 退出、管理操作和设备路径都会等待。

当前源码通过 `bql_lock()`、`bql_unlock()` 等接口明确表示持锁区间。主循环在等待宿主事件时会释放 BQL，醒来后再按需要取得；启动完成进入主循环前，也会从初始化持锁状态切换。不能把 BQL 想成“QEMU 进程从头到尾只有一条线程运行”，它是特定共享状态的粗锁，很多 IOThread 与执行路径已经在其外工作。

2024 年提交 [`195801d700c`](https://gitlab.com/qemu-project/qemu/-/commit/195801d700c) 把历史名称 `qemu_mutex_lock_iothread()` 等改为 BQL 命名，对应 qemu-devel Message-ID [`20240102153529.486531-2-stefanha@redhat.com`](https://lore.kernel.org/qemu-devel/20240102153529.486531-2-stefanha%40redhat.com/)。提交说明指出，“iothread” 源于早期 KVM vCPU 与主循环线程分离的叫法，后来又出现 `--object iothread`，旧名容易混淆。重命名没有改变锁语义，却把线程身份与锁概念拆开，给阅读带来很实在的收益。

::: {.design-note}
已核对事实是当前接口使用 BQL 名称，上游提交明确把旧名称的歧义作为重命名理由。可以进一步推断，清楚命名为缩小临界区和审查无 BQL 路径提供共同语言；但重命名本身不能证明任何设备已经细粒度化，具体路径仍要检查锁断言与执行上下文。
:::

## vCPU 线程把客户机并行映射到宿主执行

在常见系统模拟配置中，每个 RISC-V hart 对应一个 `CPUState`，运行时由 vCPU 线程推进。KVM vCPU 线程通过 ioctl 进入内核，返回时处理 exit；MTTCG 配置下多个 TCG vCPU 也可并行执行翻译代码。宿主调度、加速器模式和 QEMU 配置会影响映射，因此“一个 hart 就永久等于一个宿主核心”并不成立。

vCPU 会在设备访问、异常、时间片、kick 和 queued work 等边界回到 QEMU 控制层。设备模型有的仍在 vCPU 线程里执行，有的把工作送往主循环或 IOThread。看到函数来自 `hw/` 目录，不能推断它一定跑在主线程；一次 KVM MMIO exit 很可能让对应 vCPU 线程直接进入内存分派与设备回调，并按规则取得 BQL 或设备锁。

这也是多 vCPU 才暴露许多竞态的原因。两个 hart 可以同时访问同一 PLIC 或 virtio 状态，设备模型必须依靠 BQL、内部锁、原子操作或事件域所有权保持语义。单 hart 测试通过，只证明没有遇到并行访问；章节实验会比较一个与多个 hart 的线程拓扑，为后续设备竞态建立基线。

## `run_on_cpu()` 把修改送到架构安全点

`system/cpus.c` 的 `run_on_cpu()`、`async_run_on_cpu()` 与 `qemu_cpu_kick()` 共同表达一种模式：其他上下文先把工作排到目标 `CPUState`，再唤醒或迫使 vCPU 退出，让回调在 CPU 线程的安全位置执行。同步版本会等待完成，异步版本只排队，调用者的阻塞与锁约束不同。

为什么不能直接改 `CPURISCVState`？TCG 可能正使用翻译器假定的临时或缓存状态，KVM 的权威寄存器可能位于内核 vCPU；CPU 线程还可能正在处理 reset、异常或调试同步。queued work 把修改放在已有同步协议内，确保先退出执行后端，必要时同步状态，再调用架构回调。

同步等待尤其容易死锁。调用者若持有 BQL 或设备锁，而 vCPU 需要同一把锁才能到达 queued work，就会互相等待。使用 `run_on_cpu()` 前要阅读接口对 BQL 的要求，确认目标 CPU 是否已停止，并避免从目标 vCPU 自己等待自己。异步版本避免等待，却要求参数在回调前保持有效。

:::: {.quick-quiz}
把一个回调排给 vCPU 后立即释放其参数，异步路径为何可能出错，而同步路径通常不会？

::: {.quick-answer}
异步排队只保证未来执行，返回时回调可能尚未读取参数，调用者释放后会形成悬空引用。同步 `run_on_cpu()` 返回前等待回调完成，因此栈上或短生命周期参数通常仍有效；不过仍要遵守接口的锁和目标线程要求。
:::
::::

## 原子操作解决单次竞争，不能自动保护不变量

原子读写和 compare-exchange 适合标志、计数和指针发布，能防止数据竞争并表达内存序。它们不能自动让多个字段形成一致快照。若设备状态要求“队列指针、长度和 enabled 必须一起变化”，分别原子写三个字段，读者仍可能看到混合版本。此时需要锁、序列计数、不可变快照或把更新限制在一个事件域。

内存序比“是否原子”更细。生产者先填请求再原子发布指针，消费者 acquire 读取后才能保证看到内容；只用 relaxed 可能在弱内存序宿主上暴露未完成写入。QEMU 的原子 API 文档给出屏障与访问约定，RISC-V 客户机内存模型又是另一层，不能拿客户机 `fence` 代替宿主 C 代码同步。

原子变量也需要生命周期保护。一个指针可以被原子读到，目标对象却可能在下一刻释放；RCU、引用计数或锁要保证解引用期间对象存活。第五章的 FlatView 正是“原子发布新指针，加 RCU 延迟回收”的组合，而不是单靠一次原子赋值。

## RCU 为读多写少的结构保存旧世界

QEMU 的 [`RCU` 文档](https://gitlab.com/qemu-project/qemu/-/blob/v11.1.0-rc0/docs/devel/rcu.rst) 约定，读者在 `rcu_read_lock()` 与 `rcu_read_unlock()` 之间读取稳定对象，更新方发布新版本，再用 `call_rcu()` 把旧对象回收推迟到 grace period 之后。读侧避免重锁，适合内存映射、RAMBlock 列表等高频读取、低频修改的数据。

RCU 保护的是生命周期和发布顺序，不负责让对象内部任意字段都可并发修改。常见模式是构造不可变新快照，原子替换指针，旧读者继续看旧快照；若读者还会修改对象，就需要额外同步。离开 read-side 临界区后也不能继续保存裸指针，除非取得引用或有另一项生命周期保证。

更新方通常仍需要锁来串行多个 writer。官方 RCU 文档明确把“删除、发布、回收”分成不同阶段，BQL、BH 或子系统 mutex 可以保护 writer，RCU 让 reader 不必跟它们竞争。把 RCU 描述成“无锁所以什么锁都不需要”，会把最困难的 writer 冲突和对象回收隐去。

## 设备重入：同一线程也可能破坏并发假设

并发问题不只发生在两条线程。设备 MMIO 回调可以触发 DMA，DMA 再进入另一设备的 MMIO，形成同一线程上的嵌套调用；BH 在回调边界执行，也可能重新进入设备。若外层正在修改队列或释放描述符，内层看到半完成状态，结果与真正并发一样危险。

2023 年提交 [`a2e1753b805`](https://gitlab.com/qemu-project/qemu/-/commit/a2e1753b805) 针对 DMA 重入风险增加防护，相关系列的 [`9c86c97f12c060`](https://gitlab.com/qemu-project/qemu/-/commit/9c86c97f12c060) 又把 BH 回调纳入 `MemoryReentrancyGuard`，邮件 Message-ID 为 [`20230427211013.2994127-3-alxndr@bu.edu`](https://lore.kernel.org/qemu-devel/20230427211013.2994127-3-alxndr%40bu.edu/)。系列与 CVE-2023-0330 相关，说明回调链和 DMA 边界也是安全边界。

这段历史不能简化为“加一把锁”。重入发生在同一线程，普通非递归锁可能直接死锁，递归锁又允许不变量被中途观察。guard 的作用是识别不允许的嵌套设备访问并阻断危险路径。具体设备仍应缩小回调中的半完成状态，明确哪些操作可以重入，哪些必须推迟。

::: {.design-note}
上游补丁与安全公告可以直接支持“BH、DMA 与 MMIO 的嵌套路径曾形成可利用重入问题”。作者据此推断，QEMU 选择通用 guard 是为了在设备逐个修复之外建立防线；不过，guard 不是设备状态机正确性的替代品，也不证明所有形式的重入都被统一禁止。
:::

## QMP 命令怎样进入共享状态

大多数 in-band QMP 命令由主线程处理，并在需要时处于 BQL 保护范围。查询命令可能读取多个子系统，修改命令可能请求暂停、热插拔或迁移。命令 handler 即使写得很短，也可能调用异步 job，再通过事件向管理端报告完成。管理请求与实际数据面动作要分开。

查询一致性也有层次。暂停虚拟机后读取 CPU 与设备状态，可以获得较强停机快照；运行中查询计数器，字段可能在读取过程中变化。QAPI 类型确保数据格式，不自动提供事务快照。接口文档若未承诺原子一致，客户端不应把两个字段的瞬时关系当作严格不变量。

管理命令跨到 vCPU 时，会使用 pause、kick 或 queued work；跨到 IOThread 时，需要在目标 `AioContext` 调度并等待或异步返回。直接从 QMP handler 解引用另一事件域拥有的可变对象，会让偶发竞态只在自动化管理下出现。画一条 QMP 时间线，常能解释“压力测试稳定，监控一开就卡”的现象。

## 锁的选择从被保护的不变量出发

BQL 适合大量遗留共享状态与控制面串行化；子系统 mutex 适合多个线程需要较长时间修改一组字段；spin 或原子适合极短热路径；RCU 适合读取快照；AioContext 所有权则通过“只有一条线程修改”减少锁。选择机制之前，必须先写清不变量、读写频率、阻塞能力与生命周期。

锁顺序比锁数量重要。主线程持 BQL 等待 IOThread，IOThread 完成回调又想取 BQL，就会死锁；vCPU 持设备锁同步等待主线程，主线程正在暂停 vCPU，也会形成环。审查时画等待边，节点是执行上下文，边上写持有和请求的锁，比罗列函数中的 mutex 更容易发现环。

条件变量还要求循环检查谓词，因为唤醒可能虚假，状态也可能在重新取锁前变化。EventNotifier 同样只表示“可能有工作”。QEMU 大量通知代码遵循这条原则，读者看到 `wait` 与 `notify` 时，应寻找真正的共享谓词，而不是把通知次数配对。

## 关闭、reset 与热拔除是并发生命周期测试

正常 I/O 路径只需让请求向前，关闭路径要阻止新请求、唤醒等待者、取消 timer/BH、等待线程或协程退出，再释放后端和对象。顺序错一项，平时稳定的设备会在关机或错误回滚中出现 use-after-free。热拔除更难，因为虚拟机其他部分仍在运行。

reset 不一定销毁对象，却会让在途请求与设备状态重新对齐。工作线程可能正在写入缓冲区，vCPU 可能刚触发 MMIO，timer 已经到期但尚未分派。设备需要定义这些请求是完成、取消还是在 reset 后报告错误，并确保旧完成不会污染新一代状态。常见做法引入 generation、drain 或 guard，具体语义取决于设备协议。

线程 teardown 还要区分请求停止与确认停止。设置布尔值后立刻释放线程使用的上下文是不够的，必须通知睡眠线程并 join，或通过明确所有权保证它已不再访问。异步 job 若由全局管理器持有，则可能比设备对象寿命更长，需要独立引用与取消协议。

## 用 trace 重建时间线

一次跨上下文分析至少记录事件名、宿主线程 ID、目标 `AioContext`、客户机 CPU/hart、对象标识和关键状态。时间戳可以辅助排序，但通知与回调之间的因果关系更重要。为请求分配递增 ID，能把“主线程提交—IOThread 执行—vCPU 收到完成中断”连成一条链。

trace 点应放在状态改变前后与跨域边界，而不是每个函数入口。提交队列、敲 EventNotifier、消费队列、完成 BH、注入 IRQ 是高价值位置；一个只打印函数名的日志无法判断请求是否同一个。日志里避免直接保存会很快复用的裸指针作为唯一 ID，可以配合对象路径或 generation。

调试构建还可加入线程归属断言、BQL 持锁断言和锁顺序检查。断言失败比沉默竞态更容易定位，但只在调试配置出现的时序也要复核。最终结论应由不依赖大量日志的最小实验重现，防止 trace 扰动把问题隐藏或制造出来。

## 实验二：追踪主循环事件与一次跨线程请求

::: {.hands-on}
按照英文手册 [Trace main loop events](../experiments/part-01-system-foundations/chapter-03-main-loop-and-threads/trace-main-loop-events/README.md) 准备环境，中文报告选择一次 QMP 查询、一个设备 timer 或 BH，以及一项 vCPU queued work。使用已有 trace event 或最小临时插桩记录宿主线程 ID、AioContext、BQL 状态与请求 ID，把“产生状态、发通知、消费状态、完成反馈”四个点连起来。

先在单 hart TCG 上得到基线，再增加第二个 hart；具备 RISC-V KVM 宿主时加入 KVM 对照。不要比较总性能，本实验只验证执行上下文。若回调在预期之外的 vCPU 线程执行，沿内存分派或 KVM exit 解释；若 BH 没有运行，检查它被 schedule 到哪个 AioContext、该上下文是否有人轮询、对象是否已取消。报告最后列出每个共享字段的所有者和同步机制。
:::

把结果画成泳道图通常最清楚：主线程、vCPU0、vCPU1、IOThread 和工作线程各一条泳道，通知画虚线，函数执行画实线，等待段标出正在持有的锁。若图上出现一条线程持锁等待另一条，而另一条回程又请求同一锁，就得到可验证的死锁假设。没有闭环等待，也不能直接宣布安全，还要检查生命周期与内存序。

## 动态思考题

:::: {.quick-quiz}
某设备的所有回调都在同一个 IOThread 上运行，是否可以删除设备内部的全部同步？

::: {.quick-answer}
不能直接删除。还要确认 reset、热拔除、迁移、QMP 查询和 vCPU 快路径是否也只通过该 AioContext 访问状态，工作线程完成是否正确回送，以及对象生命周期是否受保护。单一数据面线程可以减少锁，但不自动覆盖控制面和重入。
:::
::::

:::: {.quick-quiz}
RCU 读侧没有取得 mutex，是否意味着读者能看到更新中途的任意字段组合？

::: {.quick-answer}
取决于发布模式。若更新方先完整构造不可变新对象，再原子发布指针，读者会看到旧版或新版，不会看到构造中状态；若更新方原地修改多个字段，RCU 本身不提供一致快照，仍需其他同步。RCU 主要保证发布与延迟回收。
:::
::::

:::: {.quick-quiz}
一个协程在 IOThread 中 yield，主线程能否直接调用它继续执行？

::: {.quick-answer}
不应绕过所属上下文直接恢复。协程的恢复要经过目标 AioContext 的调度机制，确保线程归属、并发和栈状态符合约定。主线程可以向该上下文发通知或 schedule 工作，再由 IOThread 恢复协程。
:::
::::

:::: {.quick-quiz}
为什么一次同线程的 DMA 重入也要按并发问题审查？

::: {.quick-answer}
外层回调的不变量尚未恢复，内层调用就观察或修改同一状态，效果等同于在临界区插入另一执行者。普通线程互斥甚至无法帮助，同一线程取得非递归锁会死锁；需要重入策略、推迟执行或专门 guard。
:::
::::

## IOThread 不是“万能后台线程”

用户通过对象配置创建 IOThread 后，得到的是带事件循环与 `AioContext` 的宿主线程。只有明确支持上下文迁移的设备或后端，才能把相应数据面放进去。某个设备有 `iothread=` 属性，也不表示它的全部控制面都移走：创建、realize、reset、迁移和 QMP 属性处理仍可能在主线程，IRQ 注入与 vCPU 访问也会跨过边界。

设备选择 IOThread 时要考虑共享程度。把两个高负载设备放在同一上下文，可以减少线程数和部分锁，却会互相影响延迟；分到两个 IOThread 能增加并行度，又增加跨线程通知、宿主调度和内存占用。如果它们最终争用同一个块后端或 BQL，增加线程不会带来吞吐。配置建议必须基于事件与锁的测量，不能从设备数量直接推导线程数量。

亲和性同样属于部署层而非 QEMU 语义。把 IOThread 与 vCPU 绑定到宿主核心，可能改善缓存和尾延迟，也可能与宿主 IRQ、KVM 线程或其他虚拟机冲突。书中实验记录亲和设置，却不把某一台服务器的最佳绑定写成架构规则。先确认工作确实在目标线程，再讨论绑定，否则只是移动了空线程。

## 忙轮询用 CPU 时间交换唤醒延迟

事件循环通常在没有工作时睡眠，内核唤醒会带来调度延迟。AioContext 支持在一定条件下 poll，让线程短时间主动检查事件，减少高 IOPS 场景的睡眠与唤醒。代价是空闲时仍消耗 CPU，并可能挤压 vCPU。poll 参数因此是一项运行负载相关的权衡，不是总应打开的加速开关。

忙轮询还会改变观测。宿主 `top` 显示 IOThread 高占用，可能是配置预期，而不是死循环；总吞吐上升但同机其他 VM 尾延迟变差，也可能来自 CPU 争用。评估时同时看请求延迟、吞吐、宿主 CPU 时间和调度，保持 vCPU/IOThread 亲和与负载一致。

实现上，poll 前后仍要正确处理并发通知，避免生产者认为线程睡眠而敲门、消费者却刚从 busy loop 转入等待时丢失唤醒。`aio_notify()` 与上下文状态协同处理这类窗口。读代码时应沿“决定睡眠—发布睡眠状态—生产者通知—再次检查条件”完整闭环，不要只看底层 poll 系统调用。

## TCG 与 KVM 的退出方式塑造安全点

TCG 执行翻译块时，QEMU 可以通过退出请求、翻译块边界和帮助函数回到 CPU loop。太频繁检查会降低性能，检查太少又让暂停、定时器和 queued work 响应变慢。翻译块长度、异常和 I/O 自然形成边界，外部 kick 则让正在运行的 CPU 尽快注意退出条件。

KVM vCPU 进入内核后，用户态不能靠修改一项普通 C 字段让它立刻返回，必须使用 KVM 支持的 kick/中断机制，随后处理 ioctl 的退出与寄存器同步。宿主信号、immediate exit 或架构接口的具体实现属于 KVM 后端，公共 `qemu_cpu_kick()` 隐藏差异。调用者依赖的是“最终到达可处理工作的位置”，而不是某一种宿主机制。

到达安全点并不代表所有状态已经自动同步。某些 KVM 寄存器按需读取，某些 TCG 派生状态要在特定回调刷新。queued work 的具体函数必须使用架构提供的同步接口。仅仅证明回调运行在 vCPU 线程，不足以证明直接改字段正确。

## 暂停所有 vCPU 是一个分布式握手

管理面请求 stop 时，要让每个正在不同位置运行的 vCPU 离开执行后端，观察停止请求并确认静止。主线程随后才能依赖“客户机不再改变 CPU/设备状态”的条件执行迁移快照、reset 或调试查询。多 hart 环境中，这是一组线程的握手，不能靠设置一个全局布尔值立即成立。

vCPU 可能在 KVM ioctl、TCG 代码、设备回调或等待状态，kick 路径要覆盖这些位置。停止方还要避免持有 vCPU 返回时需要的锁。确认阶段常通过条件变量或计数跟踪，等待必须在释放适当锁后进行。若一颗 CPU 卡在持 BQL 的长设备回调，全局暂停延迟就暴露了那个临界区。

恢复运行是反向发布：先完成需要静止状态的修改，再改变 runstate 并唤醒 vCPU。调试器若在暂停时改寄存器，KVM 后端需要在再次进入内核前写回。迁移目标端创建 vCPU 后也要先恢复全部状态，再允许任何 hart 执行，避免一个 hart 提前观察尚未恢复的设备。

## 确定性重放让时间与并发边界更严格

QEMU record/replay 需要把非确定输入和事件顺序记录下来，使再次执行能在相同客户机点重现。宿主定时器、异步 I/O 完成、网络输入和线程调度天然不确定，必须在可控边界注入。这个机制提示我们：事件“发生在宿主几点”与“客户机在哪个执行点观察到”是两种时间。

普通运行可以允许某个 timer 晚几毫秒，只要设备语义仍正确；确定性重放则要控制它相对客户机指令的顺序。TCG 更容易提供指令计数与软件边界，KVM 的硬件执行不一定具备相同重放能力。功能范围应以当前文档与配置检查为准，不应看到公共 runstate 接口就假定两种加速器完全等价。

实验中即使不用 replay，也可以借用它的思路：把外部输入、通知、客户机观察点分开记录，减少单纯依赖宿主时间戳。并发 bug 若能转化成明确事件序列，就更容易用最小测试反复触发。

## RCU 的 grace period 依赖线程合作

读者进入 RCU read-side 临界区后，旧对象不能回收；所有先前读者离开后才经过 grace period。QEMU 创建的参与线程需要按 RCU API 注册，并在合适位置报告 quiescent state。线程若长期停留在临界区或没有参与协议，会拖延回收，表现为内存增长或更新方积压，而不一定立即崩溃。

因此，read-side 区域应尽量短，不在其中执行不受控阻塞。需要把对象带出临界区时，取得引用或复制必要数据。嵌套 `rcu_read_lock()` 可以由实现支持，却不应被用来掩盖所有权不清；最外层何时退出仍决定回收进度。

`call_rcu()` 的回调在专门执行环境中运行，回调能否取得 BQL、是否可以阻塞，有自己的约束。释放函数若触发复杂设备操作，可能把回收线程带进意外锁顺序。安全做法是让 RCU 回调只完成轻量释放，复杂 teardown 在发布旧对象之前由拥有者处理。

## 并发性能要按等待图解释

CPU profile 显示某个 mutex 函数占比高，只说明线程花时间在锁附近，不直接说明锁粒度就是根因。可能是持锁者做了阻塞 I/O，可能是过多跨线程通知制造抖动，也可能是客户机本来就串行访问同一设备。需要把等待者、持有者、临界区工作和请求率放进同一张图。

BQL contention 可以从 vCPU 停顿、主线程长回调和 IOThread 回程三处观察；AioContext 瓶颈则看单线程事件积压、BH/timer 延迟和 poll 行为；缓存行争用可能不表现为睡眠，却会让原子热点拖慢多 vCPU。不同问题需要不同工具，`perf lock`、trace、调度统计和 QEMU 内建事件各自覆盖一段。

优化前先建立正确性不变量。把设备从 BQL 移到细锁，若遗漏 reset 或迁移访问，会获得漂亮基准和罕见损坏；把完成批量化，若中断合并违反客户机可见语义，同样不可接受。上游审查通常要求测试和迁移说明，性能数字只能证明收益，不能证明边界完整。

## 用 sanitizers 和断言补足时序实验

ThreadSanitizer 能发现部分数据竞争，但 QEMU 的原子、协程、手写同步和外部内核状态会产生误报或盲区。AddressSanitizer 更擅长捕捉并发生命周期最终导致的越界与释放后使用。工具结果是线索，需要结合 QEMU 同步约定判断，不能把“TSan 没报告”写成线程安全证明。

线程归属断言往往更直接。若某结构约定只在一个 `AioContext` 修改，在入口检查当前上下文；若函数要求 BQL，在调试构建检查持锁；若回调不得重入，使用 guard 并在违规时输出对象路径。断言把架构约定变成可执行文档，也能在后续重构中快速指出边界被跨越。

压力实验应同时覆盖启动、持续 I/O、reset、暂停恢复和热拔除。纯吞吐运行让对象一直存活，最容易错过取消与 teardown 竞态。随机插入管理动作能扩大时序空间，但失败后必须保存随机种子、命令和事件序列，否则难以转成可修复案例。

## 从历史演进中区分名称变化与语义变化

BQL 重命名主要消除术语歧义，DMA 重入系列则改变安全检查，两类提交的研究方法不同。前者要确认调用点机械迁移、接口语义是否保持；后者要重建可触发路径、受影响设备和阻断策略。看到大 diff 不能自动判为架构重写，看到小 diff 也不能低估安全影响。

查历史时，先用当前符号找到 `git log -S` 与 rename commit，再跨过机械变化继续追旧名。qemu-devel 系列的 cover letter 常会列出前一版反馈，单个 patch 的 Message-ID 则能定位具体 review。若正文引用“iothread 名称混淆”，可以直接依赖 `195801d700c` 的提交说明；若进一步讨论为何某设备仍需 BQL，就要找该设备自己的演进证据。

当前代码还可能包含邮件讨论之后的再修改。最终判断必须回到 `v11.1.0-rc0`：guard 是否仍在回调周围，BQL API 是否已经全部换名，`run_on_cpu()` 的锁要求是什么。历史负责解释路径，当前 tag 决定本书实验实际遵守的契约。

## 客户机内存序与宿主同步不能混成一层

RISC-V 客户机通过 load/store、原子扩展和 `fence` 建立 hart 之间的顺序，这些语义由 TCG 或 KVM 实现。QEMU 设备模型同时运行在宿主 C 内存模型下，使用 qatomic、mutex、RCU 与事件通知协调宿主线程。客户机执行了一条 `fence`，不意味着任意 QEMU C 字段自动受到宿主 happens-before 保护；设备模型使用 release/acquire，也不能替客户机驱动补上一条规范要求的 fence。

两层会在共享队列相遇。客户机驱动先写描述符，再更新可见索引并通知设备；设备后端读取索引后，必须按 virtio 等协议规定看到描述符。TCG 要保存客户机内存序，KVM 依赖硬件与内核，QEMU 后端访问客户机 RAM 时还要使用正确 barrier。任一层遗漏，问题可能只在弱内存序 RISC-V 宿主、多 vCPU 或高压力下出现。

分析这类 bug 时画两条 happens-before 链。第一条完全位于客户机 ISA 与设备协议，第二条位于宿主线程、原子变量和通知 API，再标出二者通过哪片共享内存或 MMIO 寄存器连接。直接写“这里缺屏障”信息不足，要说明缺的是客户机驱动屏障、QEMU 编译器/CPU 屏障，还是内核接口约定。

## 中断注入是一条典型的跨域回程

设备完成请求后，通常更新状态并抬高中断。若设备数据面在 IOThread，RISC-V 中断控制器和 vCPU 可能处在另一执行上下文；KVM 还可能把部分中断状态放在内核。完成回调不能只写一个 CPU 字段，它要通过 qemu_irq、irqchip 或架构接口，把电平或脉冲传播到正确上下文。

电平中断要求状态可重建：设备保持条件，中断控制器看到线为高，客户机处理并清除原因后设备拉低。脉冲中断则依赖边沿不被合并。跨线程通知若把两次变化折成一次，必须确认设备协议允许；迁移时还要保存足够状态，让目标端恢复后中断不会丢失或重复。

在 `virt` 主线上，可以选择 UART 或 virtio 完成作为起点，沿 qemu_irq 到 PLIC/AIA，再到 RISC-V CPU 的 pending 状态。TCG 与 KVM 的尾部可能不同，前端设备条件应一致。给这条回程加请求 ID 与 hart 目标，能把“设备完成了但客户机没醒”拆成状态更新、中断路由、vCPU kick 和客户机 enable 四段。

## 运行中查询得到的是哪一种快照

`info` 或 QMP 输出看似同时出现，生成过程却要逐个读取对象。若 vCPU 和 IOThread 仍在运行，前半字段与后半字段可能属于略微不同的时刻。对计数和诊断这通常可接受，对迁移一致性或证明不变量则不够。读者需要区分近似观测、加锁快照、暂停快照和 RCU 版本快照。

暂停所有 vCPU也不一定让全部外部工作线程静止，块请求和网络后端可能继续完成。迁移框架会使用 drain、设备回调和状态机建立更强条件，普通查询没有自动获得同等保证。实验若要求“同一时刻的设备与 CPU 状态”，应明确执行 pause、等待后端收束，再采集，并记录这一动作本身会改变时间相关状态。

反过来，运行中轻量查询更接近真实系统，也更适合观察吞吐和队列变化。两类快照各有用途。报告不要把暂停后看不到竞态当成问题不存在，也不要把两个运行中字段短暂不一致当成设备错误，先核对接口承诺和采集方法。

:::: {.quick-quiz}
客户机 RISC-V 驱动已经执行规范要求的 `fence`，为何 QEMU 的两个宿主线程之间仍可能需要 release/acquire？

::: {.quick-answer}
客户机 `fence` 约束的是客户机可观察的访存顺序，QEMU 宿主线程通过 C 数据结构、队列和通知交换状态，属于另一套内存模型。执行后端要把两层正确连接，但设备模型内部的发布与消费仍需宿主同步，不能借用客户机指令替代。
:::
::::

## 小结

线程、协程和事件循环是三个不同维度。先确认代码在哪里运行、状态归谁、如何被唤醒，再讨论锁和性能，问题会收敛得更快。

本章出现的机制可以压缩成两条纪律。跨上下文传递工作时，先发布由生命周期保护的状态，再通知目标执行域，消费者醒来后循环检查真实条件；修改共享结构时，先写出需要同时成立的不变量，再选择 BQL、子系统锁、原子快照、RCU 或单一 `AioContext` 所有权。通知不等于请求，原子不等于事务，协程不等于线程，RCU 也不等于没有 writer 锁。

RISC-V `virt` 给这些纪律提供了一条具体时间线：hart 在 TCG 或 KVM vCPU 线程执行，MMIO 可能让设备回调留在 vCPU 上，也可能把数据面交给 IOThread，完成后再经中断控制器唤醒目标 hart；主线程同时处理管理命令、暂停和生命周期。一次请求走完这条路径，几乎会遇到本章所有边界。

完成两个实验后，应留下线程拓扑、泳道图和共享字段表。下一章进入 QOM 时，这些图不会作废：对象的 child、link、realize 与引用计数，正是并发回调能够安全找到目标并在结束后释放的前提。第五章的 FlatView 与 RAMBlock 则会展示 RCU 和事件域如何保护高频访存路径。并发不是一章结束后就离开的专题，它会成为后续每段源码的背景坐标。

如果排查仍然发散，回到四句话：当前是哪条宿主线程，当前轮询哪个 `AioContext`，这份状态由谁修改，当前引用凭什么仍然有效。四句都能用运行证据回答后，再去谈优化或锁替换，通常就不会修掉表象、留下更隐蔽的竞态。

还要把失败路径纳入同一张图。正常完成只走一次通知，取消、reset 与热拔除却可能同时唤醒多个等待者；正常运行对象长期存活，关闭时最后一个引用会突然变得关键。实验若只覆盖稳定吞吐，最多说明快路径可工作。加入暂停、恢复和退出，才能验证执行域与生命周期真的闭环。

最后给每条跨域箭头标出同步语义：只唤醒、排队且异步、排队并等待、发布不可变快照，或转移对象所有权。箭头名称相似，调用者能作出的假设却不同。把这项标注留在代码审查记录中，后续更换 EventNotifier、BH 或 queued work 实现时，就能检查语义是否保持，而不被函数改名带偏。

章末再做一次反向检查：选择一条看似无锁的快路径，确认它依赖的是单线程所有权、原子发布、RCU，还是调用者已经持锁；如果四者都不是，所谓“快”很可能只是把同步假设藏到了注释之外。把依据写在实验图旁，后续压力测试才知道该破坏哪个条件。

同一张图还应标出取消方向。请求从 vCPU 去 IOThread，reset 或关闭可能从主线程反向追赶请求；完成事件若在两者交叉时到达，谁接管引用、谁报告结果必须有唯一答案。把这条逆向路径补齐，泳道图才覆盖完整生命周期。
