# 前言 {.unnumbered}

\markboth{前言}{前言}

2025 年第一次组织 QEMU 技术训练营时，我们面对的是一批来自调试现场的问题。有人能启动一台 RISC-V `virt` 虚拟机，却说不清 `MachineState` 在什么时候把 CPU、内存和设备连在一起；有人沿着 `cpu_exec()` 追完一条调用链，仍然不知道 TCG 为什么需要自己的 IR；也有人把 KVM 看成一条更快的执行路径，却没有继续追问，CPU 已经交给硬件执行之后，QEMU 为什么还保留如此庞大的设备模型。

两期训练营留下了在线讲义、实验手册、学员博客和大量讨论。把这些材料按组件重新排列，很容易得到一份厚重的 QEMU 使用说明。这样的书可以告诉读者某个结构体有什么字段、某个子系统有哪些功能，却很难回答更耐久的问题：当时遇到了什么约束，社区为什么选择这条路，今天看似顺理成章的抽象经历过哪些反对意见，又留下了什么代价。

这次重写从这些问题重新开始。

## 从一个能够运行的程序开始 {.unnumbered}

QEMU 最早公开时还没有今天这台“万能机器”的样子。2003 年的早期文档把它描述为一个 x86 处理器模拟器，目标是在 PowerPC、Arm 等非 x86 Linux 主机上运行 x86 Linux 用户程序，Wine 是其中一个明确场景。几个月后，QEMU 已经能够启动基本未经修改的 Linux 内核，串口和 NE2000 也进入了模型。问题边界从一个进程扩展到内核、MMU、中断和设备，一套完整机器的轮廓才逐渐形成。

这段来路决定了本书的讲法。我们会先还原工程问题，再观察方案怎样扩张。当前源码放在后面，它承担验证和落地的角色。读者看到 QOM、MemoryRegion、TranslationBlock 或 AccelOps 时，应当知道它们接住了哪一类长期压力，而不只知道它们位于哪个目录。

全书围绕五组问题展开：QEMU 为什么出现；dyngen 为什么让位于 TCG，TCG 又为什么采用小而强类型的 IR；硬件虚拟化已经存在时，QEMU 与 KVM 为什么形成今天的分层；模型库为什么同时容纳真实板、合成平台、测试设备和数据面卸载；一个数百万行、长期兼容的软件工程为什么在今天尝试引入 Rust。

## RISC-V 是全书的实现坐标 {.unnumbered}

QEMU 和 KVM 的起源发生在 x86 语境中，Arm `virt`、PC machine versioning 等案例也曾推动过关键抽象。历史无法被改写成 RISC-V 首创。它们只会在能够解释一条设计转折时进入正文，完成解释后，源码、示例代码和实验都会回到 RISC-V target。

选择 RISC-V 有现实基础。它的指令集规范开放，特权架构、虚拟化扩展、AIA 和 IOMMU 仍在公开生态中快速演进；QEMU 里的 `target/riscv/` 与 `hw/riscv/` 同时覆盖指令语义、地址转换、KVM 接口、`virt` 平台和多种真实 SoC，足以观察一个新架构如何进入成熟的系统软件。相同机制如果需要跨架构比较，正文会说明差异，却不会再铺开第二套示例。

本书作者也参与 QEMU RISC-V 子系统的日常审查与维护。这个位置带来许多一线问题：一个新扩展应当落在 CPU 属性还是 machine 属性里，TCG 与 KVM 暴露的能力怎样保持一致，设备树变化会不会破坏已有固件，补丁为何到了第八版仍然不能合入。参与经验帮助我们提出问题，最终结论仍要回到公开源码、提交、邮件和实验。书中不会用个人身份替代证据。

## 设计发生在补丁抵达之前 {.unnumbered}

当前源码保存了已经收敛的结果。一个兼容属性为何存在，一段看似重复的状态同步为何不能删除，单看最终函数往往得不到答案。Git 历史可以定位变化，却常常只留下简短的提交说明。真正的候选方案、反例和妥协，大多出现在 qemu-devel、linux-kernel、qemu-riscv 等邮件线程，以及 KVM Forum 的演讲和问答中。

这些资料数量巨大，正文不会把它们按年代搬进来。一条史料只有在满足三个条件时才进入章节：它提出了清楚的问题；讨论改变了方案或暴露了取舍；这个决定仍能解释 RISC-V 当前实现的一条边界。其余版本、旁支讨论和来源校验保留在 `book/research/` 的证据账本中。

正文中的角色也按事件发生时记录。Contributor 提出方案，Reviewer 给出技术判断，Maintainer 决定维护范围并把补丁放进子系统树；同一个人在不同时间可能承担不同职责。`Reviewed-by`、`Acked-by`、`Tested-by` 和 maintainer 的 `Signed-off-by` 含义不同，书中不会把所有参与者笼统称为“维护者”。

## Agent 帮助我们穿过资料洪流 {.unnumbered}

过去追一条设计线索，常见做法是从当前符号运行 `git blame`，找到引入提交，再用 Message-ID 搜索邮件，手工比较 v1 到 vN，最后回到固定版本验证。任何一步断掉，研究者都要在归档、镜像和改名后的文件之间来回寻找。Agent 能够并行检查源码路径、提交祖先关系、邮件版本和会议材料，把相互矛盾的说法提前暴露出来。

它提供的是研究加速，不提供事实豁免。模型摘要可能混淆作者与提交者，把今天的 maintainer 身份投射到十年前，也可能从最终代码反推一个从未在邮件中出现的动机。本书采用的工作顺序是：先写清待回答的问题；查看固定版本的当前实现；追踪引入与重构提交；阅读实际改变方案的审查；用源码检查或实验验证可观察行为；最后标出仍缺少直接证据的判断。

这套方法还有一条必须公开说明的边界。QEMU 当前的 code provenance 政策会拒绝包含或源自生成式 AI 内容的上游贡献。政策允许把 AI 用于 API、算法研究、静态分析和调试，前提是生成内容不进入提交。因而，本书可以展示 Agent 如何寻找史料、建立索引和设计验证，却不会把模型生成的补丁包装成可直接投递上游的贡献。准备真实补丁时，贡献者需要遵守目标项目当时的政策，亲自理解并承担每一行内容的来源、许可和正确性。

## 一条证据能支持多远 {.unnumbered}

本书使用四层材料。固定版本源码说明当前实现；commit 和邮件说明变化及当时公开表达的理由；架构规范、KVM UAPI 与官方文档定义外部契约；实验记录可验证行为和性能。训练营博客、个人回顾与二手文章适合提供线索，重要结论会继续追到可定位的一手来源。

事实和解释之间会保留距离。邮件中明确写出的目标可以归于作者；多个提交共同呈现出的趋势属于本书的综合判断；只找到最终代码、没有找到历史说明的内容会留在证据账本中继续核验。性能数字必须带环境和测量方法，不能从某次旧硬件结果外推到今天。滚动文档用于发现新进展，正文的源码链接则固定到研究基线或具体 commit。

本书面向 QEMU `v11.1.0` 发布线。截至 2026 年 7 月 19 日，正式 tag 尚未发布，因此本轮源码研究与实验基线冻结在 `v11.1.0-rc0`，commit `eca2c16212ef9dcb0871de39bb9d1c2efebe76be`。历史章节可以引用更早提交；描述现状时必须回到这个锚点。RISC-V 规范和 Linux KVM 仍在演进的部分会注明日期与状态，正式版发布后还要重新核验差异。

## 这五篇怎样连接起来 {.unnumbered}

第一篇从 2003 年的用户态模拟出发，追到完整机器、对象生命周期、地址空间和执行后端。第二篇进入 dyngen 与 TCG，解释一条 guest 指令为何要经过 RISC-V translator、TCG IR、host backend 和 SoftMMU。第三篇把执行交给硬件，沿 RISC-V H 扩展、Linux KVM 和 QEMU 状态同步观察分层的收益与成本。

第四篇回到机器和设备。真实 SoC、`virt`、测试设备、virtio、vhost 与 VFIO 面向不同问题，逼真度、可控性、性能、迁移和安全承诺也不同。读者会沿 RISC-V 设备完成寄存器、MMIO、IRQ、DMA、reset 和迁移状态的建模，再沿一块上游 RISC-V 板卡理解 CPU、内存图、中断拓扑、固件和设备树怎样装配。

第五篇讨论这套工程如何继续变化。调试部分会分别使用 monitor、QEMU log、trace event 与 gdb stub，覆盖 Linux 内核、裸机、RTOS 和一般 guest；随后再看测试怎样让修改可见，machine version 如何保存旧契约，邮件审查怎样把补丁收敛，Rust 又怎样迫使 QOM、BQL、迁移和 C ABI 的隐含规则变得可审查。

每章至少给出一条可复核路径。有些适合运行 QEMU、观察 trace 或比较 TCG/KVM；有些适合固定源码版本、重放一次 `git log` 或核对 patch series。实验的任务是帮助读者验证一个判断，环境不具备时会明确记录跳过条件。

## 读完以后还剩下什么 {.unnumbered}

资料检索越来越便宜，记住更多函数名的优势也会迅速衰减。能够长期留下来的，是提出问题、判断来源、识别状态所有权、理解兼容代价，以及在公开审查中把方案讲清楚的能力。

我们希望读者读完以后，能说清 QEMU 在系统软件生态中的位置，也愿意打开一条真实邮件线程，看看一个陌生人的反例怎样改变了补丁。第一次贡献可能只修正文档、补一个测试或澄清一段错误信息。随着问题逐渐深入，技术判断、表达能力和与人合作的方式会一起发生变化。QEMU 是这次练习的对象；从问题回到证据，再从证据走向协作，这条路可以带到更多大型软件中。
