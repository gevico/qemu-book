# 前言 {.unnumbered}

\markboth{前言}{前言}

这本书最初没有一本书的样子。2025 年，我们开始组织开源、开放、免费的 QEMU 技术训练营，一年一期。到书稿整理完成时，训练营已经办了两期。课堂上的在线讲义、实验手册、学员写下的博客，以及大家在社群里追问和争论过的问题，慢慢积累成了最早的一批材料。

训练营结束以后，这些内容仍然散落在不同地方。有些适合第一次把虚拟机跑起来，有些来自一次具体调试，还有一些只在讨论中留下半条线索。我们决定重新整理这批材料，并沿着一套更稳定的问题组织全书：QEMU 为什么需要这些抽象，它们解决了什么约束，今天看到的实现又是怎样一步步形成的。

## 从源码继续往前追 {.unnumbered}

QEMU 的代码量很大，更费力的地方，是在代码之间重新建立整体坐标。一条客户机指令、一次 MMIO 访问、一个块设备请求或一轮迁移，会穿过对象模型、CPU 执行、地址空间、设备后端和并发控制。沿一条调用链追到底，往往只能看见某次实现，还需要继续判断状态由谁拥有，变化发生在哪个上下文，哪些接口承诺兼容，哪些优化又受到语义约束。

因此，源码解析在本书里主要承担定位和取证的作用。我们会读结构体、函数和调用路径，也会做源码实验，但全书更关心代码背后的实现原理与工程选择。QEMU 如何描述一台机器，TCG 与 KVM 为什么共享一部分状态、又把执行放在不同位置，MemoryRegion 为什么需要配置视图和扁平视图，迁移为何迫使设备显式表达长期状态，这些问题比记住某个函数当前位于哪个文件更耐久。

当前源码只保存了设计落地后的结果。要理解一个特殊分支、一段兼容代码或一次接口拆分，还得继续查看 Git 历史和 qemu-devel 邮件列表。补丁的早期版本、作者在 cover letter 中说明的目标、reviewer 提出的反例，以及最终提交留下的取舍，合起来才比较接近设计发生时的现场。我们希望追到的，正是这条从问题、方案、审查到当前实现的演进路径。

过去靠人工完成这项工作，成本很高。一个功能可能经历多轮 patch，主题会变化，文件会移动，邮件归档和最终 commit 也不总能直接对应。研究者需要在源码、Git log、Message-ID、规范和实验输出之间反复核对，很容易在整理线索的阶段就耗尽精力。

## Agent 参与了这本书的写作 {.unnumbered}

Agent 改变了这部分工作的效率。它可以从当前符号向前查找引入提交，沿提交说明寻找邮件线程，比较不同版本的 patch，再把散落的证据整理成待核查的路径。人仍然需要判断一条邮件能支持什么结论，也要用最终源码和实验确认行为。Agent 给出的摘要只提供线索，它更像一个能够持续翻资料、整理索引和暴露矛盾的研究助手。

本书编写过程中主要使用 GPT 5.6，也使用了 Kimi。Codex 用于检索和核对源码、整理 Git 与邮件证据、维护实验代码、执行测试并验证 PDF 构建；Typeless 负责把口述内容转成文字。书中的大部分主体内容都先经过口述，我们把一个问题的来龙去脉、实际经验和暂时没有想清楚的地方说出来，再把转录文本放回源码与证据中校订。

口述没有跳过写作。直接面对空白文档时，人很容易过早修饰句子，把尚未验证的判断写得过于完整；讲给另一个人听时，问题、停顿和疑问通常会先出现。语音转写保留了这种原始顺序，模型帮助我们归拢结构，最后仍要靠人工删改、核查和实验，把能够确认的事实与作者的推断分开。

这次写作也让我们重新思考 Agent 时代的系统学习。学习一个大型软件，工作重心正在从“能不能找到资料”转向“能不能提出足够清楚的问题，建立可靠的证据链，并知道结论在哪里停止”。这套方法可以写成一条反复执行的路径：先描述问题和边界，再看当前源码，随后追踪 Git 与邮件审查，用小实验验证可观察行为，最后记录仍然缺失的证据。Agent 可以参与每一步，却不能替代最后的工程判断。

## 从 QEMU 中沉淀可复用的方法 {.unnumbered}

我们也想知道，从 QEMU 身上学到的东西，有多少能够带到其他技术领域。大型系统软件经常面对相似的问题：状态属于谁，对象在什么时候生效，快路径怎样绕开通用层，错误如何返回到控制面，兼容承诺会怎样限制后续重构。QEMU 把这些矛盾集中在一套长期演进的代码里，因此很适合用来练习系统软件工程。读懂虚拟化工具的源码，只是这项练习的一部分。

这种泛化也有边界。某个 QEMU 补丁采用的拆分方式，不会自动成为其他项目的设计准则；邮件中一次成功的论证，也不能脱离当时的性能数据、维护范围和兼容条件。本书会尽量把源码事实、上游陈述、实验观察和作者推断分开，让读者既能看到结论，也能回到证据重新作出判断。

## 资料与证据来源 {.unnumbered}

本书的技术材料主要来自三个方向：

1. QEMU 源码。固定版本的源码用于确认当前对象、字段、调用路径和明确限制，是分析实现行为时最直接的依据。
2. QEMU Git 历史与 qemu-devel 邮件列表。提交记录用于定位变化，邮件线程用于理解补丁作者的理由、reviewer 的质疑以及方案在多轮修改中的收敛过程。
3. 两期 QEMU 技术训练营沉淀的材料，包括学员贡献的博客、实验记录、在线讲义和社群讨论。它们保存了真实学习过程中容易卡住的地方，也提供了许多值得回到上游继续核对的问题。

三类材料承担的角色不同。训练营博客和社群讨论可以提供现象、经验和线索，最终的源码结论仍应回到固定版本、上游提交和可复现实验。本书在引用设计动机时，也会尽量找到对应邮件或提交说明；找不到时，会明确写成作者推断或开放问题。

## 目标读者 {.unnumbered}

本书主要面向以下读者：

- 已经会使用 QEMU，希望开始阅读或修改源码的系统软件工程师；
- 从事虚拟化、云计算、操作系统、体系结构或设备模拟工作的开发者；
- 想理解 TCG、KVM、virtio、vhost、块设备和迁移如何在同一进程中协作的读者。

## 阅读前提 {.unnumbered}

本书从设计和实现开始，不从 C 语法、编译命令或操作系统入门讲起。开始阅读前，读者最好具备下面这些基础：

- 熟悉 C 语言，能够阅读指针、函数指针、宏和并发相关代码；了解 Rust 的所有权、trait、FFI 等基本概念；
- 知道指令集、寄存器、异常和中断是什么，对 MMU、页表、缓存、内存一致性和硬件虚拟化有初步认识；
- 具备一些 RISC-V 体系结构背景，了解 hart、RV64 基础指令、M/S/U 特权级、CSR、异常与中断、SBI 和 Sv39 页表的基本含义；H 扩展与两阶段地址转换会在正文中展开，不要求读者事先掌握实现细节；
- 接触过 Makefile、Kconfig、Meson 或 CMake，知道一个大型 C 项目如何配置、编译、链接和生成目标文件；
- 了解 Linux kernel、RTOS 或其他操作系统中的进程、线程、虚拟内存、驱动与中断机制；
- 对 BIOS、固件、Bootloader 和外设初始化有基本认识，能够说清服务器或嵌入式系统从上电到操作系统接管的大致启动流程；
- 有基本的 QEMU 使用经验，能够用命令行启动虚拟机或全系统模拟器，并知道 machine、CPU、内存、镜像和串口参数各自承担什么职责。

全书以 RISC-V 为架构主线。涉及指令、CPU、MMU、中断、硬件虚拟化和机器模型时，都以 RISC-V 及其相关硬件模型为主要实例；QEMU 的通用框架仍会单独讲清，读者不需要预先熟悉某块具体的 RISC-V 开发板。

## 阅读方法 {.unnumbered}

全书从 QEMU 的系统边界出发，先建立公共对象和 CPU 模型，再分别进入 TCG 与硬件虚拟化两条执行路径，最后回到机器、设备和工程实践。每章都围绕四条线索收束：

| 线索 | 要回答的问题 |
| --- | --- |
| 概念模型 | 这一子系统对外提供什么抽象？ |
| 生命周期 | 对象何时创建、连接、运行和销毁？ |
| 热路径 | 性能敏感的数据怎样流动？ |
| 一致性边界 | 哪些锁、线程、事务或状态转换保证正确性？ |

::: {.design-note}
阅读大型系统时，先画出数据所有权和状态边界，再追函数调用，通常比从 `main()` 一路单步更有效。本书会反复使用这种方法。
:::

## 版本与实验原则 {.unnumbered}

QEMU 仍在快速演进。本书把 QEMU `v11.1.0` 设为目标源码基线；在正式 tag 发布前，当前实现的研究锚固定为官方 GitLab 的 `v11.1.0-rc0`，commit `eca2c16212ef9dcb0871de39bb9d1c2efebe76be`。正式版本发布后，再统一冻结最终源码链接和实验结果。分析设计演进时可以引用更早的 commit 和 qemu-devel 邮件讨论，但必须明确区分稳定原理、历史实现、当前 rc 事实与作者推断。示例默认使用 out-of-tree build，实验应记录宿主架构、目标架构、加速器和完整命令行。

构建本书不依赖 QEMU 源码。阅读和实验时，建议另行准备一份 QEMU 官方仓库工作树，并保留至少一个 TCG system emulator 和一个能够在宿主机运行的硬件加速版本。

每章至少安排两个中文实验入口。完整代码、输入与英文手册放在仓库的 `experiments/` 下，按篇、章和独立项目组织；需要 RISC-V KVM 硬件、特定固件或外部资源的实验会明确给出探测、跳过与校验条件，不把环境缺失伪装成成功结果。

## 本书约定 {.unnumbered}

- guest 或客户机：QEMU 模拟或虚拟化出来的软件运行环境；
- host 或宿主机：运行 QEMU 进程的环境；
- target：客户机指令集相关代码所面向的架构；
- device model：QEMU 进程内的软件设备实现；
- 加速器：TCG、KVM 等 vCPU 执行后端。

第一次出现的核心结构体和函数使用行内代码标出，例如 `Object`、`MemoryRegion` 和 `cpu_exec()`。源码路径以当前讨论版本为准，不把文件位置误当成稳定接口。
