# 前言 {.unnumbered}

\markboth{前言}{前言}

QEMU 的难，不只在于代码量大，也不只在于它同时支持许多宿主架构、客户机架构、机器和设备。真正的门槛是：一条客户机指令、一次 MMIO 访问、一个块设备请求或一轮迁移，会跨越多套抽象；如果只沿着单条调用链向下追，很快就会失去整体坐标。

《深入理解 QEMU 设计原理》希望提供这套坐标。本书不追求覆盖每个命令行参数，也不逐文件复述实现，而是围绕几个长期存在的设计问题展开：QEMU 如何描述一台机器，如何在不同执行引擎之间共享设备模型，如何把客户机地址映射到 RAM 或设备，如何组织异步 I/O，以及如何在并发和迁移中保持状态一致。

## 目标读者 {.unnumbered}

本书主要面向以下读者：

- 已经会使用 QEMU，希望开始阅读或修改源码的系统软件工程师；
- 从事虚拟化、云计算、操作系统、体系结构或设备模拟工作的开发者；
- 想理解 TCG、KVM、virtio、vhost、块设备和迁移如何在同一进程中协作的读者。

## 阅读前提 {.unnumbered}

本书从设计和实现开始，不从 C 语法、编译命令或操作系统入门讲起。开始阅读前，读者最好具备下面这些基础：

- 熟悉 C 语言，能够阅读指针、函数指针、宏和并发相关代码；了解 Rust 的所有权、trait、FFI 等基本概念；
- 知道指令集、寄存器、异常和中断是什么，对 MMU、页表、缓存、内存一致性和硬件虚拟化有初步认识；
- 接触过 Makefile、Kconfig、Meson 或 CMake，知道一个大型 C 项目如何配置、编译、链接和生成目标文件；
- 了解 Linux kernel、RTOS 或其他操作系统中的进程、线程、虚拟内存、驱动与中断机制；
- 对 BIOS、固件、Bootloader 和外设初始化有基本认识，能够说清服务器或嵌入式系统从上电到操作系统接管的大致启动流程；
- 有基本的 QEMU 使用经验，能够用命令行启动虚拟机或全系统模拟器，并知道 machine、CPU、内存、镜像和串口参数各自承担什么职责。

全书以 RISC-V 为架构主线。涉及指令、CPU、MMU、中断、硬件虚拟化和机器模型时，优先选择 RISC-V 及其相关硬件模型；QEMU 的通用框架仍会单独讲清，读者不需要预先熟悉某个具体 RISC-V 平台。

## 阅读方法 {.unnumbered}

全书从 QEMU 的系统边界出发，先建立公共对象和 CPU 模型，再分别进入 TCG 与硬件虚拟化两条执行路径，最后回到机器、设备和工程实践。每章都围绕四条线索收束：

| 线索 | 要回答的问题 |
| --- | --- |
| 概念模型 | 这一子系统对外提供什么抽象？ |
| 生命周期 | 对象何时创建、连接、运行和销毁？ |
| 热路径 | 性能敏感的数据怎样流动？ |
| 一致性边界 | 哪些锁、线程、事务或状态转换保证正确性？ |

::: {.design-note}
阅读大型系统时，先画出数据所有权和状态边界，再追函数调用，通常比从 `main()` 一路单步更有效。本书会反复使用这种方法。
:::

## 版本与实验原则 {.unnumbered}

QEMU 仍在快速演进。本书把 QEMU `v11.1.0` 设为目标源码基线；在正式 tag 发布前，当前实现的研究锚固定为官方 GitLab 的 `v11.1.0-rc0`，commit `eca2c16212ef9dcb0871de39bb9d1c2efebe76be`。正式版本发布后，再统一冻结最终源码链接和实验结果。分析设计演进时可以引用更早的 commit 和 qemu-devel 邮件讨论，但必须明确区分稳定原理、历史实现、当前 rc 事实与作者推断。示例默认使用 out-of-tree build，实验应记录宿主架构、目标架构、加速器和完整命令行。

构建本书不依赖 QEMU 源码。阅读和实验时，建议另行准备一份 QEMU 官方仓库工作树，并保留至少一个 TCG system emulator 和一个能够在宿主机运行的硬件加速版本。

每章至少安排两个中文实验入口。完整代码、输入与英文手册放在仓库的 `experiments/` 下，按篇、章和独立项目组织；需要 RISC-V KVM 硬件、特定固件或外部资源的实验会明确给出探测、跳过与校验条件，不把环境缺失伪装成成功结果。

## 本书约定 {.unnumbered}

- guest 或客户机：QEMU 模拟或虚拟化出来的软件运行环境；
- host 或宿主机：运行 QEMU 进程的环境；
- target：客户机指令集相关代码所面向的架构；
- device model：QEMU 进程内的软件设备实现；
- 加速器：TCG、KVM 等 vCPU 执行后端。

第一次出现的核心结构体和函数使用行内代码标出，例如 `Object`、`MemoryRegion` 和 `cpu_exec()`。源码路径以当前讨论版本为准，不把文件位置误当成稳定接口。
