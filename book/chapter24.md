# RISC-V GPGPU 与异构计算建模

在 QEMU 中建模一块 GPGPU，容易一开始就陷进指令和算子细节。更有效的切入点是先确定客户机可见契约：设备如何被发现，命令怎样提交，地址如何翻译，完成如何通知，错误如何报告。计算核心可以先是一个确定性的简化模型，等控制面和内存语义稳定后再逐步提高精度。

## 本章目标

- 把异构设备拆为控制面、命令处理、内存访问和计算模型四层；
- 在 RISC-V `virt` 的 PCIe、DMA 与 IOMMU 语境中定义边界；
- 比较功能模型、时序近似和真实后端代理的取舍。

## 先把证据坐标钉住

本章的源码口径固定为 QEMU `v11.1.0`，可复查锚点是官方 GitLab [`v11.1.0-rc0`](https://gitlab.com/qemu-project/qemu/-/tree/v11.1.0-rc0)，commit `eca2c16212ef9dcb0871de39bb9d1c2efebe76be`。下文把材料分成四类。能够在这个标签中逐行核对的，称为“源码事实”；提交说明、文档和 qemu-devel 讨论中由上游作者表达的动机，称为“上游陈述”；把若干事实连接起来得到的设计解释，明确写作“作者推断”；缺少公开规范、运行数据或审查结论的部分，留作“开放问题”。这四种语气不能混用，尤其不能因为一个教学模型能够运行，便倒推上游已经承诺了相同接口。

在固定源码中，RISC-V `virt` 能提供通用 PCIe 主桥、DMA 地址空间和可选的 RISC-V IOMMU，`hw/pci/` 也提供 BAR、MSI-X 与配置空间基础设施。这些都是可验证的复用条件。与此同时，该标签没有一块专门承担 RISC-V 通用计算任务的 GPGPU 设备。`virtio-gpu` 的设备契约围绕图形显示与资源传输展开，不能改个名称便当作计算队列；K230 内存图中虽有名为 `gpu`、KPU 和 AI 2D engine 的区域，当前代码使用 `create_unimplemented_device()` 建立地址占位，它们不会执行图形或张量算子。以上是源码边界，不是对未来补丁的预测。

本书采用一块树外教学设备，把复杂问题压缩到能够推理和反驳的尺寸。它挂在 riscv64 `virt` 的 PCIe 总线上，只支持小型整数向量加法，计算结果由宿主功能模型给出。选 PCIe 的理由是能直接练习设备发现、BAR、DMA 和 MSI-X，也便于后续加入 IOMMU；选整数向量的理由是结果稳定，读者可以把注意力放在协议。作者推断，这套控制面方法能迁移到更复杂的异构设备；至于真实 GPGPU 的指令格式、缓存一致性和调度层级，应等待具体公开规范，不能由教学接口代填。

## CPU 向量执行与外设卸载是两条轴

RISC-V Vector 扩展让 CPU 指令流直接操作向量寄存器，指令退休、异常、页表访问和上下文切换仍属于处理器模型。外接计算设备则由驱动构造命令，通过 MMIO 或队列提交，设备以 DMA 读取内存，再异步报告完成。两者都能做向量加法，系统可见语义却很不一样。CPU 路径遇到页故障时，异常精确地回到当前 hart；设备路径遇到 IOVA 翻译失败时，需要在设备状态或完成项中记录错误，必要时再产生中断。

这一区分也决定 QEMU 代码应该落在哪里。扩展一条 RISC-V CPU 指令，会进入 `target/riscv/` 的译码、TCG helper、特权检查和迁移状态；建模 PCIe 加速器，会进入设备 QOM 类型、PCI 配置、MemoryRegion、DMA、IRQ 与 VMState。把二者揉成一个对象，看似能少写接口，实际会让客户机 ABI 与某个执行后端绑定。实验因此把 CPU 当作命令生产者，把设备当作独立状态机，驱动即使改用 TCG 或 KVM，客户机看到的队列契约也应保持不变。

“异构”还意味着两个推进速度。hart 可以继续执行其他线程，设备在自己的执行上下文消费请求；队列满、完成中断延迟和缓存可见性会决定软件能否真正重叠工作。单次向量加法跑通，只说明数据路径有一条成功路线，不能说明异步协议成立。验证时要有意制造队列满、延迟完成、reset 插入和错误描述符，观察生产者是否会错误复用仍在设备手中的缓冲区。

## 先定 ABI，再写 C 结构

客户机与设备共享的结构是一份长期契约。宿主 C 编译器的填充、指针宽度和位域布局不应泄露到这份契约中，因此描述符应使用固定宽度、明确字节序和显式保留字段。一个可用的教学描述符至少包含操作码、标志、输入地址、输出地址、元素数量、请求标识与状态位置。每个字段还要说明谁写、谁读、何时所有权转移，以及失败后哪些字段仍可信。只有结构定义，没有状态转换，驱动和设备仍可能各自作出合理却冲突的解释。

版本策略也要在第一版出现。设备可以在配置寄存器公布 ABI 主版本、次版本和 feature bits，驱动只启用双方共同理解的能力。主版本变化表示布局或基础语义不兼容，次版本用于向后兼容扩展；未知 feature 应被拒绝或忽略，行为必须写清。QEMU machine type 会把设备可见行为带入迁移与兼容范围，因而“以后再加版本”通常会晚一步，旧客户机已经观察到的缺省值很难收回。

寄存器区同样需要按动作分类。只读 capability 由设备产生，读写配置在设备空闲时才能修改，doorbell 是一次动作，状态寄存器可能采用 write-one-to-clear。若把这些访问都交给一段通用数组读写，保留位、访问宽度和副作用很快会失控。实现应验证 offset、size、对齐与端序，对非法访问给出稳定结果并记录 guest error，日志频率还要受控，避免恶意客户机靠循环访问淹没宿主。

:::: {.quick-quiz}
为什么不能直接把宿主的 `struct Request` 当作 DMA 描述符？

::: {.quick-answer}
宿主结构可能含填充、指针和编译器相关位域，跨版本、跨端序或跨位宽后布局会变化。客户机 ABI 需要固定宽度和显式字节序，并把所有权转换写成协议。
:::
::::

## PCIe 只解决接入，不替设备定义语义

固定标签中的 [`hw/pci/pci.c`](https://gitlab.com/qemu-project/qemu/-/blob/v11.1.0-rc0/hw/pci/pci.c) 提供 `pci_register_bar()` 等通用设施，[`hw/pci/msix.c`](https://gitlab.com/qemu-project/qemu/-/blob/v11.1.0-rc0/hw/pci/msix.c) 处理 MSI-X 初始化和通知，[`hw/pci-host/gpex.c`](https://gitlab.com/qemu-project/qemu/-/blob/v11.1.0-rc0/hw/pci-host/gpex.c) 则是 `virt` 使用的通用 PCIe 主桥实现之一。这些源码事实说明教学设备可以沿成熟总线框架接入，却没有替它决定队列格式、命令完成顺序或 reset 语义。总线层给出运输工具，设备规范仍要自己写。

BAR 的尺寸与属性一旦被固件和驱动探测，就会成为 ABI。控制寄存器适合放在小型 MMIO BAR，共享队列内容留在客户机 RAM，由描述符地址引用；若再增加 doorbell page 或设备本地窗口，应分别说明缓存属性和访问宽度。为了少改一次代码便随意扩大 BAR，可能改变资源分配，给旧固件造成兼容问题。教学实现先保留足够但有界的空间，未定义偏移稳定返回零或错误，不能让未初始化宿主内存泄露给客户机。

中断能力也不等于“调用一次 notify 就完成”。驱动要能区分命令完成、队列错误和设备致命错误，设备还要处理 mask、pending 与中断合并。首个模型可只分一个 MSI-X vector，但完成队列中必须携带请求标识和状态，使驱动在多个在途请求中找到对应项。若丢中断，驱动可通过轮询完成索引继续前进；若重复中断，消费完成项也必须幂等。这样测试才能把通知机制和实际完成状态分开。

QEMU 自带的 [`hw/misc/edu.c`](https://gitlab.com/qemu-project/qemu/-/blob/v11.1.0-rc0/hw/misc/edu.c) 是很好的审查样本。它是一块教学 PCI 设备，包含 BAR、MMIO、DMA、中断、计时器和工作线程，能够挂在 RISC-V `virt` 的通用 PCIe 总线上。它不是 GPU，也不提供本章 ABI；我们只借它观察一个小设备怎样组织 realize、uninit、范围检查和异步工作。复用模式时要逐项核对语义，不能复制函数名字后宣称设计已经得到验证。

## 先定义客户机看得见什么

本章的 GPGPU 是教学模型，不是 QEMU `v11.1.0` 的既有官方 RISC-V GPU。它通过 PCIe 暴露 BAR 寄存器和队列，客户机驱动写入命令描述符与缓冲区地址，再敲 doorbell。设备读取描述符，执行一个小型向量内核，写回结果和完成状态，最后通过 MSI-X 或简化中断通知客户机。

这条链路已经包含真实设备模型的大部分困难。命令长度来自不可信客户机，必须有上限；DMA 地址可能跨页，不能当作宿主指针；reset 可能发生在命令执行中；迁移需要决定队列和在途工作如何冻结。即使计算只做向量加法，外围契约仍要认真设计。

:::: {.quick-quiz}
为什么 GPGPU 教学模型应先实现控制面，而不是先追求完整指令集？

::: {.quick-answer}
驱动首先依赖发现、队列、DMA、完成和错误语义，这些也是安全与迁移边界。计算核心可替换，若控制面不稳定，更多指令只会放大状态和兼容问题。
:::
::::

## DMA 地址不是普通指针

设备收到的是客户机或 I/O 虚拟地址。无 IOMMU 时，它经平台 DMA 地址空间访问系统内存；启用 RISC-V IOMMU 后，还要按设备上下文完成 IOVA 转换和权限检查。QEMU 的设备模型应使用地址空间 API，逐段映射或读写，正确处理短映射、跨页和失败，不能把数值强制转换成宿主地址。

scatter-gather 能减少复制，却把生命周期问题带到前台。映射期间客户机可能重配队列，reset 可能撤销工作，迁移也可能要求停止 DMA。设计通常需要明确请求对象所有权、取消点和回滚顺序。为了简化首个实验，可以先复制小型描述符与数据，再用性能实验说明零拷贝优化为什么需要更多协议。

固定标签在 [`include/hw/pci/pci_device.h`](https://gitlab.com/qemu-project/qemu/-/blob/v11.1.0-rc0/include/hw/pci/pci_device.h) 暴露 `pci_dma_read()`、`pci_dma_write()` 一类 PCI 设备入口，通用 DMA 接口位于 [`include/system/dma.h`](https://gitlab.com/qemu-project/qemu/-/blob/v11.1.0-rc0/include/system/dma.h)，辅助实现可从 [`system/dma-helpers.c`](https://gitlab.com/qemu-project/qemu/-/blob/v11.1.0-rc0/system/dma-helpers.c) 追踪。这里有一个很实用的源码阅读方法：从设备调用点沿 AddressSpace 继续走，确认访问使用哪个地址空间、权限错误如何返回、短传输由谁处理。只记住 API 名称，会漏掉设备上下文和 IOMMU 翻译究竟在哪一层生效。

长度检查要采用“先证明，再计算”的顺序。假设命令给出元素个数 `n`、元素大小 `s` 和基地址 `a`，实现需要确认 `n` 不超过协议上限、`n * s` 不溢出目标整数类型、`a + n * s` 不回绕，输入输出区也不能与设备私有数据形成意外别名。不要先做乘法再拿已经溢出的结果比较。对二维张量还要验证行数、列数、stride 和每一行末端，单个总长度正确并不保证所有步进都在范围内。

DMA 失败应进入客户机可见状态机。描述符读取失败时，设备尚未取得完整请求，完成项可以标记地址错误并推进消费索引，也可以进入队列错误态要求驱动 reset；两种都能成立，关键是规范一致。数据写回在中途失败更棘手，输出可能部分更新。教学 ABI 选择先在宿主临时缓冲区完成计算，再一次有界写回，并把任何短写报告为请求失败，这减少了半成品状态，却增加了一次复制。作者推断，这是早期模型值得支付的成本。

启用 [`hw/riscv/riscv-iommu.c`](https://gitlab.com/qemu-project/qemu/-/blob/v11.1.0-rc0/hw/riscv/riscv-iommu.c) 后，设备看到的 IOVA 能经过上下文选择与页表转换，权限和失效行为由 IOMMU 模型参与。实验若只在未启用 IOMMU 时传入 GPA，无法证明驱动正确使用 DMA API。更强的测试应建立受限映射，让输入可读、输出可写，再故意撤掉一页或改变权限，确认设备得到可控错误，QEMU 进程不会因为错误 IOVA 解引用宿主地址。

:::: {.quick-quiz}
一次 DMA 输出跨越两个客户机页面，第二页没有写权限，设备可以把请求简单记为“完全未执行”吗？

::: {.quick-answer}
不能默认这样声明，第一段可能已经写入。若 ABI 要求原子可见，模型需先写宿主临时缓冲区并在可证明的条件下提交，或明确规定部分写入及恢复方式。测试也要检查实际内存，而只看错误码不够。
:::
::::

## 一致性要拆成三个不同问题

共享队列常被一句“加内存屏障”草草带过，真正需要区分客户机 hart 的内存排序、设备 DMA 的可见性，以及 QEMU 宿主线程之间的同步。RISC-V 驱动在更新描述符后敲 doorbell，需要按设备 ABI 使用合适的 fence 或内核 DMA API，让数据在通知之前对设备可见；设备写完成数据后再更新完成索引，也要保证驱动不会先看到索引却读到旧内容。这是客户机可观察契约。

QEMU 内部若由 MMIO 回调生产工作、worker 线程消费，请求对象还需要宿主同步。C 原子、互斥量、条件变量或 AioContext 串行化解决的是宿主 data race，不能代替客户机的 RISC-V fence；反过来，客户机执行 fence 也不会自动修复两个宿主线程对普通 C 字段的竞态。调试时把这两层日志分别记录，能避免看到一个旧值后胡乱在所有路径添加屏障。

第三层是设备是否宣称 DMA coherent。若平台和设备契约提供一致性，驱动可按相应 DMA API 管理缓冲区；若设备是 non-coherent，缓存清理、失效和所有权转换必须由平台规则补足。教学模型运行在 QEMU 统一 AddressSpace 上，不代表真实硬件必然 coherent，也不代表客户机可以省略协议。书中实验只枚举一个简化的发布—获取可见性模型，明确不把它包装成完整 RVWMO 证明。

`edu` 的演进给出一个很小却有力的例子。提交 [`2482aeea4195`](https://gitlab.com/qemu-project/qemu/-/commit/2482aeea4195ad84cf3d4e5b15b28ec5b420ed5a) 在工作唤醒路径补入 `smp_mb__after_rmw()`，提交说明把目标指向避免 lost wakeup。这个上游陈述提醒我们，即便设备功能只有阶乘和 DMA，异步线程的状态转换也可能在弱序宿主上暴露问题。作者推断，GPGPU 队列越复杂，越应该用明确所有者和小状态机减少需要手工推导的同步边。

:::: {.quick-quiz}
客户机驱动已经在 doorbell 前执行 RISC-V `fence`，为什么 QEMU worker 仍可能漏掉一次唤醒？

::: {.quick-answer}
客户机 fence 约束客户机观察顺序，worker 的睡眠标志和通知位属于宿主线程共享状态。后者仍需符合 QEMU/C 内存模型的同步，二者处在不同层次。
:::
::::

## 命令处理需要有界推进

doorbell 回调处于 MMIO 访问路径，若直接执行长时间内核，vCPU 会被设备工作阻塞。更合理的模型把命令排入有界队列，由 bottom half、协程或独立 AioContext 消费。选择哪一种，取决于工作是否阻塞、能否切分、是否要求并行，以及状态由哪个线程拥有。

队列必须定义满时行为，是拒绝新命令、置错误位，还是让客户机等待。无限排队看起来方便，却允许恶意或故障客户机耗尽宿主内存。每次处理也应设置预算，让主循环有机会服务计时器、Monitor 和其他设备。这里的“像硬件”不等于模拟每个周期，而是保持可观察的流控与完成顺序。

环形队列至少有 producer、consumer、深度和代次问题。若只保存两个取模后的下标，`head == tail` 既可能表示空，也可能表示恰好绕满；可以预留一个槽位，或增加 phase/generation bit。深度最好限定为二的幂并验证上下界，使取模和差值计算有稳定语义。驱动不能在设备消费前覆盖描述符，设备也不能在完成前提前归还所有权。把这些条件写成不变量，测试才能生成靠近回绕边界的序列。

doorbell 数值代表什么也要固定。它可以是“有新工作”的提示，设备随后从共享索引读取数量；也可以携带新 producer 值。前者容忍通知合并，后者便于检查跳跃，却要处理重复写和旧值。教学模型采用提示式 doorbell，真实请求数量由受保护的共享索引决定，因此多次相同通知不会重复执行命令。队列状态机仍需防止客户机把 producer 一次推进超过深度，这类输入应触发错误，而非让设备遍历未初始化内存。

完成队列也会背压。若客户机迟迟不回收完成项，设备不能持续积累宿主对象。可以停止取新请求、置 completion-full 状态，并在客户机更新 consumer 后恢复。为了可诊断，寄存器应给出最后错误类别、相关请求 ID 和队列状态，详细地址可通过受控 trace 记录。把原始客户机数据无上限打印进日志，会形成性能和信息泄露问题。

取消语义尤其容易被漏掉。普通请求取消、queue reset、function-level reset 和整机 reset 的范围不同；有的只阻止尚未开始的工作，有的要求等待后端停下，有的允许结果被丢弃。设备可为每个请求分配 generation，完成回调同时核对设备代次与请求状态，旧代次回调只释放资源，不再触碰新队列。这个方案不能魔法般终止宿主库中的计算，却能阻止迟到完成污染 reset 后的客户机状态。

:::: {.quick-quiz}
队列深度为 256，客户机把 producer 从 250 直接写成 20，设备能否按取模差值认定有 26 个新请求？

::: {.quick-answer}
只有 ABI 明确代次且设备能证明推进量不超过深度时才可以。单凭两个取模值无法区分正常回绕、旧写入和跨越多圈，贸然消费会读取已归还或从未初始化的描述符。
:::
::::

## 选择执行上下文时先问谁拥有状态

QEMU 提供 bottom half、协程、线程池和独立 I/O thread 等多种推进工具，[`docs/devel/multiple-iothreads.rst`](https://gitlab.com/qemu-project/qemu/-/blob/v11.1.0-rc0/docs/devel/multiple-iothreads.rst) 解释 AioContext 与设备 I/O 并发。工具名称不是选择依据。若功能模型只处理几十个整数，可把请求排入 bottom half，分批完成；若后端可能阻塞在外部库或服务，需要 worker，并在回到设备所属 AioContext 后提交完成；若想让多个队列并行，还要定义队列之间共享寄存器和 reset 的锁序。

最容易维护的起点是单一状态所有者。MMIO 回调只验证 doorbell 并安排工作，所有队列索引、请求状态和完成通知都在设备所属 AioContext 修改；耗时 worker 只拿不可变输入副本，返回不可变结果。这样 reset 可以在所有者上下文改变 generation、清空待处理列表，worker 不会直接操作 MemoryRegion 或 IRQ。性能不够时再拆分所有权，能用基准数据说明新增锁的价值。

预算决定公平性。一次 bottom half 若把整个深队列跑完，其他设备和定时器可能长时间得不到服务；每轮处理固定个数或固定虚拟工作量，剩余请求重新调度，更接近可控的异步设备。预算本身若暴露为 machine 属性，会影响完成时序和可重复性，实验应记录取值。若只作为宿主实现细节，也要验证改变预算不会改变完成顺序与错误语义。

外部代理还带来进程生命周期。连接断开、请求超时、服务版本变化和宿主关闭都要映射为设备状态。realize 期间无法连接，是拒绝创建设备还是允许离线启动；迁移到另一台宿主，目标端如何证明有兼容服务；这些都是开放问题。首版教学模型不接网络服务，避免把协议设计与环境运维同时引入，扩展实验可在独立 feature 下探索，不能静默替换确定性后端。

## reset 是协议，不是一句清零

设备 reset 前可能有已读取描述符、正在计算的请求、准备写回的结果、pending 中断和映射中的 DMA 区间。逐字段 `memset` 会丢失资源所有权，也可能让迟到 callback 解引用释放后的对象。实现应按顺序停止接受新工作，提升 generation，取消或隔离未开始任务，等待必须同步的访问退出，撤销定时器和 bottom half，释放请求，再恢复寄存器缺省值。每一步都应允许重复调用，realize 失败和 unrealize 也能复用清理路径。

客户机如何观察 reset 也要写清。PCI function reset 后，队列地址、size、enable 和 error 状态回到缺省值，旧请求不会再产生完成；驱动必须重新配置。若硬件规范允许某些 capability 保留，它们仍是只读常量。教学模型用 generation 阻断旧完成，并在 debug trace 记录被丢弃的 request ID。trace 只用于验证，不成为客户机判断成功的条件。

reset 插入测试比启动测试更有价值。让后端在已读取输入、尚未写输出时暂停，触发 reset，随后用新队列提交带不同数据的请求；若旧结果写进新输出、旧中断唤醒新驱动或宿主对象泄漏，说明状态边界不完整。重复数百次能暴露少见窗口，失败时保存随机 seed 和每次状态转换，才能从“偶发挂住”收敛到具体代次错误。

:::: {.quick-quiz}
为什么不应在 doorbell 的 MMIO 回调中直接完成大规模计算？

::: {.quick-answer}
MMIO 回调通常在 vCPU 执行路径中，长计算会阻塞 vCPU 和相关锁，破坏主循环公平性。把工作转入有界异步队列，才能定义流控、取消和完成顺序。
:::
::::

## 三种计算后端

功能模型在宿主上直接计算结果，确定、易测，适合驱动开发和系统启动，但不代表真实吞吐和延迟。时序近似模型根据命令规模和资源参数安排虚拟完成时间，可研究调度与软件重叠，却需要校准并严格使用虚拟时钟。代理后端把任务交给真实 GPU 或独立服务，可能提升功能覆盖，但引入宿主依赖、安全边界和不可迁移资源。

设备前端不应把某个后端细节泄露成客户机 ABI。可以定义统一请求对象与完成结果，后端通过接口实现。若不同后端无法保证同样的浮点结果、错误码或完成顺序，就必须在 feature negotiation 或设备属性中显式呈现，不能静默改变。

功能模型最适合建立 oracle。它接受已经验证并复制的输入，执行边界明确的算法，返回结果或规范化错误，不接触客户机内存，也不触发 IRQ。前端负责 DMA 与状态转换，后端负责纯计算，单元测试因此能绕过完整虚拟机直接覆盖算子。这个拆分也让 RISC-V 相关内容停留在平台和驱动层，计算函数不会偷偷依赖宿主或客户机端序。

时序近似模型要回答“时间属于谁”。若按宿主墙钟 sleep，CI 负载变化会改变完成顺序，虚拟机暂停期间任务还可能向前走；按 QEMU virtual clock 安排 timer，暂停和快照行为更容易解释。估算式可以从固定启动延迟加每元素成本开始，用参数表达队列带宽。它只适合研究软件交叠，不能被宣传成某块真实 GPU 的周期级结果，除非有校准方法、误差范围和公开硬件证据。

代理后端的吸引力很强，宿主 GPU 库可以迅速覆盖更多算子。不过它会把宿主驱动、库版本、内存配额与线程模型带进 QEMU。若库在回调线程内部创建上下文，fork 或迁移可能受限；若输入被直接映射给第三方进程，客户机数据又跨越一层信任边界。一个负责任的实验需要列出后端版本、资源上限、取消能力、可迁移性和数值约束，任何缺项都应在启动时明示。

前端与后端接口可以保持很窄：`submit(request, completion)`、`cancel(generation)` 和 capability 查询已经足够。请求对象只含经过规范化的操作、尺寸和宿主缓冲区，完成对象只含状态、输出与可选统计。后端不能保存指向 QEMU 设备状态的裸指针，callback 也要回到正确 AioContext。这个接口使功能、时序、代理三种实现可替换，同时把无法统一的能力放进 feature negotiation。

:::: {.quick-quiz}
用宿主墙钟给时序模型增加 5 毫秒延迟，为什么会让迁移和 CI 回归难以解释？

::: {.quick-answer}
墙钟在虚拟机暂停时仍推进，宿主负载也会改变回调时机。虚拟时钟能随 VM 状态推进，并让给定参数下的完成点更容易重放；它仍只是模型时间，不代表真实硬件周期。
:::
::::

## 从能跑到可迁移

迁移保存的是客户机可观察状态，不是把整个 C 对象原样搬走。配置寄存器、队列地址与索引、feature、错误状态、中断条件和设备代次需要逐项判断；宿主线程、锁、timer 指针、外部库句柄和映射指针不能序列化。VMState 字段还要有版本与兼容策略，新增字段的缺省值必须让旧数据流可解释。

在途工作有三种常见处理。其一，迁移前停止取新请求，等待当前任务完成，适合工作很短且后端可控的功能模型；其二，保存可重启的规范化输入，目标端重新计算，要求计算确定且重复执行不会产生外部副作用；其三，明确设备忙时不可迁移，管理层得到可诊断错误。教学模型先采用排空队列，测试故意让后端延迟并设置超时，以免迁移永远等待。

“排空”仍需与客户机并发协调。迁移进入 stop-and-copy 前，客户机可能继续敲 doorbell，设备需要在 quiesce 状态拒绝或延后新请求，并保证完成项与中断在源端或目标端只出现一次。若请求已计算但输出尚未 DMA，保存输入重新执行可能产生两次写；若先写输出再保存完成索引，目标端又可能重写。设计状态机时把 DMA 读、执行、DMA 写、发布完成分成阶段，可以明确每个阶段的恢复动作。

代理后端往往无法把宿主资源搬到另一台机器。把它标成 unmigratable 是有效契约，不能在 migration pre-save 中临时丢弃上下文。若以后引入远程可重连服务，还要证明目标端服务版本与任务身份一致，处理源端未确认的完成。这里没有固定源码事实可直接给出 GPGPU 答案，属于开放问题；本章只提供检查表，不虚构上游已经解决。

## 安全审查从最小设备开始

设备模型面对客户机可控 MMIO、DMA 地址、长度、索引和时序，属于宿主攻击面。每条读取路径都要验证访问大小和范围，每次数值组合都要检查溢出，每个异步回调都要证明对象仍存活。错误路径也应限制 CPU、内存与日志消耗。一个请求即使最终返回错误，如果此前分配了数 GB 临时缓冲区或循环遍历了客户机给出的巨大计数，拒绝服务已经发生。

`edu` 的历史说明教学设备同样会积累安全债务。提交 [`2c5107e1b455`](https://gitlab.com/qemu-project/qemu/-/commit/2c5107e1b455d4a157124f021826ead4e04b4aea) 修正 DMA 上界检查，提交 [`698267415936`](https://gitlab.com/qemu-project/qemu/-/commit/69826741593644f6e9ee735cff37599c33764d67) 处理范围判断的 off-by-one；后者可以由对应的 [qemu-devel 邮件](https://lore.kernel.org/qemu-devel/20221018122551.94567-1-cfriedt%40meta.com/) 复查。它们是上游具体修订，说明“已经有范围检查”仍需追问端点是闭区间还是半开区间、加法会不会回绕。

提交 [`42f599172ae0`](https://gitlab.com/qemu-project/qemu/-/commit/42f599172ae023924f288e20af0ceed681674747) 又把 `edu` DMA 访问限制在设备缓冲区，提交说明指出旧路径可以读写任意 QEMU address-space；讨论可从 [qemu-devel Message-ID](https://lore.kernel.org/qemu-devel/aQtAotYvzFY0Vpft@tcarey.uk/) 追踪。这是上游陈述与代码修复共同形成的证据。它把启示落在边界检查上：DMA helper 会忠实执行模型给出的请求，helper 的存在不会替设备证明目标范围符合其公开契约。

模糊测试应覆盖 MMIO 写序列、队列描述符和 reset 交错。输入格式要能限制队列深度与总步数，避免 fuzzer 只靠资源耗尽制造噪声；崩溃样本要保存 tag、machine 参数和 seed。地址错误、未知操作码、重复 request ID、producer 大跳、完成队列满与中断 mask 都应成为语料。若模型支持 IOMMU，再加入映射建立、权限改变和失效时机，观察失败能否稳定回到设备状态。

安全结论也有证据等级。静态检查无告警，只能说明工具没有命中已知模式；qtest 覆盖越界样本，只能证明这些样本的行为；fuzz 长时间不崩溃，也不证明没有逻辑越权。可接受的结论应写成“在固定版本、固定配置和给定用例中未观察到越界”，再列出尚未覆盖的并发、迁移或代理后端。精确语气比宽泛的“安全”更有工程价值。

:::: {.quick-quiz}
设备调用 `pci_dma_read()` 成功，为什么仍不能据此认定这次读取安全？

::: {.quick-answer}
API 只按设备地址空间执行访问。设备仍需证明长度计算无溢出、地址属于协议允许的缓冲区、权限和生命周期正确，成功传输不等于请求符合设备契约。
:::
::::

## 确定性、浮点与可重复实验

异构计算常涉及浮点、原子操作和并行归约。同一数学表达式在不同执行顺序下可能产生不同最低位，宿主加速库也可能使用不同舍入或非规格化数策略。功能模型若服务于回归测试，应规定数据类型、舍入、NaN 和溢出行为，并用已知向量验证。

对于非确定后端，测试应使用容差和不变量，而不是假定逐位相等；迁移则要么只在队列空时允许，要么把在途工作转换为可序列化中间状态。声明“不支持迁移”也是一种清晰契约，比在迁移中悄悄丢任务更可靠。

整数模型也有细节。向量元素采用 32 位有符号数时，加法是按二进制补码回绕、饱和，还是把溢出作为错误；宿主 C 的有符号溢出不能拿来定义结果，因为那会进入未定义行为。教学 oracle 使用无符号运算表达模 `2^32` 结果，再按协议解释位模式。测试包含零、最大值、符号边界和输入输出重叠，确保实现没有依赖宿主编译器偶然行为。

浮点扩展若以后加入，应明确 IEEE 格式、舍入模式、NaN 传播、subnormal 与 fused operation。并行归约改变结合顺序，绝对误差和相对误差的适用区间也不同；单一 epsilon 常在接近零或大数时失真。比较策略可同时检查分类、有限值容差和问题特有不变量。若 feature 允许 fast-math，应由客户机显式选择，迁移目标也要提供同等语义。

性能数据需要分层。功能模型每秒处理多少元素，主要反映宿主实现和 QEMU 调度，不是虚拟 GPU 性能；时序模型输出的是参数化延迟，不是实测硅片；真实代理测得的吞吐又混有宿主设备与驱动。报告应列出后端、宿主、QEMU tag、队列深度、元素规模和计时来源。只有同层、同契约的数据才适合比较，避免把较快的代理结果包装成模型更正确。

可重复实验还要冻结客户机软件。驱动和测试程序记录 commit，镜像或工具链记录哈希，启动命令保存完整参数。若 CI 使用不同宿主，可以把功能结果设为硬门禁，把耗时只记录趋势或给出宽松上限。出现差异时，先比较输出状态、请求序列和虚拟时间，再看宿主耗时，能更快判断是协议回归还是环境波动。

## 验证从不变量展开

单个 happy path 只能确认一种排列。队列模型适合用不变量驱动测试：在途数量始终位于零到深度之间，一个 request ID 最多完成一次，consumer 从不越过已发布 producer，reset 后旧 generation 不产生客户机可见完成，错误请求不会修改输出缓冲区。每执行一次随机动作便检查这些条件，比等到最终数组错误后再回溯更容易定位首个破坏点。

差分测试把同一合法请求送给纯函数 oracle 与完整设备路径，比较输出和状态；它擅长发现 DMA 拷贝、端序和队列关联错误。metamorphic test 不需要另一个实现，例如向两个输入同时加零应保持结果，分段计算再拼接应等于一次计算，交换加法输入不改变输出。对非法请求则检查关系：把长度从上限增加一，状态应由成功变为范围错误，宿主资源占用仍有界。

qtest 可直接驱动 PCI 配置、BAR 和内存，无需启动完整 Linux，适合 reset 值、访问宽度、中断和 DMA 边界。RISC-V 客户机驱动测试补上发现、fence、IOMMU 和真实队列交互。两者职责不同：qtest 快而精确，系统测试能观察固件与内核契约。把所有断言塞进一个启动镜像，失败时往往只剩“串口没打印”，不利于定位。

演进验证还要对照 Git。每次 ABI 变化先写兼容影响，提交中附测试，review 提出的边界条件回填进不变量清单。上游已有设备的提交历史可以提供问题类别，却不能替教学设备证明正确。以 `edu` 的 ordering、off-by-one 与 DMA 限域修复为例，我们把它们转成模型测试：丢唤醒、末字节访问、地址回绕和越权目标，随后确认任何后端都通过同一前端门禁。

## 让实验结果反过来校验设计

第一个实验故意在宿主侧实现队列协议，不编译 QEMU，也不声称已经接入 PCIe。这样可以快速枚举 enqueue、满队列、非法操作、reset 和完成顺序，把环形队列的不变量稳定下来。观察输出时，应区分“模型拒绝了请求”和“进程异常退出”：前者是协议结果，后者是实现缺陷。只有宿主状态机稳定后，才值得把字段搬入设备模型与客户机驱动。

第二个实验枚举一个简化可见性场景：缺少发布顺序时，设备可能读到旧数据；加入 release 条件后，只保留更新值。它不是完整 RVWMO 模拟器，也不覆盖缓存维护、IOMMU 或真实 QEMU 线程。实验价值在于让读者看见“描述符写完”和“设备一定看到”之间存在契约，再把问题带回驱动 fence、DMA API 与宿主同步三层。若要扩展，可写 riscv64 客户机 litmus，但结论仍要限定在具体平台和配置。

从宿主模型迁移到 QEMU 时，建议一次只加入一层。先实现 PCI 发现与 BAR reset 测试，再加入队列配置但不做 DMA，然后读取一个固定描述符，接着写完成项和中断，末尾才接入向量 oracle。每一步都保留 qtest，失败位置因此短而清楚。若直接把线程、IOMMU、迁移与代理后端一口气合入，任何故障都可能被错误归因给“GPU 太复杂”。

开放问题应跟代码一起维护。真实目标设备是否 coherent，队列是否支持多个优先级，错误是否可恢复，浮点语义如何，迁移是否是产品需求，代理后端能否取消，这些在教学基线中没有上游答案。README 和正文把未知写出来，后续拿到公开规范或邮件结论时再提高证据等级；没有证据时留白，比填一个看似专业的寄存器值更可靠。

## 一次可复查的源码走读

走读可以从 RISC-V `virt` 开始。先在 [`hw/riscv/virt.c`](https://gitlab.com/qemu-project/qemu/-/blob/v11.1.0-rc0/hw/riscv/virt.c) 找 PCIe host 与可选 IOMMU 的创建，确认设备挂载后走哪个 DMA AddressSpace；再到 `gpex.c` 看主桥窗口，随后进入 PCI core 的 BAR 注册与配置访问。此时只回答“设备怎样被发现和寻址”，不要提前猜命令语义。把调用链画成文件、函数、对象和地址空间四列，后续 review 能迅速定位层次。

第二遍从 `edu.c` 的 realize 进入，记录 MemoryRegion 初始化、BAR、中断、timer、thread 与 cleanup；再从 MMIO write 跟踪工作如何被安排，从 DMA timer 跟踪地址怎样读写。每遇到 helper，就打开声明确认参数类型和返回值，特别留意 `dma_addr_t`、长度与错误传播。复制代码前写下“这个模式成立所依赖的前提”，例如 buffer 被限定在固定区间，防止模式脱离前提后变成漏洞。

第三遍读迁移和安全文档。为状态结构中的每个字段标记 guest-visible、derived、host-resource 或 transient：前两类分别考虑保存与重算，后两类通常在目标端重建或排空。然后用安全清单反向审查 MMIO、DMA、队列和 callback。三遍走读各自有问题域，比从一个设备文件首行读到末行更容易看见跨子系统契约。

走读记录必须固定 commit。链接到 `v11.1.0-rc0` blob，旁边写当前符号；若未来行号漂移，符号仍可搜索。看到历史修复时，先用 `git branch --contains` 或固定 clone 验证提交可达，再读 commit message 和邮件。网页搜索结果只能帮助定位，不能作为最终证据。

## 故障注入要卡在状态边界

错误测试若只传一个未知操作码，覆盖面很有限。更有用的注入点在所有权转换附近：描述符前半可读、后半翻译失败；输入 DMA 完成后 reset；计算完成而输出写回失败；完成项写入后中断被 mask；源端进入迁移 quiesce 时客户机再次 doorbell。每个注入点都记录请求阶段、可见内存和预期恢复动作，才能检验状态机。

实现故障注入时不要长期污染客户机 ABI。qtest 可通过构造 IOMMU 映射、受限 RAM 和虚拟时钟制造错误；树外调试属性若确有必要，默认关闭，并明确不迁移、不面向客户机。把“第 N 次 DMA 失败”设成宿主 property，适合重放，却不能让生产设备在未知情况下改变协议。

失败后检查四件事：QEMU 进程存活且资源有界，请求得到一次确定结果，输出只出现规范允许的修改，队列能通过规定 reset 恢复。若错误使设备进入 fatal，后续 doorbell 应被拒绝，直到 reset 清理；若错误只属于请求，其他队列仍可推进。规范没有决定错误域时，先保留开放问题，不靠实现偶然状态代替设计。

日志是故障注入的辅助证据。每个请求用单调 ID、generation 和阶段名关联，不打印完整客户机数据；trace 记录 `FETCH/EXECUTE/WRITEBACK/PUBLISH` 转换、错误类型与队列索引。验证脚本同时检查客户机可见状态和 trace 不变量，客户机结果是接口证据，trace 负责定位内部首个偏差，两者不能互换。

:::: {.quick-quiz}
输出 DMA 在写到一半时失败，只断言设备返回 `DMA_ERROR` 为什么不够？

::: {.quick-answer}
客户机内存可能已经部分改变，队列所有权和后续恢复也可能出错。测试还需检查实际输出范围、完成次数、资源释放与 reset 后是否能继续工作。
:::
::::

## 性能实验先定义问题

性能问题可以分成 vCPU 阻塞、设备吞吐、提交延迟、DMA 拷贝成本和公平性。要研究 doorbell 是否阻塞 vCPU，就测 MMIO 回调返回时间和其他 hart 推进；要研究队列 batching，就固定总元素数，改变每请求规模与深度；要研究临时缓冲区复制，比较相同功能后端的 copy 次数与宿主 CPU 开销。一次基准只回答一个问题，参数全部记录。

计时源也分层。客户机周期计数包含虚拟 CPU 与调度行为，QEMU trace 的虚拟时间适合时序模型，宿主 monotonic time反映实现开销。报告同时写 accelerator、线程配置和宿主信息，不能把 TCG 的客户机 cycle 当作真实 RISC-V 硅片周期。若启用 KVM，设备仍在用户态 QEMU 中运行，vCPU 退出成本又会改变提交路径，结果应单独成组。

基准之前运行正确性和边界测试，结束后复核输出哈希与完成数。一个吞吐很高的版本可能跳过长度检查、合并错误请求或丢中断，单看每秒操作数会奖励错误。性能 patch 要附带不变量测试不变的证据，新增缓存或 batch 时再检查 reset 与迁移状态是否扩大。

确定性功能模型的性能数据适合比较实现变化，不适合预测硬件。时序模型可研究驱动在不同延迟和队列深度下的重叠，参数应叫 model latency/bandwidth，避免使用未经校准的芯片型号。代理后端结果必须附宿主 GPU 与软件栈，数值和资源限制也可能随版本变化。

## 从树外模型准备一个可审查系列

若教学模型未来准备投稿，第一个补丁可以加入设备规范文档，写清 ABI、错误、reset 和迁移策略；第二个补丁加入最小 PCI 设备与 MMIO qtest；第三个补 DMA 与边界测试；第四个补异步执行和并发测试；其后再加入 RISC-V `virt` 的显式测试命令或文档。每个 commit 可构建、可运行，review 能分别质疑协议、总线接入和后端。

设备若是标准 PCI 类型，核心实现不应塞进 `hw/riscv/`。RISC-V 主线体现在使用 `virt`、riscv64 驱动、RISC-V IOMMU 与 fence 验证；设备代码可放合适通用目录。只有 platform-specific 的 MMIO/IRQ 装配才进入 Machine。这样的目录边界使未来其他体系结构平台也能挂载设备，同时避免复制两份状态机。

cover letter 应说明这是教学或研究设备、有哪些公开软件、如何运行测试、哪些能力缺失。若没有真实硬件规范，就不能提交一个冒用厂商名称的 device ID；可保留 book tree-out。性能、迁移和安全结论按已验证范围陈述。审查意见需要回填文档和测试，不能只在邮件中口头接受。

上游是否愿意接收该类型设备是开放问题。当前 `edu` 展示了教学 PCI 设备能够存在，不能据此推断任何新教学加速器都会被接受。投稿前通过 `MAINTAINERS` 找相关维护者，在 qemu-devel 先讨论用例与接口，得到的回复标作上游陈述；未收到回复不能视为批准。

## 实验记录模板

每次实验开头写 QEMU tag或说明“host-only”、仓库 commit、宿主系统、Python/编译器版本、命令与预期。中间按准备、动作、观察、解释和边界记录。观察保存原始输出，解释必须引用观察；若状态与预期不同，先标 failure，不修改预期去迎合输出。实验结束列出可重复条件与清理步骤。

队列实验至少保存深度、输入向量、动作序列、每步 producer/consumer/generation 和完成项。可见性实验保存枚举规则与全部允许结果，不能只截取符合观点的一次输出。未来 QEMU 实验再加 machine 参数、设备属性、客户机镜像哈希、IOMMU 映射和 trace event 列表。

正文里的中文描述承担原理、观察和推理，英文 README 承担可执行命令、文件结构和预期输出。二者通过稳定相对链接连接，README 改动后要检查正文仍与真实能力一致。若实验从 host-only 扩展到 QEMU 集成，应在同一 README 增加新阶段，并保留旧阶段边界，避免历史读者误以为早期结果已经覆盖 DMA。

复现实验的人若得到不同结果，应先比较版本与动作序列，再报告最短差异。Issue 模板可要求附 seed、完整命令和错误阶段，不上传受限镜像或客户机数据。能够重放的反例会进入测试，无法重放的观察保留为线索，证据等级不越过实际材料。

## 章节边界账本

读完后可以把结论压成一张账本。已由固定源码确认的，是 RISC-V `virt` 的 PCIe/IOMMU 接入条件、通用 PCI/DMA/迁移接口，以及 `edu` 的当前代码与历史修复；由本书实验确认的，是宿主侧有界队列和简化可见性枚举；作者方案包括教学设备 ABI、执行上下文、reset 和迁移阶段；真实 GPGPU 的指令、coherence、时序和代理协议仍是开放问题。四栏长期并列，更新版本时逐项迁移，不能整章一次性改成“已验证”。

读者可选择一个设计决定做反证。若认为零拷贝一定优于复制，构造 reset 与迁移插入，列出映射生命周期和清理成本；若认为 doorbell 回调直接计算更简单，测量长请求对其他 vCPU、timer 和 QMP 的影响；若认为一个完成中断足够，注入丢失与合并，检查驱动能否靠完成索引前进。反证写进实验，比在段落中争论偏好更有说服力。

章节更新也要保留旧结果。QEMU 后续若合入真正的相关设备，新增当前版本小节，列出 commit、设备契约和与本书模型的差异；不能倒写成 `v11.1.0-rc0` 已经具备。实验目录可增加新的集成阶段，host-only 队列仍作为状态机单测。历史可复现，读者才看得见设计如何从概念走向上游。

还有一项适合留给读者的设计题：如果设备增加第二条命令队列，应共享一个 MSI-X vector，还是每队列独立通知；reset 是按队列还是按 function；完成顺序只在队列内保证，还是跨队列也有全序。先写两种 ABI 和状态图，再为队列满、并发 reset、跨队列 DMA 别名各设计一个反例。没有真实规范时不必选“唯一正确”方案，答案质量取决于约束是否完整、测试能否推翻实现，而非寄存器数量。

:::: {.quick-quiz}
为什么增加第二条队列会影响迁移状态，而不只是多保存一组 head/tail？

::: {.quick-answer}
还会引入跨队列完成顺序、共享后端资源、IRQ pending、reset 范围和在途任务归属。若迁移只复制两个索引，目标端可能以不同顺序发布结果或重复通知。
:::
::::

:::: {.quick-quiz}
功能正确的宿主 GPU 代理为何仍可能破坏可重复性？

::: {.quick-answer}
并行归约顺序、浮点模式、驱动和硬件实现可能不同，逐位结果与完成时序会变化。设备契约必须规定可比较范围，并在测试和迁移策略中显式处理非确定性。
:::

## 小结

异构设备建模沿着有界、可验证的命令与内存协议推进。控制面稳定以后，计算后端才有条件从简单功能模型演进到时序近似或真实代理。
