# 后记 {.unnumbered}

\markboth{后记}{后记}

一封补丁邮件发出去以后，最先等来的回复常常只有几行。某个错误路径没有回滚，设备树里的地址和数据手册对不上，migration state 漏了一个客户机可见字段，或者 patch 过大，reviewer 无法判断哪一处改变了行为。第一次遇到这些回复，人很容易把它们理解成否定。继续做几轮以后会发现，公开审查正在帮助我们把脑中的假设变成别人能够检查的工程事实。

QEMU 的设计就沉淀在这样的往返中。一个作者带着真实需求提交 v1，reviewer 从另一种固件、另一种 host 或旧 machine 的迁移路径提出反例，maintainer 再判断这个接口是否值得长期维护。最终进入源码的内容可能只占讨论的一小部分，那些被删除的字段、推迟的优化和补上的测试，同样塑造了今天的边界。

## 技术能力会在协作中改变 {.unnumbered}

独自阅读源码时，我们很容易接受一条能够自圆其说的解释。准备向上游提交时，问题会变得严格：这个现象能否复现，修改属于哪位 maintainer 的范围，commit message 是否说清原因，测试能否在别人的机器上运行，兼容性结论有没有覆盖旧版本。写代码、做实验、组织证据和回应异议由此连在一起。

参与开源带来的成长也不只体现在掌握更多 API。我们会学着在证据不足时停下来，在别人指出错误时迅速修正，在意见不同的时候讨论约束和反例，也会逐渐理解维护者为什么对一个看似优雅的重构保持谨慎。大型软件依靠这些习惯跨过人员、公司和硬件世代继续演进。

作者参与 QEMU RISC-V 子系统的 review 与维护后，对这一点有更深的感受。RISC-V 生态仍在快速变化，新规范、新硬件和新客户机软件不断进入视野。补丁需要给未来留出空间，也必须照顾已经发布的 QEMU 和正在使用的固件。Reviewer 的工作并非替作者给出标准答案；更多时候，是帮助双方把还没说清的假设找出来，共同决定项目愿意承担哪一部分长期成本。

## Agent 让检索变快，责任没有转移 {.unnumbered}

编写这本书时，Agent 能够同时搜索提交、邮件、规范和源码，过去需要数天整理的线索，现在可能在一轮研究中形成索引。信息稀缺不再是主要困难，新的困难是判断一段材料能支持多远，以及它与当前版本是否仍然一致。

这也是全书反复强调证据边界的原因。Agent 可以指出一条值得阅读的线程，不能替我们确认作者究竟说了什么；它可以生成一个实验草案，不能替我们观察真实输出；它可以帮忙比较 patch 版本，不能替贡献者承担 DCO、许可和技术正确性。QEMU 当前对生成式 AI 内容有明确的 provenance 政策，准备上游贡献时必须尊重这条边界。

工具越强，提出问题、核验结果和诚实表达不确定性的价值越高。

## 从一次小贡献开始 {.unnumbered}

进入 QEMU 社区不需要先理解几百万行源码。一次文档修正、一个能够稳定复现的测试、对错误信息的改进，或者在自己熟悉的 RISC-V 扩展上核对一处规范差异，都可以成为起点。把范围缩小，找到对应 maintainer，说明复现环境，发送一组便于审查的补丁，然后认真处理每条回复。这个过程本身就是学习系统软件的路径。

读完本书以后，读者也许不会记住每个结构体的字段，但应当能够画出一条 RISC-V 指令经过 TCG 或 KVM 的执行边界，能够解释设备寄存器、IRQ、DMA、reset 和迁移状态怎样组成模型，能够用 monitor、log、trace event 与 gdb stub 把一次 guest 故障逐层收窄。更进一步，还能打开 Git 和邮件归档，判断当前实现经历过哪些选择，哪些结论仍然需要新的实验。

QEMU 是这次探索的对象。真正值得带走的，是面对复杂系统时仍愿意从问题出发，耐心找到证据，把自己的判断交给公开讨论，并在合作中继续修正它。这样的能力不会随着某个函数改名而过期。
