# QEMU 的问题边界与 RISC-V 主线

在终端里运行一条 `qemu-system-riscv64 -machine virt`，宿主机只多出一个普通进程。可这个进程很快就会分裂出几种完全不同的工作：它要读懂几十乃至上百个命令行选项，要拼出客户机能看见的 CPU、内存和外设，要让客户机指令向前执行，还要在磁盘、网络、定时器、中断与管理命令之间来回切换。程序一旦卡住，表面现象常常只有一句“虚拟机没有输出”，根因却可能落在固件入口、设备树、地址映射、vCPU 线程、字符后端，甚至宿主内核的虚拟化能力上。

因此，阅读 QEMU 的第一步并不是打开某个巨大的源文件，从 `main()` 一路单步。更有用的起点，是先画清它解决什么问题，每一层向上一层承诺什么，又把哪些工作交给下一层。边界画清以后，函数名会逐渐落到合适的位置；边界没画清，调用栈越长，脑子里的机器反而越模糊。

## 本章目标

读完本章，读者应当能够区分系统模拟、用户态模拟与硬件辅助虚拟化，能够说明 TCG 和 KVM 共享哪些机器状态、又在哪个位置分流；能够把 RISC-V `virt` 看成一份客户机平台契约，而不是某块实体开发板的照片；还应当建立版本、静态源码、运行时状态和历史证据四种坐标。后续章节会沿这些坐标展开，不再每遇到一个新缩写就重新搭一遍世界观。

## 先问清楚：我们究竟在模拟什么

QEMU 这个名字经常被笼统地解释为“模拟器”，但同一份工程里至少存在三种使用边界。`qemu-system-riscv64` 面向完整系统，它需要给客户机提供处理器、物理内存、固件入口、中断控制器、串口、存储与网络设备。客户机内核看到的是一台机器，应用程序看到的是客户机内核提供的系统调用。即使屏幕上只跑了一个裸机小程序，系统模拟仍然保留了“CPU 通过客户机物理地址访问设备”这条基本关系。

`qemu-riscv64` 这样的 Linux user-mode 程序则不同。它接收一份 RISC-V Linux 用户态可执行文件，翻译其中的客户机指令，并把客户机系统调用适配到宿主系统调用。这里没有 RISC-V `virt` 主板，没有 PLIC，也不会因为应用调用 `write()` 就真的驱动一块虚拟 UART。它解决的是“让另一种指令集的用户程序在当前操作系统进程语义里运行”。两者都可能使用 TCG，却模拟了不同的抽象层，调试方法也不能混用。

第三种常被放在同一句话里的能力，是 KVM 硬件辅助虚拟化。它对应一条不同的 CPU 执行路径，并不另造 Machine 类型。在宿主和客户机体系结构、内核能力以及 QEMU 支持条件匹配时，QEMU 可以把 vCPU 的大部分执行交给 `/dev/kvm`，自己继续维护设备模型、固件装载、管理面与大量平台状态。于是，“系统模拟还是用户态模拟”回答的是客户机边界，“TCG 还是 KVM”回答的是系统模拟内部怎样推进 CPU。把这两个问题交叉起来，许多似是而非的争论会自然消失。

:::: {.quick-quiz}
一条 RISC-V 用户态程序在 `qemu-riscv64` 下能够打印文本，能否据此证明 RISC-V `virt` 的串口设备已经工作？

::: {.quick-answer}
不能。Linux user-mode QEMU 适配的是客户机应用程序的系统调用，文本通常经宿主进程的文件描述符输出，没有建立 `virt` 机器，也没有经过客户机 UART 的 MMIO 寄存器。验证串口设备要使用 `qemu-system-riscv64`，并观察客户机物理地址访问、中断和字符后端。
:::
::::

本书讨论的是完整系统模拟，主命令固定为 `qemu-system-riscv64`，主机器固定为 `virt`。用户态模拟只在解释 TCG 的复用边界时出现，不把它扩展成另一条叙事线。这样的取舍让 CPU、MMU、设备、固件和虚拟化能够在同一台机器上互相印证。

## 一个进程里的四类责任

把 QEMU 进程先粗分成四类责任，比背目录更可靠。第一类是配置：命令行、配置文件、QMP 命令以及编译期选项，最终都要落成类型、属性和后端对象。第二类是造机器：Machine、CPU、总线、内存区域、设备和固件数据要按依赖关系创建、连接并实现。第三类是执行：vCPU 需要获得寄存器和地址空间，持续运行，遇到异常、退出或外部事件时返回合适的控制层。第四类是 I/O 与控制：块设备、网络、字符设备、定时器、迁移、监控器和生命周期事件要在并发环境中得到服务。

这四类工作没有严格对应四个目录，也不会按顺序只执行一次。启动阶段以配置和造机器为主，运行阶段执行与 I/O 互相唤醒，热插拔又会把配置和造机器带回来，迁移则横跨全部层次。所谓“边界”，说的是状态归谁拥有、由谁改变、通过什么接口观察，并不等于给每一层砌一堵不能跨越的墙。

在 `v11.1.0-rc0` 中，进程入口位于 [`system/main.c`](https://gitlab.com/qemu-project/qemu/-/blob/v11.1.0-rc0/system/main.c)，`main()` 调用 `qemu_init()` 完成主要初始化，再进入主循环。大量启动编排仍可在 [`system/vl.c`](https://gitlab.com/qemu-project/qemu/-/blob/v11.1.0-rc0/system/vl.c) 找到，机器类型的公共定义在 `hw/core/machine.c` 一带，RISC-V `virt` 的板级实现则集中于 [`hw/riscv/virt.c`](https://gitlab.com/qemu-project/qemu/-/blob/v11.1.0-rc0/hw/riscv/virt.c)。这一组路径只表示职责落点，不能据此断言运行时永远沿目录层次调用；QOM 回调、MemoryRegion 操作和事件循环会不断穿越它们。

## TCG：把客户机执行变成宿主可运行的代码

TCG 的任务不是“模拟所有硬件”，它主要负责客户机指令执行。在 RISC-V 主线里，前端解码客户机指令，把指令语义转换为中间操作，再由宿主后端生成宿主代码。翻译块缓存让一段反复执行的客户机代码不必每次重新解码；控制流、异常、页表权限、代码失效和外部中断，又决定一个翻译块能安全运行到哪里。后文会逐项拆开，这里先记住一个范围：TCG 加速的是 CPU 模型的执行，UART、PLIC、virtio 等设备并不会因此自动变成“指令翻译的一部分”。

TCG 的价值也不只在“没有硬件虚拟化时还能运行”。当宿主与 RISC-V 客户机使用不同 ISA，KVM 无法把客户机指令直接交给宿主 CPU，TCG 仍能提供跨体系结构执行；当开发者需要观察每条指令、验证新扩展、注入异常，软件执行又给出了更可控的插桩位置。代价同样明确：翻译、状态同步、地址转换和退出检查都消耗宿主资源，客户机与宿主的并行语义还必须由软件谨慎保存。

当前 RISC-V TCG 相关实现分布在 `target/riscv/` 与 `target/riscv/tcg/`，CPU 类型及公共状态可从 [`target/riscv/cpu.c`](https://gitlab.com/qemu-project/qemu/-/blob/v11.1.0-rc0/target/riscv/cpu.c) 和 `cpu.h` 进入。通用 TCG 加速器操作位于 [`accel/tcg/`](https://gitlab.com/qemu-project/qemu/-/tree/v11.1.0-rc0/accel/tcg)。阅读时要留意“体系结构状态”和“执行后端状态”的差别：`CPURISCVState` 中可观察的寄存器、CSR 与特权状态需要符合 RISC-V 语义，翻译块哈希、宿主代码缓存等则属于实现机制，不应泄漏成客户机 ABI。

## KVM：CPU 进入内核，机器没有消失

在具备 RISC-V KVM 支持的宿主上，QEMU 可以用 `-accel kvm` 打开 `/dev/kvm`，创建虚拟机和 vCPU，把寄存器、内存槽及中断等状态与内核接口对接。vCPU 线程通过 KVM ioctl 进入内核运行客户机，当发生需要用户态处理的 MMIO、关机、调试事件或其他退出原因时，再返回 QEMU。此时客户机指令不是由 TCG 翻译，但命令行解析、QOM 对象、`virt` 地址布局、固件与设备后端仍然存在。

这解释了为什么 KVM 不能被描述成“绕过 QEMU”。它绕过的是一部分用户态 CPU 执行工作，并没有绕过机器创建。一个 `virtio-blk` 请求可能由客户机驱动在 KVM vCPU 上发起，设备寄存器或共享队列却仍需 QEMU、vhost 后端或内核子系统处理。中断随后注入 vCPU，客户机继续执行。边界上的往返次数、状态同步量以及能否把数据面下沉，直接影响性能，也形成了后续许多工程设计。

RISC-V KVM 的用户态对接可从 [`target/riscv/kvm/`](https://gitlab.com/qemu-project/qemu/-/tree/v11.1.0-rc0/target/riscv/kvm) 和 [`accel/kvm/`](https://gitlab.com/qemu-project/qemu/-/tree/v11.1.0-rc0/accel/kvm) 进入。前者处理 RISC-V 寄存器、扩展与 vCPU 特有能力，后者提供 QEMU 各体系结构共用的 KVM 框架。`accel/kvm/kvm-all.c` 并不懂完整的 RISC-V ISA，`target/riscv/kvm/kvm-cpu.c` 也不该重新实现通用机器创建；这类切分正是跨体系结构复用与体系结构特化之间的一次取舍。

:::: {.quick-quiz}
同一条 `-machine virt -device virtio-net-device` 命令分别配合 TCG 与 KVM 时，哪些东西通常保持不变，哪些东西必然发生变化？

::: {.quick-answer}
客户机可见的 `virt` 平台契约、设备类型、主要地址布局和后端配置应尽量保持一致，CPU 执行路径必然不同：TCG 经过翻译与软件 CPU 循环，KVM 通过内核 vCPU 运行并处理退出。CPU 可用扩展、计时细节、调试能力和某些中断实现也可能随加速器能力变化，不能仅凭相同命令行假定两次运行逐位等价。
:::
::::

## 共享边界与分流点

理解两种加速器，最实用的办法是沿一次设备访问追踪。客户机执行一条访问 UART 寄存器的指令。在 TCG 路径里，翻译后的访存帮助函数完成地址转换，进入 QEMU 的物理地址空间，最终调用 UART 的 MemoryRegion 回调。在 KVM 路径里，若该地址没有由内核直接处理，vCPU 以 MMIO 退出回到用户态，QEMU 根据退出信息访问同一设备模型。入口不同，后半段可能汇合。

“可能汇合”比“总是汇合”更准确。KVM 可以使用 irqchip、内存槽等内核能力，vhost 能把一部分 virtio 数据面移到内核，设备还可能由 VFIO 交给真实硬件。每次下沉都减少用户态往返，却带来能力协商、迁移状态、错误恢复和安全隔离的新问题。判断某段状态属于哪里，必须看当前加速器、设备后端和命令行，不能只看设备在对象树中的名字。

一份可靠的分层记录至少回答三个问题。客户机能观察到什么，例如寄存器、地址、中断和时间；状态的权威副本在哪里，例如 QEMU 对象、KVM vCPU 或后端线程；状态在什么事件上同步，例如进入 vCPU、KVM exit、迁移停机或设备 reset。只写“这里调用了 KVM”无法解释错误，也无法判断迁移时少保存了一项状态会造成什么后果。

## `virt` 是平台契约，不是开发板照片

固定 tag 下的 [`virt` 平台文档](https://gitlab.com/qemu-project/qemu/-/blob/v11.1.0-rc0/docs/system/riscv/virt.rst) 明确说明，它不对应真实硬件，而是为虚拟机设计的通用平台。它通过生成的设备树向固件和操作系统描述 CPU、内存与设备，并可以随所选功能加入 PCIe、virtio、IOMMU 等节点。对读者来说，这是一项优势：不必先追查某家芯片手册的勘误，就能观察规范化的启动和 I/O 路径。但“虚拟”不代表可以随意变动，一旦固件和内核依赖某个可见行为，它就进入了兼容性讨论。

在当前 `virt.c` 的 `virt_memmap[]` 中，启动 ROM 从 `0x1000` 开始，测试设备位于 `0x100000`，RTC 和串口分别落在 `0x101000`、`0x10000000` 附近，CLINT、PLIC 或 AIA 占据各自的中断区域，八个 virtio-mmio 窗口从 `0x10001000` 排列，DRAM 从 `0x80000000` 开始。PCIe ECAM 和 MMIO 窗口、flash、fw_cfg 等也在这份表里占位。具体数字应以固定 tag 的源码与运行时设备树为准，不能从另一版博客复制过来。

地址表只是骨架。`virt_machine_init()` 会根据 socket、hart 数、AIA、ACLINT、IOMMU、固件和其他选项创建不同对象，并把它们连接到系统内存与中断拓扑；`virt_machine_class_init()` 则设置机器类的初始化函数、默认 CPU、最大 CPU 数以及可配置属性。机器因此是“类型定义、实例属性和运行时能力”共同计算出来的结果，不是一张永远不变的静态清单。

:::: {.quick-quiz}
为什么只读 `virt_memmap[]` 仍不能完整回答“这次启动有哪些设备”？

::: {.quick-answer}
地址表只给出板级资源的候选位置。实际对象还受命令行属性、编译配置、加速器能力、固件选择和条件分支影响，有些设备通过总线动态创建，有些功能关闭后不会实现。应同时核对 `virt_machine_init()`、生成的设备树、`info qom-tree` 与 `info mtree`。
:::
::::

这里还要修正一个容易沿用自其他体系结构的说法：当前 RISC-V `virt` 的类型名没有公开一串带版本的 machine 名称。可是客户机可见平台仍然构成兼容性表面，地址、设备树、默认属性和迁移状态的修改仍要审查。能否通过显式版本机型冻结行为，与“某项行为是不是 ABI”是两个问题。后文凡谈 machine version，都会先核对 RISC-V 当前实现，不借用其他目标的机制填空。

## 为什么全书选择 RISC-V

RISC-V 把非特权指令、特权级、页表、虚拟化扩展和平台规范分成相对清楚的文档层次。遇到一条访存，我们可以先问指令语义，再问当前特权级与 `satp`/`hgatp` 的地址转换，最后问 QEMU 的物理地址空间；遇到中断，也能区分 ISA 定义的陷入规则与 `virt` 平台采用的 CLINT、PLIC 或 AIA 模型。这种层次感很适合把“规范要求”“QEMU 实现”和“工程选择”分开。

选择 RISC-V 也意味着承担它仍在快速演进的现实。扩展组合、AIA、IOMMU 与 H 扩展的支持会随 QEMU、内核和规范版本变化，默认 CPU 能力不应靠记忆推断。正文固定以 QEMU `v11.1.0` 为参考版本，预研与源码链接锚定官方 GitLab 的 `v11.1.0-rc0`；正式 tag 可用后，发布流程应复核差异再统一切换。实验命令还要保存 `-version`、`-machine help` 和 `-cpu help` 输出，给结果留下环境指纹。

需要确认 QEMU 行为时，先看官方 GitLab 固定 tag 的源码与 Git 历史；需要确认设计动机时，继续找 qemu-devel 的补丁说明和 review；涉及 ISA、OpenSBI 或 Linux 启动约定，再回到各自官方规范与仓库。二手文章可以帮助发现关键词，却不能代替最终证据，实验环境也不应依赖未经说明的非上游补丁树。

## 从宿主坐标切换到客户机坐标

初学者最容易在“地址”和“线程”两个词上混淆坐标。宿主虚拟地址属于 QEMU 进程，客户机虚拟地址由 RISC-V 页表解释，客户机物理地址再落入 `AddressSpace` 与 MemoryRegion。KVM 还引入内核维护的内存槽和 vCPU 状态。日志里同样一个十六进制数，如果没有注明坐标，很可能被错误地拿去与另一层的映射比较。

线程也一样。主循环线程处理大量控制与事件，vCPU 线程推进客户机 CPU，IOThread 可以拥有自己的 `AioContext`，块后端还可能使用线程池。客户机里的 hart 是客户机并行单元，不等于某个固定宿主线程；在不同加速器和 TCG 配置下，它们的映射会变化。实验记录应写“客户机 hart 1 在这次运行中由宿主线程 1234 推进”，而不要写成永久规则。

这两组坐标可以组合成一张问题表：谁在什么执行上下文里，读写哪一层地址空间中的哪份状态。后面的 QOM、主循环、内存和设备章节看似分别讲对象、线程与地址，最终都要回到这句话。它也是审查并发错误时最省时间的提问方式。

## 启动、运行与管理面不是三套机器

启动阶段创建的 QOM 对象，运行阶段仍然承载状态；QMP 查到的属性，并非另一份管理数据库；迁移保存的设备字段，也必须对应客户机已经观察到的行为。QEMU 把这些阶段放在同一进程内，能直接共享对象和生命周期，却也要求初始化次序、错误回滚与并发规则足够清楚。

例如，某个设备属性在 realize 前可以改变，用来决定寄存器窗口大小；设备实现后再修改，就可能与已经生成的设备树、已经注册的 MemoryRegion 不一致。QOM 负责属性和对象关系，qdev 给设备增加 realize、总线与 reset 语义，machine 负责安排创建顺序。看见一个字符串属性时，不能只问它在哪里解析，还要问它在什么阶段冻结，改变后有哪些下游消费者。

管理面同样会反向影响执行。QMP 可以请求暂停、热插拔、迁移或查询状态，主循环要把这些动作与 vCPU、IOThread 协调。于是，“控制路径不在性能关键路径”只是局部判断；一次错误的锁顺序或生命周期处理，仍可能让数据面停住。第三章会把这些执行上下文展开，本章先保留这条整体联系。

## 从 2018 年的两组补丁看边界如何长出来

QEMU 的 RISC-V 支持不是从今天的目录结构一次生成的。提交 [`dc5bd18fa57254e4b8597747c2100c92a55fc409`](https://gitlab.com/qemu-project/qemu/-/commit/dc5bd18fa57254e4b8597747c2100c92a55fc409) 在 2018 年引入 RISC-V CPU 核心定义，提交 [`04331d0b56a0cab2e40a39135a92a15266b37c36`](https://gitlab.com/qemu-project/qemu/-/commit/04331d0b56a0cab2e40a39135a92a15266b37c36) 引入早期 `virt` 机器。后者的提交说明直接列出设备树、CLINT、PLIC、16550A UART 和 VirtIO MMIO 等板级组成。历史顺序让 CPU 语义与板级机器的边界变得可见：平台使用 CPU 类型，却没有把整块主板塞进 CPU 模型。

随后 PCIe、OpenSBI、AIA、IOMMU 等能力逐步进入 `virt`。只看当前 `virt_machine_init()`，这些分支像是理所当然；沿 `git log -- hw/riscv/virt.c hw/riscv/boot.c` 回看，才会发现每次增加平台能力都要处理设备树描述、默认行为、错误路径和兼容性。历史记录不能自动给出唯一动机，但能排除“从第一天就设计完整”的神话，也能找到当时 review 真正争论的接口。

::: {.design-note}
已核对的事实是，早期 CPU 与 `virt` 机器由不同提交引入，当前源码仍把 `target/riscv/` 的体系结构实现与 `hw/riscv/virt.c` 的板级装配分开。上游提交说明强调了 `virt` 的具体平台组件。由这些证据可以作出一个强推断：这种分层既服务代码复用，也把 ISA 语义与客户机平台 ABI 的审查范围隔开；不过，若要把某次具体重构归因于兼容性压力，还需要对应邮件中的明确陈述，不能用目录结构替作者发言。
:::

## KVM 支持展示了第二条执行维度

RISC-V KVM 支持在 2022 年的一组补丁中进入 QEMU。提交 [`91654e613bf`](https://gitlab.com/qemu-project/qemu/-/commit/91654e613bf) 建立公开的 KVM 接口，后续提交完成 vCPU 初始化、直接内核启动与虚拟时间处理，最终由 [`fbf43c7dbf`](https://gitlab.com/qemu-project/qemu/-/commit/fbf43c7dbf) 启用 RISC-V KVM 加速器。系列邮件可从 Message-ID [`20220112081329.1835-14-jiangyifei@huawei.com`](https://lore.kernel.org/qemu-devel/20220112081329.1835-14-jiangyifei%40huawei.com/) 回到上下文与 review。

这组演进很适合校正“加一个 KVM 开关就完成硬件虚拟化”的想象。vCPU 寄存器接口、客户机启动状态、时间、退出处理和 capability 探测都要分别对接，其中直接内核启动还会改变 reset 时由谁把哪个入口写入 PC。机器仍叫 `virt`，执行的权威层却部分移到宿主内核。两条维度在这里相交：一条是 TCG 与 KVM 的加速器选择，另一条是 RISC-V H 扩展定义的客户机虚拟化语义。前者属于 QEMU 如何执行，后者属于客户机体系结构能够表达什么，后续硬件虚拟化章节会专门并排讨论。

::: {.design-note}
上游系列能直接支持的陈述是：RISC-V KVM 接入需要公开接口、vCPU 初始化、启动与计时等多块工作。当前源码也能证明 TCG 与 KVM 各有体系结构后端。作者据此推断，QEMU 刻意让大部分 machine 和设备模型留在公共层，以避免为每种执行后端复制一台机器；这项推断与代码形态一致，但不应伪装成某位维护者逐字说过的话。
:::

## 当前代码、上游陈述与作者推断要分栏保存

技术书最危险的句式，是把“我在当前代码里看见了”悄悄写成“项目从来就是为了”。本书把证据分成三层。第一层是可重复核对的事实，例如某 tag 下的函数、数据结构、命令输出和规范条文。第二层是上游明确陈述，包括提交说明、cover letter、review 回复和维护者文档。第三层才是作者推断，用来把事实串成工程解释，并明确写出推断依赖什么、还存在哪些替代解释。

Git 历史同样需要谨慎使用。一次提交把函数移到新文件，能证明移动发生，也可能在提交说明里解释目的；它不能单独证明最初设计者十年前的全部意图。`git blame` 适合定位最后改动某一行的提交，不适合给复杂架构寻找唯一作者。遇到大规模重排，先用 `git log --follow`、按符号搜索和补丁系列还原上下文，再去邮件列表确认 review 中的约束。

引用邮件时保存 Message-ID，而不是只保存搜索页面。Message-ID 是跨归档站点较稳定的标识，读者可以在 lore 或 GNU qemu-devel 归档中重新定位线程。引用源码则保存官方 GitLab 的 tag 或完整 commit，避免链接到不断前移的 `master` 行号。实验输出还要附命令和版本；没有这些元数据，再精确的截图也很难复现。

## 一套不会很快迷路的阅读顺序

面对新组件，可以先从客户机可见现象反推。以串口为例，先确认设备树给出的 compatible、地址和中断，再在 `info mtree` 中找对应 MemoryRegion，在 QOM 树里找设备实例，然后定位 realize 函数注册的回调，最后追到字符后端。若现象只在 KVM 出现，再检查 MMIO exit、irq 路由和加速器能力。这样每一步都有运行时证据，不必在全仓库里凭函数名游泳。

另一条顺序从生命周期出发：类型何时注册，实例何时创建，属性何时设置，realize 何时把对象接入机器，reset 如何恢复客户机可见状态，unrealize 与 finalize 怎样释放资源。设备初始化问题沿这条路更清晰，并发与热插拔问题也能找到“谁还持有引用”。第四章会完整使用这条线。

还有一条顺序从数据所有权出发：谁产生状态，谁读取状态，谁能并发修改，迁移时谁序列化，reset 时谁清空。它尤其适合 KVM、vhost 和 IOMMU，因为同一个客户机动作会穿过用户态、内核与后端。三条顺序并不冲突，现象、生命周期和所有权常常在同一个关键函数会合。

## 实验一：比较 TCG 与 KVM 的公共机器边界

::: {.hands-on}
本实验使用中文完成分析，命令、目录和验收项见英文手册 [Compare accelerators](../experiments/part-01-system-foundations/chapter-01-qemu-boundaries/compare-accelerators/README.md)。先在任意宿主上以 TCG 启动最小 RISC-V `virt`，保存完整命令、QEMU 版本、`query-accels`、`info qom-tree`、`info mtree` 和宿主线程列表；若有支持 RISC-V KVM 的宿主，再以等价机器参数运行 KVM。不要为了得到两份输出而在不支持 RISC-V KVM 的机器上伪造结果，缺少硬件条件时，把 KVM 步骤标记为待验证，并分析 capability 探测的失败位置。

对照时分成三栏。第一栏记录两次都存在的机器与设备对象，第二栏记录执行后端、vCPU 线程和内核接口差异，第三栏记录暂时无法解释的变化。重点观察 DRAM、UART、virtio-mmio 等 MemoryRegion 是否仍处于相同客户机物理地址，观察 CPU 属性和线程名怎样变化。实验报告最后给出一条带限定条件的结论：哪些结果支持“公共机器模型”，哪些结果只能说明本次配置相同，不能外推到所有 KVM 能力。
:::

这个实验的目标不是证明 KVM 更快。性能比较需要统一宿主、客户机负载、计时源和预热条件，本章尚未建立这些控制变量。此处只验证边界：更换执行后端之后，哪些客户机平台对象仍由同一套板级代码建立，哪些状态移入了另一条执行路径。

## 实验二：从一次 `virt` 启动建立四种坐标

::: {.hands-on}
按照英文手册 [Trace a RISC-V virt boot](../experiments/part-01-system-foundations/chapter-01-qemu-boundaries/trace-riscv-virt-boot/README.md) 准备实验，在正文笔记中使用中文记录。固定 `v11.1.0-rc0` 源码或等价构建产物，以最小固件或测试镜像启动 `virt`，同时收集生成的 DTB、`info mtree`、QOM 树、线程信息和可用的启动 trace。为每条观察补四种坐标：QEMU tag/commit、静态文件与符号、运行时对象或事件、能够解释设计演进的 Git 提交或 qemu-devel Message-ID。

选择 UART、DRAM 和一个中断控制器作为三个锚点。先从设备树读取客户机视图，再回到 `virt_memmap[]` 与 `virt_machine_init()`，最后在运行时输出里闭环。若三处信息不一致，不要立刻判断 QEMU 出错，先确认是否用了不同二进制、是否有命令行覆盖、是否把宿主地址误当成客户机物理地址。报告应保留失败尝试，因为“为何没有对应上”往往比一张正确截图更能训练坐标切换。
:::

实验可以很小，但必须能够推翻一个判断。只抄命令和预期输出，得到的是演示；先写假设，再用独立观察验证，才开始接近分析。例如可以假设“切换 TCG/KVM 不改变 UART 的客户机物理地址”，再分别从 DTB 和 `info mtree` 检查。若只能运行 TCG，结论就应缩小为“当前 TCG 配置与源码表一致”，把未验证部分明确留下。

## 进一步思考：稳定到底稳定什么

:::: {.quick-quiz}
RISC-V `virt` 不对应实体主板，而且当前没有公开的逐版本机型名称，是否意味着上游可以随时改动默认地址而无需兼容性讨论？

::: {.quick-answer}
不意味着。固件、内核、设备树处理和迁移数据都会观察平台行为，默认地址一旦被广泛依赖，就可能形成事实上的 ABI。是否存在显式版本机型，只影响兼容行为如何选择，不会消除兼容性本身。判断某次改动能否接受，要查看当前实现、用户可见影响和上游 review，不能从类型名推导。
:::
::::

:::: {.quick-quiz}
如果一项设备功能被 vhost 或 KVM 下沉到内核，还能否说它属于 QEMU 虚拟机的一部分？

::: {.quick-answer}
可以，但要说清层次。QEMU 仍负责创建虚拟机、协商能力、配置资源、管理生命周期和迁移边界，运行时数据面可能由内核实现。客户机观察的是联合形成的设备行为，调试时则必须注明权威状态位于用户态还是内核，以及它们在何时同步。
:::
::::

:::: {.quick-quiz}
为什么不能只链接 `master` 分支，再在正文写一个大致行号？

::: {.quick-answer}
`master` 会前移，重构会移动函数，默认值和接口也可能改变。读者日后打开同一链接，看到的未必是作者分析的代码。固定 tag 或完整 commit 能冻结事实，Message-ID 能冻结邮件上下文；行号只适合作为当次阅读的临时导航，不应承担版本身份。
:::
::::

## CPU、Machine 与 Accelerator 是三根互相垂直的轴

QEMU 命令行把 `-cpu`、`-machine` 和 `-accel` 分成不同选项，这不是语法上的偶然。CPU 类型回答客户机看到哪些寄存器、扩展和特权能力；Machine 类型回答 CPU 被放进怎样的地址、中断和设备环境；Accelerator 回答这些 CPU 状态由哪套执行机制推进。三者需要互相校验，却不应合并成一个巨型“平台类型”。同一个 `virt` 可以选择不同的 RISC-V CPU 配置，同一套客户机平台可以在 TCG 或 KVM 下运行，而同一个 RISC-V CPU 类型也可能被其他板级机器复用。

三根轴并非任意组合。KVM 能向客户机暴露什么扩展，受宿主硬件与内核接口约束，不能照搬 TCG 能实现的全部软件 CPU 特性；某个 Machine 还可能要求 CPU 提供特定中断或计时能力。QEMU 在初始化阶段做兼容性检查，失败时应尽早报告，而不是等客户机运行到某条指令才产生含糊异常。因此，看到 `-cpu` 属性校验失败，不应马上去 `virt.c` 修改板级代码；先判断约束属于 ISA 模型、加速器能力，还是平台连接。

RISC-V 的扩展命名让这组三轴关系更加显眼。`H` 扩展规定 HS/VS 等虚拟化相关状态，属于客户机 CPU 的体系结构能力；宿主 KVM 是 QEMU 的加速器选择；`virt` 则可以提供适合虚拟机的中断与 IOMMU 平台。三者在硬件虚拟化实验中会同时出现，却不能用一句“打开 H 扩展就是用了 KVM”概括。TCG 可以模拟带 H 扩展的 RISC-V CPU，用它运行一个客户机内的 hypervisor；KVM 也可能因宿主能力或配置而不向嵌套客户机暴露某项扩展。

:::: {.quick-quiz}
在 ISA 与 RISC-V 不同的宿主上，用 TCG 启动一颗带 H 扩展的 RISC-V 虚拟 CPU，这算不算硬件辅助虚拟化？

::: {.quick-answer}
从 QEMU 执行后端看，它仍是软件翻译，不是宿主 KVM 加速；从客户机体系结构看，这颗虚拟 CPU 可以实现 H 扩展语义，供客户机中的 hypervisor 使用。两个判断描述不同层次，应分别记录，不能因为都出现“虚拟化”就合成一个布尔值。
:::
::::

## 机器配置最终要落成可拥有的状态

命令行很像一串文字，但 QEMU 不能靠字符串驱动整段运行。`-m` 要变成机器内存大小与具体 RAM 后端，`-smp` 要变成 socket、core、thread 与 hart 的拓扑，`-device` 要变成具有类型、属性、父子关系和引用计数的对象，`-drive` 与 `-blockdev` 则会建立块图及其后端。配置解析只完成“用户想要什么”的第一步，造机器阶段还要回答资源是否存在、组合是否合法、错误能否安全回滚。

这也是 QEMU 看起来拥有多套配置机制的原因之一。某些选项属于全局进程，某些属于 Machine 属性，某些创建对象或设备，另一些构造独立后端图。把所有内容都塞进 Machine 会让复用和热插拔变得困难，把所有内容都留成无类型字符串又会推迟错误。当前结构是在多年兼容压力下形成的折中，阅读者应从数据最终归属反推选项，而不是看到一个旧式短选项就假定它是特殊旁路。

配置还形成了一个重要的可复现边界。两条表面相近的命令，若省略了 CPU 型号、固件或设备默认值，跨版本后可能得到不同机器。实验手册因此要求保存完整命令与版本，并尽量显式写出影响结论的属性。这里不主张把所有默认值都展开，那会制造噪声；需要显式的是实验假设所依赖的部分，例如加速器、CPU、内存大小、固件和所研究设备。

## 错误边界比正常调用链更能暴露设计

阅读启动代码时，人们容易只追成功路径：创建对象、realize、进入主循环。工程上更难的是中途失败。假如第二个 hart 创建失败，第一个 hart、已经分配的 RAM 和部分设备该由谁释放；假如字符后端打开失败，UART 对象是否已经接入 QOM 树；假如 KVM capability 不满足，程序应回退 TCG、拒绝启动，还是由用户显式决定。错误策略会决定对象生命周期、引用所有权和初始化阶段的边界。

QEMU 广泛使用 `Error **` 传播可解释错误，并通过对象 unparent、unrealize、引用计数和自动清理辅助回滚，但这不意味着任意初始化函数都可以失败。QOM 的 instance initialization、qdev 的 realize 以及 Machine 板级装配承担不同契约。第四章会说明为什么需要把可能失败的外部资源获取留到 realize；此处先把它当作观察边界的一扇窗：一个函数能否返回错误，通常和调用者是否有能力恢复有关。

运行期错误还要区分客户机错误与宿主实现错误。客户机向不存在的 MMIO 地址访问，可能得到总线错误或默认读值；设备模型违反内部不变量，则可能触发断言；宿主后端断开连接，可能需要向客户机报告 I/O 失败。三类错误若混在一起，既会把恶意客户机输入变成 QEMU 崩溃，也会把真正的程序缺陷悄悄吞掉。后文分析设备回调时，会持续追问输入是否由客户机控制，以及错误如何穿过边界。

## 可观察性必须跨过静态目录

源码给出可能路径，trace 和运行时查询给出这一次走过的路径。QEMU 的 trace-events 分散在子系统附近，经构建配置选择后可输出函数级事件；QMP/HMP 可以查询机器、CPU、内存树、块图和运行状态；宿主调试器、`strace`、`perf` 或线程列表则能观察内核交互。它们处在不同层次，不宜用一种工具包打天下。

例如，`info mtree` 能证明某个 MemoryRegion 最终落在地址树中，却不能证明客户机真的访问过它；设备 read/write trace 能证明回调发生，却未必告诉你是哪条客户机指令触发；TCG 指令日志能靠近客户机执行，却可能大幅改变时间和输出量。好的实验先写问题，再选择最小观察集合。采集越多不一定越可靠，若时间戳、CPU 编号和版本没有对齐，海量日志只会让关联更困难。

观察本身还会扰动系统。打开逐指令日志会改变调度和性能，GDB 单步会打断定时关系，暂停虚拟机查询状态会把一个并发快照变成停机快照。报告中应注明是否停机、是否启用加速器、trace 是否带时间戳，以及结论对扰动是否敏感。对竞态问题，最好用一个轻量事件先确认范围，再有针对性地增加插桩，而不是一开始就输出全部事件。

## 兼容性是一组可观察行为，不是一份口号

虚拟硬件的兼容表面包括寄存器语义、设备地址、中断连接、设备树或 ACPI 描述、默认设备、CPU 特性、复位值与迁移状态。某项内部数据结构完全可以重写，只要客户机和管理工具看不到差异；反过来，一行默认值修改也可能让旧固件无法启动。判断风险不能按补丁行数，必须沿观察者逐一检查。

迁移把这件事推到最严格的位置。源端 QEMU 保存 CPU 与设备状态，目标端恢复后，客户机应像只经历了一次允许的暂停。若 TCG 与 KVM 对某个 CSR 的权威状态不同，迁移代码就要在合适时刻同步；若新版本设备多了状态，VMState 描述需要处理旧流。RISC-V `virt` 没有显式版本机型，并不自动解决这些问题，反而要求维护者在新增默认功能时更小心地界定是否影响旧配置。

兼容并不等于永远禁止改动。上游可以增加显式属性、让新功能默认关闭、根据迁移版本选择行为，或在确认没有用户后移除实验接口。工程选择取决于受影响用户、维护成本与安全性。正文遇到此类改动时，会引用补丁系列中的约束，而不会用“为了兼容”四个字替代证据。

## 如何阅读一组 qemu-devel 补丁

邮件列表中的一个功能通常不是孤立 patch。cover letter 说明问题范围和系列结构，单个补丁解释局部变化，review 会追问命名、迁移、线程安全、测试与兼容性，后续版本再把意见折回代码。只读最终提交说明，容易错过被否决的方案；只读第一版邮件，又可能把后来修掉的设计当作现状。可靠做法是从合入 commit 的 Message-ID 回到系列，再沿 `v2`、`v3` 或 supersedes 关系确认最终讨论。

review 中的建议也有层次。维护者提出疑问，不等于项目已经采纳；作者回复“下一版修改”，还要在下一版或最终 commit 中核对；一个 `Reviewed-by` 表示审阅者认可该补丁版本，不代表他为书中的所有推论背书。因此，本书引用“上游陈述”时会说明说话位置，必要时写明它属于提案、review 还是合入后的提交说明。

本地 Git 与邮件应配合使用。可以先运行 `git log --oneline --follow -- hw/riscv/virt.c` 找到功能引入点，再用 `git show --stat --format=fuller` 检查提交正文、作者和 Message-ID，随后到官方归档阅读上下文。若一次机械重命名淹没了历史，可用 `git log -S'关键符号'` 或 `-G'正则'` 找语义变化。最后回到当前 tag，确认旧讨论中的函数或约束是否仍然存在。

## 给每条强结论做一张证据卡

写作阶段可以为关键结论保留一张很小的证据卡。卡片先写问题，例如“为什么 RISC-V KVM 要单独处理虚拟时间”，再列当前源码位置、引入提交、邮件 Message-ID、可复现实验和反例。最后才写解释，并标注信心。这个过程看似慢，却能防止段落沿着一个顺耳的故事无限发散。

证据卡还应记录时间范围。`v11.1.0-rc0` 能证明的是该 tag 的状态，2018 年提交能证明的是某项功能在当时如何引入，2022 年 KVM 系列则说明另一阶段的约束。把它们串联成演进路径，需要确认中间没有推翻结论的提交。若证据不足，最诚实的写法是“当前结构与这一解释一致，尚未找到上游明确陈述”，而不是把推断写成历史事实。

实验结果也可以成为证据，但其适用范围要窄。一次 TCG 运行观察到 UART 地址不变，只证明这份二进制和命令；多版本对照、源码表与设备树一致，才逐渐提高结论强度。实验无法替代规范，规范也无法证明实现没有 bug。三者互相校验，正是本书每章都设计小实验的原因。

## 本书的收敛规则

QEMU 的子系统足够多，任何章节都可能顺着一个函数扩展到数万行。为了保持主线，正文遇到体系结构相关示例时只使用 RISC-V，板级示例只使用 `virt` 及其 RISC-V 设备模型；需要对比其他体系结构时，只解释公共框架为何存在，不展开另一套启动细节。源码分析锚定 `v11.1.0-rc0`，目标参考版本是 `v11.1.0`，不把 `master` 的临时状态混进结论。

每章还遵守三个停止条件。第一，已经能从客户机现象闭环到当前源码和运行时对象，就不继续罗列相邻函数；第二，设计理由已有提交或邮件直接支撑，就不再用更多二手说法堆砌；第三，作者推断出现多个合理解释时，明确保留不确定性，把可验证问题转成实验或后续阅读。收敛仍然保留深度，只把篇幅用在能改变理解的位置。

最后，实验目录中的手册使用英文，是为了让命令、文件名和上游术语易于复用；书中对实验目的、观察和推理全部使用中文，避免读者在关键概念上来回翻译。正文不会把手册整段复制进来，手册也不承担章节论证。二者通过精确相对链接连接，各自保持清楚的职责。

## 性能边界要从退出与共享状态开始量

讨论 QEMU 性能时，“TCG 慢、KVM 快”只能作为非常粗的背景。真实开销来自具体边界：TCG 的翻译与代码缓存是否命中，客户机访存是否频繁触发帮助函数，KVM vCPU 是否反复退出到用户态，设备请求是否跨线程、跨进程或跨内核边界，锁与缓存行又是否在 CPU 间争用。同一台 `virt`，计算密集负载和小包网络负载可能得到完全不同的瓶颈。

因此，性能实验先统计事件，再测总时间。KVM 场景可以关注退出类型与频率，TCG 场景可以关注翻译块生成和失效，设备场景则观察通知、批处理与数据复制。总时间告诉我们“变快还是变慢”，事件计数帮助解释“在哪条边界上变化”。若只跑一次 wall-clock，宿主调度、频率变化和缓存状态都可能被误认成架构结论。

优化还会移动责任。vhost 把数据面移出 QEMU 主循环，IOThread 把工作移到另一个事件域，批处理减少通知次数，却可能增加单次延迟。每次移动都要重新检查迁移、错误报告和生命周期。后文分析具体机制时，不会把“少一次退出”自动写成更优设计，而会说明它用什么复杂度交换了哪类开销。

## 信任边界决定哪些输入必须防御

完整系统模拟面对的输入远不止命令行。客户机可以读写设备寄存器、构造 DMA 描述符、改变页表并触发异常；网络和磁盘后端还会带来外部数据。即便客户机由同一位开发者启动，设备模型仍应把这些值当成不可信输入，检查长度、索引、地址溢出和生命周期。历史上的重入、越界与 use-after-free 漏洞，往往出现在“客户机动作穿过了开发者原先假定的边界”这一刻。

KVM 并不会让这项责任消失。客户机指令在内核执行后，仍可能以精心构造的 MMIO 或 I/O 请求回到用户态；直通与共享内存则引入 IOMMU、pinning 和权限问题。TCG 也不能因为完全在用户态就放松检查，翻译器和设备模型仍处理客户机控制的数据。分析一段回调时，除了功能正确性，还要标出字段来自命令行、受信后端还是客户机内存。

安全与性能经常拉扯。为了减少复制而直接映射客户机内存，需要保证映射期间对象不被释放；为了避免每次访问加锁而使用 RCU，需要把回收推迟到读者退出；为了支持异步完成，设备请求要携带足够的所有权信息。第三章和第五章会看到这些机制。这里先留下判断标准：优化如果改变了输入到状态的路径，就必须同步审查信任和生命周期边界。

:::: {.quick-quiz}
一次 KVM 运行中，客户机执行了绝大多数指令且总耗时很短，能否据此断言设备模型不再是性能或安全边界？

::: {.quick-answer}
不能。少量高代价退出、共享队列处理或后端阻塞仍可能决定尾延迟，客户机控制的 MMIO、DMA 描述符也仍会进入设备路径。应分别测量 vCPU 执行、退出与 I/O 事件，并检查输入验证和状态所有权，不能用指令执行占比替代边界分析。
:::
::::

## 本章小结

QEMU 系统模拟把配置、机器创建、CPU 执行和 I/O 控制装进同一个可组合进程。TCG 与 KVM 是推进 vCPU 的两条路径，不是两套彼此无关的 `virt` 主板；它们共享大量客户机平台状态，也会因内核加速、设备下沉和能力差异形成真实分流。RISC-V `virt` 虽非实体主板，却向固件和操作系统提供可观察的平台契约，地址、中断、设备树和迁移行为都需要在兼容性语境中审查。

到这里还不必记住所有路径，真正需要带走的是一套定位习惯。遇到现象，先注明系统模拟还是用户态模拟，再注明 CPU、Machine 与 Accelerator；随后写清宿主或客户机坐标、状态所有者和发生阶段。当前源码负责确认实现，上游邮件负责说明被公开讨论的理由，实验负责检验这次配置，作者推断只在三者之间搭桥。只要这几行元数据没有丢，后面即使进入生成代码、回调和异步线程，也能找到返回主线的路。

本章的两个实验还会留下第一份基线材料：固定版本的命令、DTB、对象树、内存树、线程和加速器能力。后续章节每次改变一个条件，再与它比较。这样的累积比每章重新截一组互不相关的输出更有价值，因为它能让代码演进、配置变化与运行现象落在同一条时间线上。

基线还要保存“没有发生什么”。例如 TCG 环境没有 `/dev/kvm` 交互，单 hart 没有第二条 vCPU 泳道，关闭 IOMMU 时 DTB 与对象树都不应凭空出现对应设备。负证据必须说明观测范围，不能写成全局不存在；它却能在后续打开功能时清楚显示新增边界，帮助我们判断变化来自配置还是源码版本。

章末复核可以随机抽一项设备，要求自己用一句话分别说清 ISA 是否规定它、`virt` 怎样暴露它、TCG/KVM 在哪里影响它、运行时怎样观察它。若四句话混成一句，说明边界仍未建立；回到源码与实验补坐标，比继续背更多目录更有效。

对一项历史结论也做同样抽查：当前 tag 能证明什么，引入提交写了什么，邮件 review 又实际讨论了什么。三份材料若只支持机制，就把设计目的留作推断；若实验只覆盖 TCG，就别把结果扩到 KVM。这种克制不会削弱叙事，反而让后续新证据有清楚的落点。

接下来我们会从一条命令行进入这台机器，跟随 `main()`、`qemu_init()`、Machine 创建、板级实现、固件装载与 reset vector，直到第一条客户机指令。到了那里，本章的四类责任会第一次沿真实调用路径会合。
