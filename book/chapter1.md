# QEMU 全景与源码地图

本章从一条最小的 system emulation 命令开始，建立 QEMU 进程的整体模型。目标不是记住目录，而是理解哪些部分描述机器，哪些部分执行 vCPU，哪些部分服务 I/O，以及它们如何通过公共抽象连接。

## 本章目标

- 区分 system emulation、user-mode emulation 与硬件辅助虚拟化；
- 识别启动、配置、执行、I/O 和管理面的主要入口；
- 能根据一个问题选择正确的源码入口，而不是全仓库盲搜。

## 一条命令创建了什么

从命令行解析开始，依次观察 `MachineState`、CPU、内存后端、总线、设备、字符设备和块设备后端的创建。说明“用户配置”如何转化为“运行时对象图”，以及 realize 之前和之后分别允许做什么。

## 三种视角

### 控制面

命令行、QMP、HMP 和配置对象负责表达用户意图。这里关注配置何时生效、错误如何传播，以及运行期可变属性与启动期属性的边界。

### vCPU 执行面

比较 TCG 与 KVM 的共同入口和不同执行路径，建立“执行 guest 代码 -> 退出 -> 处理事件 -> 再次进入”的循环模型。

### I/O 数据面

从一次 guest MMIO 或 virtio 请求出发，标出设备模型、AioContext、块层、网络后端和宿主系统调用的位置。

## 源码目录不是架构图

按职责介绍 `accel/`、`target/`、`hw/`、`system/`、`softmmu/`、`block/`、`net/`、`migration/`、`io/`、`qapi/` 和 `util/`。重点解释跨目录的公共类型和回调关系。

::: {.source-path}
建议先记录启动入口、默认 machine 的 class 初始化、CPU realize 和主循环入口，再用 QMP 的 `query-*` 命令对照运行时状态。不要在第一次阅读时展开所有设备。
:::

## 小结

QEMU 是由对象拓扑、vCPU 执行引擎、事件循环和 I/O 后端共同组成的进程。后续章节会分别拆开这些子系统，再在迁移和调试章节重新合拢。

## 思考题

1. 为什么同一套设备模型可以同时服务 TCG 和 KVM？
2. 哪些配置必须在设备 realize 之前确定？
3. 如果一次 QMP 查询卡住，应优先检查哪个线程和锁边界？
